Dùng Smart Pointer Như Reference Bình Thường

Khi bạn implement trait Deref, bạn có thể tùy chỉnh hành vi của toán tử dereference * (đừng nhầm với toán tử nhân hay glob nhé). Bằng cách implement Deref sao cho smart pointer có thể được dùng như một reference thông thường, bạn có thể viết code hoạt động với references, rồi tái sử dụng code đó cho cả smart pointers luôn.

Hãy cùng xem toán tử dereference hoạt động thế nào với references thông thường trước. Sau đó, ta sẽ thử định nghĩa một kiểu tùy chỉnh hoạt động giống Box<T> và xem tại sao toán tử dereference lại không tự động hoạt động trên kiểu mới toanh đó. Chúng ta sẽ khám phá cách implement trait Deref giúp smart pointers hoạt động tương tự references, rồi sau đó tìm hiểu tính năng deref coercion — thứ giúp ta thoải mái làm việc với cả references lẫn smart pointers.

Theo Dấu Reference Tới Giá Trị

Một reference thông thường là một loại pointer, và cách hình dung dễ nhất là coi pointer như mũi tên trỏ tới một giá trị nào đó đang nằm chỗ khác. Trong Listing 15-6, ta tạo một reference tới giá trị i32 rồi dùng toán tử dereference để đi theo reference đó tới giá trị thực sự.

Biến x chứa giá trị i32 là 5. Ta gán y bằng một reference tới x. Ta có thể assert rằng x bằng 5. Nhưng nếu muốn assert về giá trị trong y, ta phải dùng *y để đi theo reference tới giá trị nó trỏ tới (tức là dereference) để compiler so sánh được giá trị thực. Sau khi dereference y, ta có thể truy cập giá trị integer mà y đang trỏ tới và so sánh với 5.

Nếu thay vào đó ta viết assert_eq!(5, y);, ta sẽ nhận được lỗi compile:

$ cargo run
   Compiling deref-example v0.1.0 (file:///projects/deref-example)
error[E0277]: can't compare `{integer}` with `&{integer}`
 --> src/main.rs:6:5
  |
6 |     assert_eq!(5, y);
  |     ^^^^^^^^^^^^^^^^ no implementation for `{integer} == &{integer}`
  |
  = help: the trait `PartialEq<&{integer}>` is not implemented for `{integer}`
  = note: this error originates in the macro `assert_eq` (in Nightly builds, run with -Z macro-backtrace for more info)

For more information about this error, try `rustc --explain E0277`.
error: could not compile `deref-example` (bin "deref-example") due to 1 previous error

Không thể so sánh một số với reference tới số vì chúng là hai kiểu khác nhau. Ta phải dùng toán tử dereference để đi theo reference tới giá trị nó trỏ tới.

Dùng Box<T> Như Một Reference

Ta có thể viết lại code trong Listing 15-6 dùng Box<T> thay vì reference — toán tử dereference dùng trên Box<T> trong Listing 15-7 hoạt động y hệt như khi dùng trên reference ở Listing 15-6.

Điểm khác nhau chính giữa Listing 15-7 và 15-6 là ở đây ta gán y bằng một box trỏ tới bản copy của x, thay vì reference trỏ tới giá trị của x. Trong assertion cuối, ta có thể dùng toán tử dereference để đi theo con trỏ của box — y hệt như khi y là reference. Tiếp theo, hãy khám phá điều gì đặc biệt ở Box<T> khiến ta có thể dùng được toán tử dereference bằng cách tự định nghĩa một kiểu box của riêng mình.

Tự Xây Dựng Smart Pointer Của Riêng Mình

Hãy cùng xây dựng một kiểu wrapper tương tự Box<T> của thư viện chuẩn để thấy tận mắt cách smart pointer khác biệt với reference như thế nào theo mặc định. Sau đó ta sẽ xem cách thêm khả năng sử dụng toán tử dereference.

Lưu ý: Có một điểm khác biệt lớn giữa kiểu MyBox<T> ta sắp xây dựng và Box<T> thật sự: phiên bản của ta sẽ không lưu dữ liệu trên heap. Ví dụ này tập trung vào Deref, nên chỗ nào dữ liệu thực sự được lưu ít quan trọng hơn hành vi giống pointer.

Kiểu Box<T> thực chất được định nghĩa như một tuple struct có một phần tử, nên Listing 15-8 định nghĩa MyBox<T> theo cách tương tự. Ta cũng định nghĩa hàm new để match với hàm new của Box<T>.

Ta định nghĩa struct tên MyBox và khai báo generic parameter T vì ta muốn kiểu này chứa được giá trị của bất kỳ kiểu nào. MyBox là một tuple struct với một phần tử kiểu T. Hàm MyBox::new nhận một parameter kiểu T và trả về instance MyBox chứa giá trị đó.

Thử thêm hàm main từ Listing 15-7 vào Listing 15-8 và đổi sang dùng MyBox<T> thay cho Box<T>. Code trong Listing 15-9 sẽ không compile vì Rust không biết cách dereference MyBox.

Đây là lỗi compile nhận được:

$ cargo run
   Compiling deref-example v0.1.0 (file:///projects/deref-example)
error[E0614]: type `MyBox<{integer}>` cannot be dereferenced
  --> src/main.rs:14:19
   |
14 |     assert_eq!(5, *y);
   |                   ^^ can't be dereferenced

For more information about this error, try `rustc --explain E0614`.
error: could not compile `deref-example` (bin "deref-example") due to 1 previous error

Kiểu MyBox<T> của ta không thể bị dereference vì ta chưa implement khả năng đó. Để bật toán tử *, ta cần implement trait Deref.

Implement Trait Deref

Như đã thảo luận trong phần "Implementing a Trait on a Type" ở Chương 10, để implement một trait ta cần cung cấp các method mà trait đó yêu cầu. Trait Deref từ thư viện chuẩn yêu cầu ta implement một method tên deref nhận self bằng borrow và trả về reference tới dữ liệu bên trong. Listing 15-10 chứa implementation của Deref để thêm vào định nghĩa của MyBox<T>.

Cú pháp type Target = T; định nghĩa một kiểu associated type cho trait Deref. Associated types là cách khai báo generic parameter hơi khác một chút, nhưng bạn chưa cần lo về chúng bây giờ — ta sẽ đề cập chi tiết hơn ở Chương 20.

Ta điền body của method deref bằng &self.0 để deref trả về reference tới giá trị ta muốn truy cập bằng toán tử *. Nhớ lại từ phần "Creating Different Types with Tuple Structs" ở Chương 5 rằng .0 truy cập phần tử đầu tiên của tuple struct. Hàm main trong Listing 15-9 gọi * trên giá trị MyBox<T> giờ đã compile được, và các assertion đều pass!

Nếu không có trait Deref, compiler chỉ có thể dereference reference &. Method deref cho compiler khả năng nhận một giá trị của bất kỳ kiểu nào implement Deref và gọi method deref để lấy về reference mà nó biết cách dereference.

Khi ta viết *y trong Listing 15-9, thực chất Rust chạy đoạn code này phía sau hậu trường:

*(y.deref())

Rust thay thế toán tử * bằng lời gọi tới method deref rồi mới dereference thông thường, giúp ta không phải nghĩ xem có cần gọi deref không. Tính năng này cho phép ta viết code hoạt động y hệt dù ta có reference hay kiểu implement Deref.

Lý do method deref trả về reference tới giá trị, và rằng toán tử dereference bên ngoài dấu ngoặc trong *(y.deref()) vẫn cần thiết, là do hệ thống ownership. Nếu deref trả về giá trị trực tiếp thay vì reference, giá trị đó sẽ bị move ra khỏi self — thứ ta không muốn trong hầu hết các trường hợp dùng toán tử dereference.

Lưu ý rằng toán tử * chỉ được thay thế bằng lời gọi deref rồi một lần * duy nhất, không đệ quy vô hạn. Vì vậy ta kết thúc với dữ liệu kiểu i32, khớp với 5 trong assert_eq! của Listing 15-9.

Sử Dụng Deref Coercion trong Hàm và Method

Deref coercion chuyển đổi reference tới một kiểu implement Deref thành reference tới một kiểu khác. Ví dụ, deref coercion có thể chuyển &String thành &str vì String implement trait Deref sao cho nó trả về &str. Deref coercion là tiện ích Rust thực hiện trên các đối số của hàm và method, và chỉ hoạt động trên các kiểu implement Deref. Nó diễn ra tự động khi ta truyền reference tới giá trị của một kiểu nào đó cho hàm hoặc method mà kiểu tham số không khớp. Một chuỗi lời gọi method deref sẽ chuyển kiểu ta cung cấp thành kiểu mà tham số cần.

Deref coercion được thêm vào Rust để giúp lập trình viên viết lời gọi hàm và method không phải thêm quá nhiều & và * tường minh. Tính năng này cũng giúp ta viết code linh hoạt hơn, hoạt động được cho cả references lẫn smart pointers.

Để thấy deref coercion trong thực tế, hãy dùng kiểu MyBox<T> từ Listing 15-8 cùng implementation Deref ta thêm trong Listing 15-10. Listing 15-11 cho thấy định nghĩa một hàm có parameter là string slice.

Ta có thể gọi hàm hello với một string slice làm đối số, ví dụ hello("Rust");. Deref coercion giúp ta gọi hello với reference tới giá trị kiểu MyBox<String> như trong Listing 15-12.

Ở đây ta gọi hàm hello với đối số &m là reference tới giá trị MyBox<String>. Vì ta đã implement trait Deref trên MyBox<T> ở Listing 15-10, Rust có thể biến &MyBox<String> thành &String bằng cách gọi deref. Thư viện chuẩn cung cấp implementation Deref trên String trả về string slice, và Rust gọi deref thêm một lần nữa để biến &String thành &str, khớp với định nghĩa hàm hello.

Nếu Rust không có deref coercion, ta sẽ phải viết code trong Listing 15-13 thay vì Listing 15-12 để gọi hello với giá trị kiểu &MyBox<String>.

(*m) dereference MyBox<String> thành String. Rồi & và [..] lấy string slice của String bằng với toàn bộ chuỗi để khớp với signature của hello. Code không có deref coercion trông rối rắm hơn nhiều với đống ký hiệu đó. Deref coercion cho phép Rust tự động xử lý những chuyển đổi này cho ta.

Khi trait Deref được định nghĩa cho các kiểu liên quan, Rust sẽ phân tích các kiểu và dùng Deref::deref bao nhiêu lần cần thiết để có được reference khớp với kiểu parameter. Số lần cần chèn Deref::deref được giải quyết tại compile time, nên không có penalty runtime nào khi tận dụng deref coercion!

Xử Lý Deref Coercion Với Mutable References

Tương tự như dùng trait Deref để override toán tử * trên immutable references, bạn có thể dùng trait DerefMut để override toán tử * trên mutable references.

Rust thực hiện deref coercion khi tìm thấy kiểu và trait implementation trong ba trường hợp:

  1. Từ &T sang &U khi T: Deref<Target=U>
  2. Từ &mut T sang &mut U khi T: DerefMut<Target=U>
  3. Từ &mut T sang &U khi T: Deref<Target=U>

Hai trường hợp đầu giống nhau ngoại trừ trường hợp thứ hai xử lý mutability. Trường hợp đầu: nếu bạn có &T và T implement Deref tới kiểu U, bạn có thể lấy &U một cách trong suốt. Trường hợp thứ hai: deref coercion tương tự xảy ra với mutable references.

Trường hợp thứ ba tinh tế hơn: Rust cũng sẽ coerce mutable reference thành immutable reference. Nhưng chiều ngược lại thì không thể: immutable references sẽ không bao giờ được coerce thành mutable references. Do quy tắc borrowing, nếu bạn có mutable reference, thì mutable reference đó phải là reference duy nhất tới dữ liệu đó (không thì chương trình không compile được). Chuyển một mutable reference thành một immutable reference sẽ không bao giờ vi phạm quy tắc borrowing. Nhưng chuyển immutable reference thành mutable reference sẽ đòi hỏi immutable reference ban đầu là immutable reference duy nhất tới dữ liệu đó — điều mà borrowing rules không đảm bảo. Vì vậy Rust không thể giả định rằng việc chuyển immutable reference thành mutable reference là khả thi.
