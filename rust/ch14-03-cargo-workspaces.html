Cargo Workspaces

Ở Chương 12, chúng ta đã xây dựng một package gồm cả binary crate lẫn library crate. Khi project phát triển, bạn có thể thấy library crate ngày càng phình to và muốn tách package ra thành nhiều library crate hơn. Cargo cung cấp một tính năng gọi là workspaces để giúp quản lý nhiều package liên quan được phát triển song song.

Tạo một Workspace

Một workspace là tập hợp các package cùng chia sẻ một file Cargo.lock và thư mục output chung. Hãy cùng tạo một project dùng workspace — chúng ta sẽ dùng code đơn giản để tập trung vào cấu trúc của workspace. Có nhiều cách tổ chức workspace khác nhau, ở đây chúng ta sẽ trình bày một cách phổ biến. Workspace của chúng ta sẽ gồm một binary và hai library. Binary sẽ cung cấp chức năng chính và phụ thuộc vào hai library. Một library cung cấp hàm add_one và library kia cung cấp hàm add_two. Ba crate này sẽ thuộc cùng một workspace. Hãy bắt đầu bằng cách tạo thư mục mới cho workspace:

$ mkdir add
$ cd add

Tiếp theo, trong thư mục add, chúng ta tạo file Cargo.toml để cấu hình toàn bộ workspace. File này sẽ không có phần [package]. Thay vào đó, nó bắt đầu bằng phần [workspace] cho phép chúng ta thêm các thành viên vào workspace. Chúng ta cũng chỉ định dùng phiên bản mới nhất của thuật toán resolver của Cargo bằng cách đặt giá trị resolver là "3":

Tên file: Cargo.toml

[workspace]
resolver = "3"

Tiếp theo, chúng ta tạo binary crate adder bằng cách chạy cargo new trong thư mục add:

$ cargo new adder
     Created binary (application) `adder` package
      Adding `adder` as member of workspace at `file:///projects/add`

Chạy cargo new bên trong một workspace cũng tự động thêm package vừa tạo vào key members trong phần [workspace] của file Cargo.toml workspace, như thế này:

[workspace]
resolver = "3"
members = ["adder"]

Lúc này, chúng ta có thể build workspace bằng cách chạy cargo build. Các file trong thư mục add của bạn trông như thế này:

├── Cargo.lock
├── Cargo.toml
├── adder
│   ├── Cargo.toml
│   └── src
│       └── main.rs
└── target

Workspace có một thư mục target duy nhất ở cấp cao nhất để chứa các artifact được compile; package adder không có thư mục target riêng. Thậm chí nếu chúng ta chạy cargo build từ bên trong thư mục adder, các artifact vẫn sẽ nằm trong add/target chứ không phải add/adder/target. Cargo tổ chức thư mục target như vậy trong workspace vì các crate trong workspace được thiết kế để phụ thuộc lẫn nhau. Nếu mỗi crate có thư mục target riêng, mỗi crate sẽ phải recompile tất cả các crate khác trong workspace để đặt artifact vào thư mục target của mình. Bằng cách chia sẻ một thư mục target duy nhất, các crate có thể tránh việc rebuild không cần thiết.

Tạo Package Thứ Hai trong Workspace

Tiếp theo, hãy tạo một member package khác trong workspace và đặt tên là add_one. Tạo library crate mới tên add_one:

$ cargo new add_one --lib
     Created library `add_one` package
      Adding `add_one` as member of workspace at `file:///projects/add`

File Cargo.toml cấp cao nhất giờ sẽ chứa đường dẫn add_one trong danh sách members:

Tên file: Cargo.toml

[workspace]
resolver = "3"
members = ["adder", "add_one"]

Thư mục add của bạn giờ có các thư mục và file như sau:

├── Cargo.lock
├── Cargo.toml
├── add_one
│   ├── Cargo.toml
│   └── src
│       └── lib.rs
├── adder
│   ├── Cargo.toml
│   └── src
│       └── main.rs
└── target

Trong file add_one/src/lib.rs, hãy thêm hàm add_one:

Tên file: add_one/src/lib.rs

pub fn add_one(x: i32) -> i32 {
    x + 1
}

Bây giờ chúng ta có thể cho package adder (binary) phụ thuộc vào package add_one (library). Đầu tiên, chúng ta cần thêm path dependency vào add_one trong file adder/Cargo.toml.

Tên file: adder/Cargo.toml

[dependencies]
add_one = { path = "../add_one" }

Cargo không tự giả định rằng các crate trong workspace sẽ phụ thuộc nhau, vì vậy chúng ta cần khai báo rõ ràng các mối quan hệ dependency.

Tiếp theo, hãy dùng hàm add_one (từ crate add_one) trong crate adder. Mở file adder/src/main.rs và sửa hàm main để gọi hàm add_one, như trong Listing 14-7.

Hãy build workspace bằng cách chạy cargo build trong thư mục add cấp cao nhất!

$ cargo build
   Compiling add_one v0.1.0 (file:///projects/add/add_one)
   Compiling adder v0.1.0 (file:///projects/add/adder)
    Finished `dev` profile [unoptimized + debuginfo] target(s) in 0.22s

Để chạy binary crate từ thư mục add, chúng ta có thể chỉ định package nào trong workspace cần chạy bằng argument -p và tên package với cargo run:

$ cargo run -p adder
    Finished `dev` profile [unoptimized + debuginfo] target(s) in 0.00s
     Running `target/debug/adder`
Hello, world! 10 plus one is 11!

Lệnh này chạy code trong adder/src/main.rs, vốn phụ thuộc vào crate add_one.

Phụ thuộc vào Package Bên Ngoài

Lưu ý rằng workspace chỉ có một file Cargo.lock ở cấp cao nhất, thay vì có một Cargo.lock trong thư mục của mỗi crate. Điều này đảm bảo rằng tất cả các crate đang dùng cùng một phiên bản của mọi dependency. Nếu chúng ta thêm package rand vào file adder/Cargo.toml và add_one/Cargo.toml, Cargo sẽ resolve cả hai về một phiên bản duy nhất của rand và ghi lại trong file Cargo.lock duy nhất đó. Việc tất cả các crate trong workspace dùng cùng dependency đảm bảo chúng luôn tương thích với nhau. Hãy thêm crate rand vào phần [dependencies] trong file add_one/Cargo.toml để có thể dùng rand trong crate add_one:

Tên file: add_one/Cargo.toml

[dependencies]
rand = "0.8.5"

Bây giờ chúng ta có thể thêm use rand; vào file add_one/src/lib.rs, và build toàn bộ workspace bằng cách chạy cargo build trong thư mục add sẽ tải về và compile crate rand. Chúng ta sẽ nhận được một cảnh báo vì chúng ta không thực sự dùng rand mà chúng ta đã đưa vào scope:

$ cargo build
    Updating crates.io index
  Downloaded rand v0.8.5
   --snip--
   Compiling rand v0.8.5
   Compiling add_one v0.1.0 (file:///projects/add/add_one)
warning: unused import: `rand`
 --> add_one/src/lib.rs:1:5
  |
1 | use rand;
  |     ^^^^
  |
  = note: `#[warn(unused_imports)]` on by default

warning: `add_one` (lib) generated 1 warning (run `cargo fix --lib -p add_one` to apply 1 suggestion)
   Compiling adder v0.1.0 (file:///projects/add/adder)
    Finished `dev` profile [unoptimized + debuginfo] target(s) in 0.95s

File Cargo.lock cấp cao nhất giờ chứa thông tin về dependency của add_one với rand. Tuy nhiên, dù rand được dùng ở đâu đó trong workspace, chúng ta vẫn không thể dùng nó trong các crate khác trong workspace trừ khi thêm rand vào file Cargo.toml của chúng. Ví dụ, nếu thêm use rand; vào file adder/src/main.rs của package adder, chúng ta sẽ gặp lỗi:

$ cargo build
  --snip--
   Compiling adder v0.1.0 (file:///projects/add/adder)
error[E0432]: unresolved import `rand`
 --> adder/src/main.rs:2:5
  |
2 | use rand;
  |     ^^^^ no external crate `rand`

Để khắc phục, hãy chỉnh file Cargo.toml của package adder và chỉ rõ rằng rand cũng là dependency của nó. Build package adder sẽ thêm rand vào danh sách dependency của adder trong Cargo.lock, nhưng sẽ không tải thêm bản copy nào của rand. Cargo đảm bảo rằng mọi crate trong mọi package của workspace dùng package rand sẽ dùng cùng phiên bản, miễn là chúng khai báo các phiên bản tương thích, giúp tiết kiệm dung lượng và đảm bảo các crate trong workspace tương thích với nhau.

Nếu các crate trong workspace khai báo các phiên bản không tương thích của cùng một dependency, Cargo sẽ resolve từng cái một nhưng vẫn cố gắng resolve ít phiên bản nhất có thể.

Thêm Test vào Workspace

Để thêm một cải tiến nữa, hãy thêm test cho hàm add_one::add_one trong crate add_one:

Tên file: add_one/src/lib.rs

pub fn add_one(x: i32) -> i32 {
    x + 1
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn it_works() {
        assert_eq!(3, add_one(2));
    }
}

Bây giờ hãy chạy cargo test trong thư mục add cấp cao nhất. Chạy cargo test trong một workspace có cấu trúc như này sẽ chạy test cho tất cả các crate trong workspace:

$ cargo test
   Compiling add_one v0.1.0 (file:///projects/add/add_one)
   Compiling adder v0.1.0 (file:///projects/add/adder)
    Finished `test` profile [unoptimized + debuginfo] target(s) in 0.20s
     Running unittests src/lib.rs (target/debug/deps/add_one-93c49ee75dc46543)

running 1 test
test tests::it_works ... ok

test result: ok. 1 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out; finished in 0.00s

     Running unittests src/main.rs (target/debug/deps/adder-3a47283c568d2b6a)

running 0 tests

test result: ok. 0 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out; finished in 0.00s

   Doc-tests add_one

running 0 tests

test result: ok. 0 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out; finished in 0.00s

Phần đầu tiên của output cho thấy test it_works trong crate add_one đã pass. Phần tiếp theo cho thấy không có test nào trong crate adder, và phần cuối cùng cho thấy không có documentation test nào trong crate add_one.

Chúng ta cũng có thể chạy test cho một crate cụ thể trong workspace từ thư mục cấp cao nhất bằng cách dùng flag -p và chỉ định tên crate cần test:

$ cargo test -p add_one
    Finished `test` profile [unoptimized + debuginfo] target(s) in 0.00s
     Running unittests src/lib.rs (target/debug/deps/add_one-93c49ee75dc46543)

running 1 test
test tests::it_works ... ok

test result: ok. 1 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out; finished in 0.00s

   Doc-tests add_one

running 0 tests

test result: ok. 0 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out; finished in 0.00s

Output này cho thấy cargo test chỉ chạy test cho crate add_one và không chạy test của crate adder.

Nếu bạn xuất bản các crate trong workspace lên crates.io, mỗi crate trong workspace cần được xuất bản riêng biệt. Giống như cargo test, chúng ta có thể xuất bản một crate cụ thể trong workspace bằng flag -p và chỉ định tên crate cần xuất bản.

Để luyện tập thêm, hãy thêm crate add_two vào workspace này theo cách tương tự như crate add_one!

Khi project của bạn phát triển, hãy cân nhắc dùng workspace: nó giúp bạn làm việc với các thành phần nhỏ hơn, dễ hiểu hơn thay vì một khối code khổng lồ. Hơn nữa, giữ các crate trong workspace có thể giúp việc phối hợp giữa chúng dễ dàng hơn nếu chúng thường xuyên được thay đổi cùng lúc.
