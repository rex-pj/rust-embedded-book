Paths — Đường dẫn tham chiếu item trong Module Tree

Để chỉ cho Rust biết một item đang nằm ở đâu trong cây module, ta dùng path — hoàn toàn giống cách bạn dùng đường dẫn khi điều hướng filesystem. Muốn gọi một hàm, bạn cần biết path của nó.

Một path có thể ở hai dạng:

  • Absolute path là path đầy đủ tính từ crate root; với code từ crate ngoài thì bắt đầu bằng tên crate, còn code trong crate hiện tại thì bắt đầu bằng từ khóa crate.
  • Relative path bắt đầu từ module hiện tại và dùng self, super, hoặc một identifier trong module đó.

Cả hai loại path đều được nối tiếp bởi một hoặc nhiều identifier phân cách nhau bằng dấu hai chấm đôi (::).

Quay lại Listing 7-1 — giả sử ta muốn gọi hàm add_to_waitlist. Câu hỏi đặt ra là: path của hàm add_to_waitlist là gì? Listing 7-3 là Listing 7-1 với một số module và hàm đã được lược bỏ.

Ta sẽ trình bày hai cách gọi hàm add_to_waitlist từ hàm mới eat_at_restaurant được định nghĩa ở crate root. Hai path này đúng, nhưng vẫn còn một vấn đề khác khiến ví dụ chưa compile được — ta sẽ giải thích ngay sau đây.

Hàm eat_at_restaurant là một phần public API của library crate, nên ta đánh dấu nó bằng từ khóa pub. Phần "Exposing Paths với từ khóa pub" sẽ đi sâu hơn về pub.

Lần đầu gọi add_to_waitlist trong eat_at_restaurant, ta dùng absolute path. Hàm add_to_waitlist được định nghĩa trong cùng crate với eat_at_restaurant, nghĩa là ta có thể dùng từ khóa crate để bắt đầu absolute path. Sau đó ta lần lượt liệt kê từng module cho đến khi đến được add_to_waitlist. Hình dung như hệ thống file: ta chỉ định /front_of_house/hosting/add_to_waitlist để chạy chương trình; dùng crate để bắt đầu từ crate root giống như dùng / để bắt đầu từ filesystem root trong shell.

Lần thứ hai gọi add_to_waitlist, ta dùng relative path bắt đầu từ front_of_house — module được định nghĩa cùng cấp với eat_at_restaurant. Equivalent trong filesystem là front_of_house/hosting/add_to_waitlist. Bắt đầu bằng tên module nghĩa là path đó là relative.

Việc chọn absolute hay relative là quyết định bạn đưa ra dựa trên dự án, tùy thuộc khả năng bạn sẽ di chuyển code định nghĩa item riêng biệt hay cùng với code sử dụng nó. Ví dụ, nếu ta chuyển module front_of_house và hàm eat_at_restaurant vào module customer_experience, ta phải cập nhật absolute path đến add_to_waitlist, nhưng relative path vẫn hợp lệ. Ngược lại, nếu ta chuyển riêng hàm eat_at_restaurant vào module dining, absolute path đến add_to_waitlist vẫn nguyên vẹn nhưng relative path cần cập nhật. Nhìn chung ta thiên về dùng absolute path vì khả năng muốn di chuyển phần định nghĩa và phần sử dụng độc lập nhau là cao hơn.

Giờ thử compile Listing 7-3 xem sao! Lỗi xuất hiện như ở Listing 7-4.

Thông báo lỗi nói module hosting là private. Tức là ta có path đúng đến module hosting và hàm add_to_waitlist, nhưng Rust không cho phép dùng chúng vì không có quyền truy cập phần private. Trong Rust, tất cả item (hàm, method, struct, enum, module, constant) đều private với module cha theo mặc định. Nếu muốn giữ một item là private, hãy đặt nó trong module.

Item trong module cha không thể dùng item private bên trong module con, nhưng item trong module con có thể dùng item của các module tổ tiên. Lý do là module con bọc và ẩn chi tiết implementation của nó, nhưng module con vẫn thấy được ngữ cảnh mà nó được định nghĩa. Hình dung như phòng back office nhà hàng: những gì xảy ra trong đó là private với khách, nhưng quản lý có thể thấy và làm mọi thứ trong nhà hàng họ điều hành.

Rust chọn cách này để việc ẩn chi tiết implementation trở thành mặc định — bạn biết phần code nào có thể thay đổi mà không làm vỡ code bên ngoài. Tuy nhiên Rust cho phép phơi bày phần bên trong module con ra cho module tổ tiên bằng cách dùng từ khóa pub.

Tiếc là code ở Listing 7-5 vẫn còn lỗi compiler, như ở Listing 7-6.

Chuyện gì xảy ra vậy? Thêm pub trước mod hosting làm module đó public. Với thay đổi này, nếu ta truy cập được front_of_house, ta truy cập được hosting. Nhưng nội dung bên trong hosting vẫn còn private — làm module public không có nghĩa là làm public nội dung bên trong nó. Từ khóa pub trên module chỉ cho phép code ở module tổ tiên tham chiếu đến nó, chứ không truy cập được code bên trong. Vì module là container, chỉ làm module public thôi chưa đủ — ta cần chọn một hoặc nhiều item bên trong để public hóa thêm.

Lỗi ở Listing 7-6 nói hàm add_to_waitlist là private. Quy tắc privacy áp dụng cho cả struct, enum, hàm, method và module.

Hãy cũng làm hàm add_to_waitlist public bằng cách thêm pub trước định nghĩa của nó, như ở Listing 7-7.

Giờ code compile được rồi! Để hiểu tại sao thêm pub lại cho phép dùng các path này theo quy tắc privacy, hãy xem xét cả hai dạng path.

Với absolute path, ta bắt đầu từ crate — root của cây module. Module front_of_house được định nghĩa ở crate root. Dù front_of_house không public, vì eat_at_restaurant được định nghĩa trong cùng module với front_of_house (chúng là siblings), ta có thể tham chiếu đến front_of_house từ eat_at_restaurant. Tiếp theo là module hosting được đánh dấu pub — ta truy cập được module cha của hosting nên truy cập được hosting. Cuối cùng, hàm add_to_waitlist được đánh dấu pub và ta truy cập được module cha của nó, nên lời gọi hàm này hợp lệ!

Với relative path, logic tương tự ngoại trừ bước đầu: thay vì bắt đầu từ crate root, path bắt đầu từ front_of_house. Module front_of_house được định nghĩa trong cùng module với eat_at_restaurant, nên relative path này hợp lệ. Vì hosting và add_to_waitlist đều pub, phần còn lại cũng hợp lệ, và lời gọi hàm ổn!

Nếu bạn định chia sẻ library crate để các project khác dùng, public API là hợp đồng với người dùng crate — quy định cách họ tương tác với code của bạn. Có nhiều vấn đề xoay quanh việc quản lý thay đổi public API để người khác dễ phụ thuộc vào crate; những vấn đề đó nằm ngoài phạm vi cuốn sách. Nếu bạn quan tâm, xem Rust API Guidelines.

Lưu ý: Best Practices cho Package có cả Binary và Library

Ta đã nhắc rằng một package có thể chứa cả binary crate root src/main.rs lẫn library crate root src/lib.rs, và cả hai crate mang tên package mặc định. Thông thường, package theo pattern này chỉ đặt vừa đủ code trong binary crate để khởi chạy executable gọi vào code trong library crate. Điều này giúp các project khác tận dụng tối đa chức năng của package vì library crate có thể được chia sẻ.

Module tree nên được định nghĩa trong src/lib.rs. Từ đó, mọi item public có thể dùng trong binary crate bằng cách bắt đầu path bằng tên package. Binary crate trở thành user của library crate — giống hệt một crate ngoài hoàn toàn: nó chỉ dùng được public API. Điều này giúp bạn thiết kế API tốt; không chỉ là tác giả, bạn còn là client của chính mình!

Ở Chương 12, ta sẽ minh họa quy tắc tổ chức này với một chương trình command line chứa cả binary crate lẫn library crate.

Bắt đầu Relative Path với super

Ta có thể xây dựng relative path bắt đầu từ module cha thay vì module hiện tại hoặc crate root, bằng cách dùng super ở đầu path. Điều này giống như bắt đầu filesystem path bằng .. — nhảy lên thư mục cha. Dùng super giúp ta tham chiếu item biết chắc nằm trong module cha, làm việc sắp xếp lại cây module dễ hơn khi module gắn chặt với cha nhưng cha có thể bị chuyển đi chỗ khác trong tương lai.

Xem code ở Listing 7-8 mô tả tình huống đầu bếp sửa một order sai rồi đích thân mang ra cho khách. Hàm fix_incorrect_order trong module back_of_house gọi hàm deliver_order ở module cha bằng cách dùng super để bắt đầu path.

Hàm fix_incorrect_order nằm trong module back_of_house, nên ta dùng super để nhảy lên module cha của back_of_house — tức là crate, root. Từ đó ta tìm deliver_order và tìm thấy nó. Thành công! Ta cho rằng module back_of_house và hàm deliver_order sẽ giữ nguyên quan hệ và được chuyển cùng nhau nếu ta tổ chức lại cây module. Vì vậy dùng super sẽ có ít chỗ cần cập nhật code hơn nếu code này được chuyển sang module khác.

Làm Struct và Enum Public

Ta cũng có thể dùng pub để biến struct và enum thành public, nhưng có vài chi tiết thêm. Nếu dùng pub trước định nghĩa struct, ta làm struct đó public, nhưng các field của struct vẫn còn private. Ta có thể làm từng field public hay không tùy trường hợp. Ở Listing 7-9, ta định nghĩa struct public back_of_house::Breakfast với field toast public nhưng field seasonal_fruit private. Mô hình này giống nhà hàng: khách có thể chọn loại bánh mì đi kèm bữa ăn, nhưng đầu bếp quyết định loại trái cây dựa vào mùa và tồn kho. Trái cây thay đổi nhanh nên khách không được chọn hay thậm chí biết họ sẽ nhận loại gì.

Vì field toast trong back_of_house::Breakfast là public, trong eat_at_restaurant ta có thể đọc và ghi vào toast bằng dot notation. Chú ý ta không thể dùng field seasonal_fruit vì nó là private. Thử bỏ comment dòng sửa seasonal_fruit để xem lỗi gì xuất hiện!

Ngoài ra, vì back_of_house::Breakfast có field private, struct cần cung cấp một associated function public để tạo instance của Breakfast (ta đặt tên là summer). Nếu Breakfast không có hàm như vậy, ta không thể tạo instance của Breakfast trong eat_at_restaurant vì không thể đặt giá trị cho field private seasonal_fruit.

Ngược lại, nếu ta làm một enum public thì tất cả variant của nó đều public. Ta chỉ cần đặt pub trước từ khóa enum, như ở Listing 7-10.

Vì ta làm enum Appetizer public, ta có thể dùng variant Soup và Salad trong eat_at_restaurant.

Enum không hữu ích lắm nếu variant không public — sẽ rất phiền nếu phải annotate từng variant với pub mỗi lần. Vì vậy, mặc định variant của enum là public. Còn struct thường hữu ích dù field không public, nên field của struct mặc định là private trừ khi được annotate bằng pub.

Còn một tình huống liên quan đến pub ta chưa đề cập — đó là tính năng cuối của module system: từ khóa use. Ta sẽ nói về use trước, rồi sau đó trình bày cách kết hợp pub với use.
