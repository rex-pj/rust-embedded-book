Packages và Crates

Các thành phần đầu tiên của hệ thống module mà chúng ta sẽ tìm hiểu là packages (gói) và crates.

Một crate là đơn vị nhỏ nhất chứa mã mà trình biên dịch Rust xem xét tại một thời điểm. Thậm chí nếu bạn chạy rustc thay vì cargo và truyền vào một file mã nguồn đơn lẻ (như chúng ta đã làm ở phần ”Cơ bản về chương trình Rust” trong Chương 1), trình biên dịch vẫn coi file đó là một crate. Các crate có thể chứa các module, và các module đó có thể được định nghĩa ở các file khác nhau và sẽ được biên dịch cùng với crate, như chúng ta sẽ thấy ở các phần tiếp theo.

Một crate có thể tồn tại dưới một trong hai dạng: binary crate hoặc library crate. Binary crates là các chương trình có thể biên dịch thành file thực thi để chạy, chẳng hạn như một chương trình dòng lệnh hoặc một server. Mỗi binary crate phải có một hàm main xác định điều gì xảy ra khi file thực thi được chạy. Tất cả các crate chúng ta đã tạo từ trước đến nay đều là binary crate.

Library crates không có hàm main và không biên dịch thành file thực thi. Thay vào đó, chúng định nghĩa các chức năng để chia sẻ với nhiều dự án. Ví dụ, crate rand mà chúng ta đã dùng trong Chương 2 cung cấp chức năng tạo số ngẫu nhiên. Phần lớn khi các Rustacean nói “crate”, họ thường ý chỉ library crate, và dùng từ “crate” như từ đồng nghĩa với khái niệm “library” trong lập trình nói chung.

Crate root là file mã nguồn mà trình biên dịch Rust bắt đầu từ đó và tạo thành module gốc của crate (chúng ta sẽ tìm hiểu sâu hơn về module ở phần ”Kiểm soát Phạm Vi và Quyền Riêng Tư với Modules”).

Một package là tập hợp một hoặc nhiều crate cung cấp một bộ chức năng nhất định. Một package chứa file Cargo.toml mô tả cách build các crate đó. Bản thân Cargo cũng là một package chứa binary crate cho công cụ dòng lệnh mà bạn đang dùng để build mã. Package Cargo còn chứa một library crate mà binary crate phụ thuộc vào. Các dự án khác có thể phụ thuộc vào library crate của Cargo để sử dụng cùng logic mà công cụ dòng lệnh Cargo dùng.

Một package có thể chứa bao nhiêu binary crate tùy ý, nhưng chỉ có thể chứa tối đa một library crate. Một package phải chứa ít nhất một crate, dù là library hay binary crate.

Hãy xem điều gì xảy ra khi chúng ta tạo một package. Đầu tiên, ta nhập lệnh cargo new my-project:

$ cargo new my-project
     Created binary (application) `my-project` package
$ ls my-project
Cargo.toml
src
$ ls my-project/src
main.rs

Sau khi chạy cargo new my-project, chúng ta dùng ls để xem những gì Cargo tạo ra. Trong thư mục my-project, có một file Cargo.toml — đây là package của chúng ta. Ngoài ra còn có thư mục src chứa file main.rs. Mở Cargo.toml trong trình soạn thảo văn bản và bạn sẽ thấy không có đề cập nào tới src/main.rs. Cargo quy ước rằng src/main.rs là crate root của một binary crate có cùng tên với package. Tương tự, Cargo biết rằng nếu thư mục package chứa file src/lib.rs, thì package đó chứa một library crate có cùng tên với package, và src/lib.rs là crate root của nó. Cargo truyền các file crate root cho rustc để build library hoặc binary.

Ở đây, chúng ta có một package chỉ chứa src/main.rs, nghĩa là nó chỉ có một binary crate tên my-project. Nếu một package chứa cả src/main.rs lẫn src/lib.rs, nó có hai crate: một binary và một library, cả hai đều có cùng tên với package. Một package có thể có nhiều binary crate bằng cách đặt các file trong thư mục src/bin: mỗi file sẽ là một binary crate riêng biệt.
