Advanced Types

Hệ thống kiểu của Rust có một số tính năng mà ta đã đề cập rải rác nhưng chưa thảo luận chi tiết. Ta sẽ bắt đầu bằng việc nói về newtypes nói chung — tại sao chúng lại hữu ích như một kiểu. Sau đó ta sẽ chuyển sang type aliases, một tính năng tương tự newtype nhưng có ngữ nghĩa hơi khác. Ta cũng sẽ bàn về kiểu ! và dynamically sized types.

Type Safety và Abstraction với Newtype Pattern

Phần này giả định bạn đã đọc phần trước "Implementing External Traits with the Newtype Pattern". Newtype pattern còn hữu ích cho những tác vụ khác ngoài những gì ta đã thảo luận, bao gồm việc kiểm tra tĩnh rằng các giá trị không bao giờ bị nhầm lẫn với nhau và chỉ định đơn vị của một giá trị. Bạn đã thấy ví dụ dùng newtype để chỉ định đơn vị trong Listing 20-16: Nhớ lại rằng các struct Millimeters và Meters bọc giá trị u32 trong newtype. Nếu ta viết một hàm với tham số kiểu Millimeters, ta sẽ không thể compile chương trình nếu vô tình cố gọi hàm đó với giá trị kiểu Meters hay u32 thông thường.

Ta cũng có thể dùng newtype pattern để trừu tượng hóa một số chi tiết implementation của kiểu: Kiểu mới có thể expose public API khác với API của kiểu private bên trong.

Newtypes cũng có thể ẩn đi implementation nội bộ. Ví dụ, ta có thể cung cấp kiểu People bọc một HashMap<i32, String> lưu trữ ID của người liên kết với tên của họ. Code sử dụng People sẽ chỉ tương tác với public API ta cung cấp, chẳng hạn như method để thêm tên vào collection People; code đó không cần biết ta gán i32 ID cho tên bên trong. Newtype pattern là cách nhẹ nhàng để đạt được encapsulation nhằm ẩn đi chi tiết implementation, điều ta đã thảo luận trong phần "Encapsulation that Hides Implementation Details" ở Chương 18.

Type Synonyms và Type Aliases

Rust cung cấp khả năng khai báo type alias để đặt tên khác cho một kiểu đã có. Ta dùng từ khóa type cho việc này. Ví dụ, ta có thể tạo alias Kilometers cho i32 như sau:

fn main() {
    type Kilometers = i32;

    let x: i32 = 5;
    let y: Kilometers = 5;

    println!("x + y = {}", x + y);
}

Giờ alias Kilometers là synonym (tên đồng nghĩa) của i32; không giống với các kiểu Millimeters và Meters ta tạo trong Listing 20-16, Kilometers không phải là một kiểu mới tách biệt. Các giá trị có kiểu Kilometers sẽ được xử lý giống hệt như giá trị kiểu i32:

fn main() {
    type Kilometers = i32;

    let x: i32 = 5;
    let y: Kilometers = 5;

    println!("x + y = {}", x + y);
}

Vì Kilometers và i32 là cùng một kiểu, ta có thể cộng giá trị của cả hai loại và truyền giá trị Kilometers vào hàm nhận tham số i32. Tuy nhiên, với cách này ta không nhận được lợi ích kiểm tra kiểu từ newtype pattern đã thảo luận trước đó. Nói cách khác, nếu ta nhầm lẫn giá trị Kilometers và i32 ở đâu đó, compiler sẽ không báo lỗi.

Use case chính của type synonym là giảm sự lặp lại. Ví dụ, ta có thể có kiểu dài như sau:

Box<dyn Fn() + Send + 'static>

Viết kiểu dài dòng này trong signature của hàm và các type annotation khắp nơi trong code có thể rất mệt mỏi và dễ nhầm. Hãy tưởng tượng một dự án đầy code như trong Listing 20-25.

Type alias giúp code này dễ quản lý hơn bằng cách giảm sự lặp lại. Trong Listing 20-26, ta đã giới thiệu alias tên Thunk cho kiểu dài dòng và có thể thay thế tất cả lần dùng kiểu đó bằng alias ngắn hơn Thunk.

Code này dễ đọc và viết hơn nhiều! Đặt tên có ý nghĩa cho type alias cũng giúp truyền đạt mục đích của bạn (thunk là từ chỉ code sẽ được thực thi vào lúc sau, nên đây là tên thích hợp cho closure được lưu trữ).

Type aliases cũng thường được dùng với kiểu Result<T, E> để giảm lặp lại. Hãy xem module std::io trong thư viện chuẩn. Các thao tác I/O thường trả về Result<T, E> để xử lý trường hợp thao tác thất bại. Thư viện có struct std::io::Error đại diện cho tất cả các lỗi I/O có thể xảy ra. Nhiều hàm trong std::io trả về Result<T, E> với E là std::io::Error, chẳng hạn như các hàm trong trait Write:

use std::fmt;
use std::io::Error;

pub trait Write {
    fn write(&mut self, buf: &[u8]) -> Result<usize, Error>;
    fn flush(&mut self) -> Result<(), Error>;

    fn write_all(&mut self, buf: &[u8]) -> Result<(), Error>;
    fn write_fmt(&mut self, fmt: fmt::Arguments) -> Result<(), Error>;
}

Result<..., Error> xuất hiện rất nhiều. Do đó, std::io có khai báo type alias như sau:

use std::fmt;

type Result<T> = std::result::Result<T, std::io::Error>;

pub trait Write {
    fn write(&mut self, buf: &[u8]) -> Result<usize>;
    fn flush(&mut self) -> Result<()>;

    fn write_all(&mut self, buf: &[u8]) -> Result<()>;
    fn write_fmt(&mut self, fmt: fmt::Arguments) -> Result<()>;
}

Vì khai báo này nằm trong module std::io, ta có thể dùng alias đầy đủ std::io::Result<T> — tức Result<T, E> với E là std::io::Error. Signature của các hàm trong trait Write trở nên gọn hơn như sau:

use std::fmt;

type Result<T> = std::result::Result<T, std::io::Error>;

pub trait Write {
    fn write(&mut self, buf: &[u8]) -> Result<usize>;
    fn flush(&mut self) -> Result<()>;

    fn write_all(&mut self, buf: &[u8]) -> Result<()>;
    fn write_fmt(&mut self, fmt: fmt::Arguments) -> Result<()>;
}

Type alias giúp ích theo hai cách: nó làm cho code dễ viết hơn và cho ta một interface nhất quán trên toàn bộ std::io. Vì nó là alias, nó vẫn chỉ là Result<T, E>, nghĩa là ta có thể dùng tất cả các method hoạt động trên Result<T, E> với nó, bao gồm cú pháp đặc biệt như toán tử ?.

Kiểu Never — Không Bao Giờ Trả Về

Rust có một kiểu đặc biệt tên !, được gọi trong lý thuyết kiểu là empty type (kiểu rỗng) vì nó không có giá trị nào. Ta thích gọi nó là never type vì nó đứng ở vị trí kiểu trả về khi một hàm sẽ không bao giờ trả về. Đây là một ví dụ:

fn bar() -> ! {
    // --snip--
    panic!();
}

Code này được đọc là "hàm bar trả về never." Các hàm không bao giờ trả về được gọi là diverging functions. Ta không thể tạo ra giá trị của kiểu !, vì vậy bar không bao giờ có thể trả về được.

Nhưng kiểu mà bạn không bao giờ tạo ra giá trị thì có ích gì? Hãy nhớ lại code từ Listing 2-5, một phần của trò chơi đoán số; ta đã tái hiện một đoạn ở đây trong Listing 20-27.

Lúc đó, ta đã bỏ qua một số chi tiết trong code này. Trong phần "The match Control Flow Construct" ở Chương 6, ta đã thảo luận rằng các arm của match phải trả về cùng một kiểu. Vậy ví dụ, code sau đây không hoạt động:

fn main() {
    let guess = "3";
    let guess = match guess.trim().parse() {
        Ok(_) => 5,
        Err(_) => "hello",
    };
}

Kiểu của guess trong code này sẽ phải vừa là integer vừa là string, nhưng Rust yêu cầu guess chỉ có một kiểu. Vậy continue trả về gì? Làm thế nào ta có thể trả về u32 từ một arm và có arm khác kết thúc bằng continue trong Listing 20-27?

Như bạn có thể đoán, continue có giá trị !. Nghĩa là, khi Rust tính kiểu của guess, nó nhìn vào cả hai arm — arm đầu với giá trị u32 và arm sau với giá trị !. Vì ! không bao giờ có giá trị, Rust quyết định kiểu của guess là u32.

Cách diễn đạt chính thức hành vi này là các expression có kiểu ! có thể được ép buộc thành bất kỳ kiểu nào khác. Ta được phép kết thúc arm match bằng continue vì continue không trả về giá trị; thay vào đó, nó đưa control trở lại đầu vòng lặp, vì vậy trong trường hợp Err, ta không bao giờ gán giá trị cho guess.

Never type cũng hữu ích với macro panic!. Hãy nhớ lại hàm unwrap mà ta gọi trên các giá trị Option<T> để lấy ra giá trị hoặc panic với định nghĩa này:

enum Option<T> {
    Some(T),
    None,
}

use crate::Option::*;

impl<T> Option<T> {
    pub fn unwrap(self) -> T {
        match self {
            Some(val) => val,
            None => panic!("called `Option::unwrap()` on a `None` value"),
        }
    }
}

Trong code này, điều tương tự xảy ra như trong match ở Listing 20-27: Rust thấy val có kiểu T và panic! có kiểu !, vì vậy kết quả của biểu thức match tổng thể là T. Code này hoạt động vì panic! không tạo ra giá trị; nó kết thúc chương trình. Trong trường hợp None, ta sẽ không trả về giá trị từ unwrap, vì vậy code này hợp lệ.

Một biểu thức cuối cùng có kiểu ! là vòng lặp loop:

fn main() {
    print!("forever ");

    loop {
        print!("and ever ");
    }
}

Ở đây, vòng lặp không bao giờ kết thúc, vì vậy ! là giá trị của biểu thức. Tuy nhiên, điều này sẽ không đúng nếu ta thêm break, vì vòng lặp sẽ kết thúc khi đến lệnh break.

Dynamically Sized Types và Trait Sized

Rust cần biết một số chi tiết về các kiểu của nó, chẳng hạn như cần bao nhiêu bộ nhớ để cấp phát cho một giá trị của một kiểu cụ thể. Điều này khiến một góc trong hệ thống kiểu của nó hơi khó hiểu lúc đầu: khái niệm về dynamically sized types. Đôi khi được gọi là DSTs hay unsized types, các kiểu này cho phép ta viết code sử dụng các giá trị mà kích thước của chúng chỉ có thể biết vào lúc runtime.

Hãy đào sâu vào chi tiết của một dynamically sized type gọi là str, mà ta đã dùng xuyên suốt cuốn sách. Đúng vậy, không phải &str, mà là str đứng một mình, là DST. Trong nhiều trường hợp, chẳng hạn khi lưu trữ văn bản người dùng nhập vào, ta không thể biết chuỗi dài bao nhiêu cho đến lúc runtime. Điều đó có nghĩa là ta không thể tạo biến kiểu str, cũng không thể nhận tham số kiểu str. Hãy xem code sau, nó không hoạt động:

fn main() {
    let s1: str = "Hello there!";
    let s2: str = "How's it going?";
}

Rust cần biết cần bao nhiêu bộ nhớ để cấp phát cho bất kỳ giá trị nào của một kiểu cụ thể, và tất cả giá trị của một kiểu phải dùng cùng một lượng bộ nhớ. Nếu Rust cho phép ta viết code này, hai giá trị str này sẽ cần chiếm cùng một lượng không gian. Nhưng chúng có độ dài khác nhau: s1 cần 12 byte và s2 cần 15 byte. Đó là lý do tại sao không thể tạo biến giữ dynamically sized type.

Vậy phải làm gì? Trong trường hợp này, bạn đã biết câu trả lời: Ta làm cho kiểu của s1 và s2 là string slice (&str) thay vì str. Hãy nhớ lại từ phần "String Slices" ở Chương 4 rằng cấu trúc dữ liệu slice chỉ lưu trữ vị trí bắt đầu và độ dài của slice. Vì vậy, mặc dù &T là một giá trị đơn lưu trữ địa chỉ bộ nhớ nơi T nằm, một string slice là hai giá trị: địa chỉ của str và độ dài của nó. Như vậy, ta có thể biết kích thước của một giá trị string slice tại compile time: Nó gấp đôi độ dài của usize. Tức là, ta luôn biết kích thước của string slice, bất kể chuỗi nó trỏ đến dài bao nhiêu. Tổng quát, đây là cách dynamically sized types được dùng trong Rust: Chúng có thêm một mẩu metadata lưu trữ kích thước của thông tin động. Quy tắc vàng của dynamically sized types là ta phải luôn đặt giá trị của dynamically sized types sau một con trỏ nào đó.

Ta có thể kết hợp str với tất cả các loại con trỏ: ví dụ, Box<str> hay Rc<str>. Thực ra, bạn đã thấy điều này trước đây nhưng với dynamically sized type khác: traits. Mọi trait đều là dynamically sized type mà ta có thể tham chiếu bằng cách dùng tên của trait. Trong phần "Using Trait Objects to Abstract over Shared Behavior" ở Chương 18, ta đã đề cập rằng để dùng trait như trait objects, ta phải đặt chúng sau một con trỏ, chẳng hạn &dyn Trait hay Box<dyn Trait> (Rc<dyn Trait> cũng được).

Để làm việc với DSTs, Rust cung cấp trait Sized để xác định xem kích thước của một kiểu có được biết tại compile time hay không. Trait này được tự động implement cho mọi thứ có kích thước biết tại compile time. Ngoài ra, Rust ngầm thêm một bound trên Sized vào mọi generic function. Nghĩa là, một định nghĩa generic function như sau:

fn generic<T>(t: T) {
    // --snip--
}

thực ra được xử lý như thể ta đã viết:

fn generic<T: Sized>(t: T) {
    // --snip--
}

Theo mặc định, generic functions chỉ hoạt động với các kiểu có kích thước biết tại compile time. Tuy nhiên, bạn có thể dùng cú pháp đặc biệt sau để nới lỏng ràng buộc này:

fn generic<T: ?Sized>(t: &T) {
    // --snip--
}

Trait bound trên ?Sized có nghĩa là "T có thể có hoặc không có Sized," và ký hiệu này ghi đè mặc định rằng generic types phải có kích thước biết tại compile time. Cú pháp ?Trait với ý nghĩa này chỉ có sẵn cho Sized, không áp dụng cho các trait khác.

Cũng lưu ý rằng ta đã đổi kiểu của tham số t từ T thành &T. Vì kiểu có thể không phải Sized, ta cần dùng nó sau một loại con trỏ nào đó. Trong trường hợp này, ta chọn một reference.

Tiếp theo, ta sẽ nói về functions và closures!
