Lưu trữ Keys kèm Values trong Hash Map

Collection phổ biến cuối cùng là hash map. Kiểu HashMap<K, V> lưu trữ ánh xạ từ các key kiểu K sang các value kiểu V bằng cách dùng một hashing function, thứ quyết định cách đặt key và value vào bộ nhớ. Nhiều ngôn ngữ lập trình hỗ trợ kiểu cấu trúc dữ liệu này, nhưng thường dưới tên khác như hash, map, object, hash table, dictionary, hay associative array, chỉ kể một vài cái thôi.

Hash map hữu ích khi bạn muốn tra cứu dữ liệu không phải bằng index như với vector, mà bằng một key có thể thuộc bất kỳ kiểu nào. Ví dụ, trong một trò chơi, bạn có thể theo dõi điểm của mỗi đội trong một hash map, trong đó mỗi key là tên đội và value là điểm số tương ứng. Biết tên đội, bạn lấy được điểm của họ.

Ta sẽ đi qua API cơ bản của hash map trong phần này, nhưng còn nhiều điều thú vị hơn trong các hàm được định nghĩa trên HashMap<K, V> bởi thư viện chuẩn. Như thường lệ, hãy tra tài liệu thư viện chuẩn để biết thêm.

Tạo một Hash Map mới

Một cách để tạo hash map rỗng là dùng new rồi thêm phần tử bằng insert. Ở Listing 8-20, ta theo dõi điểm của hai đội tên là Blue và Yellow. Đội Blue bắt đầu với 10 điểm, đội Yellow bắt đầu với 50 điểm.

Lưu ý là ta cần use HashMap từ phần collections của thư viện chuẩn trước. Trong ba collection phổ biến của ta, đây là cái ít được dùng nhất, nên nó không được đưa vào scope tự động qua prelude. Hash map cũng được thư viện chuẩn hỗ trợ ít hơn; ví dụ không có macro built-in để tạo chúng.

Giống vector, hash map lưu dữ liệu trên heap. HashMap này có key kiểu String và value kiểu i32. Cũng như vector, hash map là đồng nhất: tất cả các key phải cùng kiểu, và tất cả các value phải cùng kiểu.

Truy cập Values trong Hash Map

Ta có thể lấy value từ hash map bằng cách truyền key vào method get, như Listing 8-21 dưới đây.

Ở đây, score sẽ có giá trị gắn với đội Blue, tức là 10. Method get trả về Option<&V>; nếu không có value nào cho key đó trong hash map, get sẽ trả về None. Chương trình này xử lý Option bằng cách gọi copied để lấy Option<i32> thay vì Option<&i32>, rồi dùng unwrap_or để gán score bằng 0 nếu scores không có entry cho key đó.

Ta có thể duyệt qua từng cặp key-value trong hash map tương tự như với vector, dùng vòng lặp for:

fn main() {
    use std::collections::HashMap;

    let mut scores = HashMap::new();

    scores.insert(String::from("Blue"), 10);
    scores.insert(String::from("Yellow"), 50);

    for (key, value) in &scores {
        println!("{key}: {value}");
    }
}

Đoạn code này sẽ in từng cặp theo thứ tự tùy ý:

Yellow: 50
Blue: 10

Quản lý Ownership trong Hash Map

Với các kiểu implement trait Copy như i32, các value sẽ được copy vào hash map. Với các owned value như String, các value sẽ bị move vào và hash map sẽ là owner của chúng, như minh họa ở Listing 8-22.

Ta không thể dùng các biến field_name và field_value sau khi chúng đã bị move vào hash map qua lời gọi insert.

Nếu ta insert tham chiếu đến các value vào hash map, các value đó sẽ không bị move. Các value mà tham chiếu trỏ đến phải hợp lệ ít nhất là trong khoảng thời gian hash map còn hợp lệ. Ta sẽ nói thêm về những vấn đề này trong "Validating References with Lifetimes" ở Chương 10.

Cập nhật Hash Map

Mặc dù số lượng cặp key-value có thể tăng, mỗi key duy nhất chỉ có thể có một value gắn với nó tại một thời điểm (nhưng không phải ngược lại: ví dụ, cả đội Blue lẫn Yellow đều có thể có value 10 trong hash map scores).

Khi bạn muốn thay đổi dữ liệu trong hash map, bạn phải quyết định cách xử lý trường hợp key đã có value. Bạn có thể thay giá trị cũ bằng giá trị mới, bỏ qua hoàn toàn giá trị cũ. Hoặc giữ giá trị cũ, bỏ qua giá trị mới, chỉ thêm khi key chưa có value. Hay kết hợp giá trị cũ và mới. Hãy xem cách làm từng trường hợp!

Ghi đè một Value

Nếu ta insert một key và value vào hash map rồi insert cùng key đó với value khác, value gắn với key đó sẽ bị thay thế. Dù đoạn code ở Listing 8-23 gọi insert hai lần, hash map chỉ chứa một cặp key-value vì ta đang insert value cho cùng một key của đội Blue cả hai lần.

Đoạn code này sẽ in ra {"Blue": 25}. Giá trị ban đầu là 10 đã bị ghi đè.

Thêm Key-Value chỉ khi Key chưa tồn tại

Một tình huống hay gặp là kiểm tra xem một key đã tồn tại trong hash map với một value chưa, rồi thực hiện: nếu key đó đã có value thì giữ nguyên; nếu chưa có thì mới insert key và value.

Hash map có một API đặc biệt cho việc này tên là entry, nhận key bạn muốn kiểm tra làm tham số. Giá trị trả về của method entry là một enum tên Entry đại diện cho một value có thể tồn tại hoặc không. Giả sử ta muốn kiểm tra xem key của đội Yellow có value gắn với nó chưa. Nếu chưa, ta muốn insert value 50, và tương tự với đội Blue. Dùng API entry, code trông như Listing 8-24.

Method or_insert trên Entry được định nghĩa để trả về mutable reference đến value của key Entry tương ứng nếu key đó tồn tại, còn nếu không thì insert tham số làm value mới cho key này và trả về mutable reference đến value mới. Kỹ thuật này gọn gàng hơn nhiều so với tự viết logic, và ngoài ra còn phối hợp tốt hơn với borrow checker.

Chạy code ở Listing 8-24 sẽ in ra {"Yellow": 50, "Blue": 10}. Lần gọi entry đầu tiên sẽ insert key của đội Yellow với value 50 vì Yellow chưa có value. Lần gọi entry thứ hai sẽ không thay đổi hash map vì đội Blue đã có value 10 rồi.

Cập nhật Value dựa trên Value cũ

Một trường hợp hay dùng khác của hash map là tra cứu value của một key rồi cập nhật nó dựa trên giá trị cũ. Ví dụ, Listing 8-25 cho thấy code đếm số lần mỗi từ xuất hiện trong một đoạn văn bản. Ta dùng hash map với các từ làm key và tăng value để theo dõi số lần đã gặp từ đó. Nếu là lần đầu tiên gặp từ đó, ta sẽ insert value 0 trước.

Đoạn code này sẽ in ra {"world": 2, "hello": 1, "wonderful": 1}. Bạn có thể thấy các cặp key-value được in theo thứ tự khác: Nhớ lại từ "Truy cập Values trong Hash Map" rằng việc duyệt qua hash map xảy ra theo thứ tự tùy ý.

Method split_whitespace trả về iterator trên các subslice, phân tách bởi khoảng trắng, của value trong text. Method or_insert trả về mutable reference (&mut V) đến value của key được chỉ định. Ở đây, ta lưu mutable reference đó vào biến count, vì vậy để gán cho value đó, ta phải dereference count trước bằng dấu sao (*). Mutable reference sẽ ra khỏi scope ở cuối vòng lặp for, nên tất cả các thay đổi này đều an toàn và được phép bởi borrowing rules.

Hashing Functions

Mặc định, HashMap dùng hashing function gọi là SipHash, có khả năng chống lại các tấn công denial-of-service (DoS) nhắm vào hash table1. Đây không phải thuật toán hashing nhanh nhất hiện có, nhưng sự đánh đổi giữa bảo mật tốt hơn và hiệu suất thấp hơn là xứng đáng. Nếu bạn profile code và thấy hashing function mặc định quá chậm cho mục đích của bạn, bạn có thể chuyển sang hàm khác bằng cách chỉ định một hasher khác. Một hasher là kiểu implement trait BuildHasher. Ta sẽ bàn về trait và cách implement chúng ở Chương 10. Bạn không nhất thiết phải tự cài đặt hasher từ đầu; crates.io có các thư viện do cộng đồng Rust chia sẻ, cung cấp hashers cài đặt nhiều thuật toán hashing phổ biến.

Tổng kết

Vector, string, và hash map sẽ cung cấp lượng lớn chức năng cần thiết trong các chương trình khi bạn cần lưu trữ, truy cập và chỉnh sửa dữ liệu. Dưới đây là một số bài tập bạn giờ đã có thể giải:

  1. Cho một danh sách số nguyên, dùng vector để tìm và trả về median (giá trị ở giữa khi đã sắp xếp) và mode (giá trị xuất hiện nhiều nhất; hash map sẽ hữu ích ở đây) của danh sách.
  2. Chuyển đổi string sang Pig Latin. Phụ âm đầu tiên của mỗi từ được chuyển ra cuối từ và thêm ay, ví dụ first trở thành irst-fay. Các từ bắt đầu bằng nguyên âm thì thêm hay vào cuối (apple trở thành apple-hay). Nhớ tính đến các chi tiết về mã hóa UTF-8!
  3. Dùng hash map và vector, tạo giao diện văn bản cho phép người dùng thêm tên nhân viên vào phòng ban trong công ty; ví dụ "Add Sally to Engineering" hoặc "Add Amir to Sales." Sau đó cho người dùng lấy danh sách tất cả mọi người trong một phòng ban hoặc tất cả mọi người trong công ty theo phòng ban, được sắp xếp theo thứ tự bảng chữ cái.

Tài liệu API của thư viện chuẩn mô tả các method mà vector, string và hash map có, rất hữu ích cho các bài tập này!

Ta đang bước vào các chương trình phức tạp hơn, nơi các thao tác có thể thất bại, vì thế đây là lúc hoàn hảo để bàn về xử lý lỗi. Ta sẽ làm điều đó tiếp theo!
