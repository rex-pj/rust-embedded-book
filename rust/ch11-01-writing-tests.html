Cách Viết Test

Test là những hàm Rust có nhiệm vụ xác minh rằng code không phải test đang vận hành đúng như kỳ vọng. Phần thân của các hàm test thường thực hiện ba việc sau:

  • Thiết lập dữ liệu hoặc trạng thái cần thiết.
  • Chạy đoạn code bạn muốn test.
  • Khẳng định rằng kết quả đúng như bạn mong đợi.

Hãy cùng nhìn qua các tính năng mà Rust cung cấp riêng cho việc viết test, bao gồm attribute test, một số macro, và attribute should_panic.

Cấu Trúc Của Một Hàm Test

Đơn giản nhất, một test trong Rust là một hàm được đính kèm attribute test. Attribute là metadata gắn kèm với các đoạn code Rust; một ví dụ quen thuộc là attribute derive mà ta dùng với struct ở Chương 5. Để biến một hàm thành hàm test, hãy thêm #[test] vào dòng trước fn. Khi bạn chạy test bằng lệnh cargo test, Rust sẽ xây dựng một file binary test runner để chạy các hàm được đánh dấu đó và báo cáo xem từng hàm test có pass hay fail.

Mỗi khi ta tạo project thư viện mới bằng Cargo, một module test với một hàm test mẫu sẽ tự động được sinh ra cho ta. Module này cung cấp sẵn một khung xương để viết test, giúp ta không phải tra cứu cấu trúc và cú pháp chính xác mỗi lần bắt đầu project mới. Bạn có thể thêm bao nhiêu hàm test và module test tùy thích!

Ta sẽ khám phá một số khía cạnh về cách test hoạt động bằng cách thử nghiệm với hàm test mẫu trước khi thực sự test bất kỳ đoạn code nào. Sau đó, ta sẽ viết một số test thực tế gọi code mà ta đã viết và khẳng định hành vi của nó là đúng.

Hãy tạo một project thư viện mới tên adder sẽ cộng hai số:

$ cargo new adder --lib
     Created library `adder` project
$ cd adder

Nội dung file src/lib.rs trong thư viện adder của bạn trông sẽ như Listing 11-1.

File bắt đầu với một hàm add mẫu để ta có thứ gì đó để test.

Trước mắt, hãy tập trung vào hàm it_works. Chú ý annotation #[test]: attribute này chỉ ra đây là một hàm test, nhờ đó test runner biết phải xử lý hàm này như một test. Ta cũng có thể có các hàm không phải test trong module tests để hỗ trợ thiết lập các kịch bản chung hay thực hiện các thao tác thông thường, nên ta luôn cần chỉ rõ hàm nào là test.

Phần thân hàm mẫu dùng macro assert_eq! để khẳng định rằng result — chứa kết quả của add(2, 2) — bằng 4. Assertion này là ví dụ điển hình cho định dạng của một test thông thường. Hãy chạy thử để xem test này có pass không.

Lệnh cargo test chạy tất cả các test trong project, như được hiển thị ở Listing 11-2.

Cargo biên dịch và chạy test. Ta thấy dòng running 1 test. Dòng tiếp theo hiển thị tên hàm test được sinh ra, gọi là tests::it_works, và kết quả của lần chạy test đó là ok. Dòng tổng kết test result: ok. có nghĩa là tất cả test đã pass, và phần 1 passed; 0 failed tổng hợp số test đã pass hay fail.

Ta có thể đánh dấu một test là ignored để nó không chạy trong một trường hợp nhất định; ta sẽ tìm hiểu ở phần "Bỏ qua Test trừ Khi Được Yêu Cầu Rõ Ràng" ở phần sau của chương này. Vì ta chưa làm điều đó ở đây, phần tổng kết hiển thị 0 ignored. Ta cũng có thể truyền một đối số cho lệnh cargo test để chỉ chạy các test có tên khớp với một chuỗi ký tự; đây gọi là filtering (lọc), và ta sẽ tìm hiểu ở phần "Chạy Một Tập Hợp Con Các Test Theo Tên". Ở đây ta chưa lọc test nào, nên cuối phần tổng kết hiển thị 0 filtered out.

Thống kê 0 measured dành cho benchmark test đo lường hiệu suất. Benchmark test, tính đến thời điểm viết bài này, chỉ có trong Rust nightly. Xem tài liệu về benchmark test để tìm hiểu thêm.

Phần tiếp theo của kết quả test bắt đầu từ Doc-tests adder là cho kết quả của các documentation test. Ta chưa có documentation test nào, nhưng Rust có thể biên dịch các ví dụ code xuất hiện trong tài liệu API của ta. Tính năng này giúp đồng bộ docs và code! Ta sẽ bàn về cách viết documentation test ở phần "Documentation Comments as Tests" ở Chương 14. Hiện tại, ta sẽ bỏ qua phần output Doc-tests.

Hãy bắt đầu tùy chỉnh test theo nhu cầu của mình. Trước tiên, đổi tên hàm it_works thành tên khác, chẳng hạn exploration, như sau:

Tên file: src/lib.rs

pub fn add(left: u64, right: u64) -> u64 {
    left + right
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn exploration() {
        let result = add(2, 2);
        assert_eq!(result, 4);
    }
}

Giờ ta sẽ thêm một test nữa, nhưng lần này ta sẽ cố tình làm cho nó fail! Test fail khi có gì đó trong hàm test gây ra panic. Mỗi test được chạy trong một thread mới, và khi thread chính thấy một thread test đã chết, test đó được đánh dấu là failed. Ở Chương 9, ta đã nói về cách đơn giản nhất để panic là gọi macro panic!. Nhập test mới vào như một hàm tên another, để file src/lib.rs của bạn trông như Listing 11-3.

Thay vì ok, dòng test tests::another hiển thị FAILED. Hai phần mới xuất hiện giữa kết quả từng test và phần tổng kết: Phần đầu tiên hiển thị chi tiết lý do mỗi test fail. Trong trường hợp này, ta thấy tests::another fail vì nó panic với thông báo Make this test fail ở dòng 17 trong file src/lib.rs. Phần tiếp theo chỉ liệt kê tên của tất cả test fail, hữu ích khi có nhiều test và nhiều kết quả fail dài dòng. Ta có thể dùng tên của test fail để chạy chỉ riêng test đó để debug dễ hơn; ta sẽ nói thêm về các cách chạy test ở phần "Kiểm Soát Cách Chạy Test".

Dòng tổng kết hiển thị ở cuối: Nhìn chung, kết quả test là FAILED. Ta có một test pass và một test fail.

Giờ khi bạn đã thấy kết quả test trong các tình huống khác nhau, hãy cùng xem một số macro khác ngoài panic! hữu ích trong test.

Kiểm Tra Kết Quả Với assert!

Macro assert!, được cung cấp bởi thư viện chuẩn, rất hữu ích khi bạn muốn đảm bảo rằng một điều kiện nào đó trong test cho kết quả là true. Ta truyền vào macro assert! một đối số mà khi tính toán cho ra giá trị Boolean. Nếu giá trị là true, không có gì xảy ra và test pass. Nếu giá trị là false, macro assert! sẽ gọi panic! để làm test fail. Sử dụng macro assert! giúp ta kiểm tra xem code có đang hoạt động đúng như ta muốn không.

Ở Chương 5, Listing 5-15, ta đã dùng struct Rectangle và method can_hold, được nhắc lại ở Listing 11-5. Hãy đặt code này vào file src/lib.rs, rồi viết một số test cho nó dùng macro assert!.

Method can_hold trả về Boolean, nghĩa là nó là trường hợp dùng macro assert! hoàn hảo. Ở Listing 11-6, ta viết một test kiểm tra method can_hold bằng cách tạo một instance Rectangle có chiều rộng 8 và chiều cao 7, rồi khẳng định nó có thể chứa một instance Rectangle khác có chiều rộng 5 và chiều cao 1.

Chú ý dòng use super::*; bên trong module tests. Module tests là một module bình thường tuân theo các quy tắc visibility thông thường mà ta đã tìm hiểu ở Chương 7 trong phần "Paths for Referring to an Item in the Module Tree". Vì module tests là module con, ta cần đưa code từ module cha vào scope của module con. Ta dùng glob ở đây để mọi thứ ta định nghĩa trong module cha đều có sẵn trong module tests này.

Ta đặt tên test là larger_can_hold_smaller, và tạo hai instance Rectangle cần thiết. Rồi ta gọi macro assert! và truyền kết quả của larger.can_hold(&smaller). Biểu thức này được cho là trả về true, nên test của ta sẽ pass. Hãy xem thử!

Tên file: src/lib.rs

#[derive(Debug)]
struct Rectangle {
    width: u32,
    height: u32,
}

impl Rectangle {
    fn can_hold(&self, other: &Rectangle) -> bool {
        self.width > other.width && self.height > other.height

}

Kiểm Tra Tính Bằng Nhau Với assert_eq! và assert_ne!

Một cách phổ biến để xác minh chức năng là kiểm tra sự bằng nhau giữa kết quả của code đang được test và giá trị bạn kỳ vọng code trả về. Bạn có thể làm điều đó dùng macro assert! với biểu thức sử dụng toán tử ==. Tuy nhiên, đây là kiểu test phổ biến đến mức thư viện chuẩn cung cấp một cặp macro — assert_eq! và assert_ne! — để thực hiện test này thuận tiện hơn. Các macro này so sánh hai đối số xem có bằng nhau hay không bằng nhau, tương ứng. Chúng cũng sẽ in ra hai giá trị nếu assertion fail, giúp dễ dàng thấy tại sao test fail; ngược lại, macro assert! chỉ cho biết nó nhận được giá trị false cho biểu thức ==, mà không in ra các giá trị dẫn đến kết quả false đó.

Ở Listing 11-7, ta viết một hàm tên add_two cộng thêm 2 vào tham số của nó, rồi test hàm này bằng macro assert_eq!.

Hãy kiểm tra xem nó có pass không!

Ta tạo biến result chứa kết quả của add_two(2). Rồi ta truyền result và 4 làm đối số cho macro assert_eq!. Dòng output cho test này là test tests::it_adds_two ... ok, và chữ ok cho biết test đã pass!

Hãy nhét một bug vào code để xem assert_eq! trông thế nào khi fail. Đổi phần triển khai hàm add_two để cộng 3 thay vì:

pub fn add_two(a: u64) -> u64 {
    a + 3
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn it_adds_two() {
        let result = add_two(2);
        assert_eq!(result, 4);
    }
}

Test của ta đã bắt được bug! Test tests::it_adds_two fail, và thông báo cho ta biết assertion bị fail là left == right cùng với giá trị left và right. Thông báo này giúp ta bắt đầu debug: đối số left — nơi ta có kết quả của add_two(2) — là 5, nhưng đối số right là 4. Bạn có thể hình dung điều này sẽ đặc biệt hữu ích khi ta có nhiều test chạy cùng lúc.

Lưu ý rằng trong một số ngôn ngữ và framework test, các tham số của hàm assertion bằng nhau được gọi là expected và actual, và thứ tự chỉ định đối số rất quan trọng. Tuy nhiên, trong Rust, chúng được gọi là left và right, và thứ tự chỉ định giá trị kỳ vọng và giá trị code tạo ra không quan trọng. Ta có thể viết assertion trong test này là assert_eq!(4, result), kết quả fail sẽ hiển thị assertion `left == right` failed tương tự.

Macro assert_ne! sẽ pass nếu hai giá trị ta truyền vào không bằng nhau và fail nếu chúng bằng nhau. Macro này hữu ích nhất khi ta không chắc giá trị sẽ là gì, nhưng biết chắc giá trị không được phép là gì. Ví dụ, nếu ta đang test một hàm được đảm bảo thay đổi đầu vào theo một cách nào đó, nhưng cách thay đổi phụ thuộc vào ngày trong tuần chạy test, điều tốt nhất để khẳng định có thể là output của hàm không bằng input.

Về cốt lõi, macro assert_eq! và assert_ne! sử dụng các toán tử == và != tương ứng. Khi assertion fail, các macro này in ra các đối số của chúng dùng định dạng debug, nghĩa là các giá trị được so sánh phải implement trait PartialEq và Debug. Tất cả các primitive type và hầu hết các type trong thư viện chuẩn đều implement các trait này. Với struct và enum tự định nghĩa, bạn sẽ cần implement PartialEq để khẳng định tính bằng nhau của các type đó. Bạn cũng cần implement Debug để in ra giá trị khi assertion fail. Vì cả hai trait đều là derivable trait, như đề cập ở Listing 5-12 Chương 5, điều này thường đơn giản như thêm annotation #[derive(PartialEq, Debug)] vào định nghĩa struct hoặc enum. Xem Phụ lục C, "Derivable Traits," để biết thêm chi tiết về các trait này và các trait derivable khác.

Thêm Thông Báo Fail Tùy Chỉnh

Bạn cũng có thể thêm thông báo tùy chỉnh để in kèm thông báo fail như các đối số tùy chọn cho macro assert!, assert_eq!, và assert_ne!. Bất kỳ đối số nào được chỉ định sau các đối số bắt buộc đều được chuyển tới macro format! (đã bàn ở phần "Nối chuỗi với + hoặc format!" trong Chương 8), nên bạn có thể truyền một format string chứa placeholder {} và các giá trị để điền vào. Thông báo tùy chỉnh hữu ích để ghi lại ý nghĩa của assertion; khi test fail, bạn sẽ có ý tưởng tốt hơn về vấn đề với code là gì.

Ví dụ, giả sử ta có một hàm chào người dùng theo tên và ta muốn test xem tên được truyền vào hàm có xuất hiện trong output không:

Tên file: src/lib.rs

pub fn greeting(name: &str) -> String {
    format!("Hello {name}!")
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn greeting_contains_name() {
        let result = greeting("Carol");
        assert!(result.contains("Carol"));
    }
}

Các yêu cầu cho chương trình này vẫn chưa được thống nhất, và ta khá chắc phần chữ Hello ở đầu lời chào sẽ thay đổi. Ta quyết định không muốn phải cập nhật test khi yêu cầu thay đổi, nên thay vì kiểm tra tính bằng nhau chính xác với giá trị trả về từ hàm greeting, ta chỉ khẳng định rằng output chứa văn bản của tham số đầu vào.

Bây giờ hãy nhét bug vào code bằng cách đổi greeting để không bao gồm name, để xem thông báo fail mặc định trông thế nào:

Kiểm Tra Panic Với should_panic

Ngoài việc kiểm tra giá trị trả về, điều quan trọng là phải kiểm tra rằng code của ta xử lý các điều kiện lỗi đúng như ta kỳ vọng. Ví dụ, hãy xét kiểu Guess mà ta đã tạo ở Chương 9, Listing 9-13. Code khác sử dụng Guess phụ thuộc vào đảm bảo rằng các instance Guess sẽ chỉ chứa các giá trị từ 1 đến 100. Ta có thể viết một test đảm bảo rằng cố gắng tạo instance Guess với giá trị ngoài khoảng đó sẽ gây panic.

Ta làm điều này bằng cách thêm attribute should_panic vào hàm test. Test pass nếu code bên trong hàm gây panic; test fail nếu code bên trong không gây panic.

Listing 11-8 cho thấy một test kiểm tra rằng các điều kiện lỗi của Guess::new xảy ra khi ta kỳ vọng.

Ta đặt attribute #[should_panic] sau attribute #[test] và trước hàm test nó áp dụng. Hãy xem kết quả khi test này pass:

Test dùng should_panic có thể không chính xác. Test should_panic sẽ pass ngay cả khi test panic vì lý do khác với lý do ta kỳ vọng. Để làm test should_panic chính xác hơn, ta có thể thêm tham số expected tùy chọn cho attribute should_panic. Test harness sẽ đảm bảo thông báo fail chứa đoạn text được cung cấp. Test này sẽ pass vì giá trị ta đặt vào tham số expected của attribute should_panic là chuỗi con của thông báo panic từ hàm Guess::new. Ta có thể chỉ định toàn bộ thông báo panic mà ta kỳ vọng, trong trường hợp này là Guess value must be less than or equal to 100, got 200. Tùy thuộc vào mức độ riêng biệt hay động của thông báo panic và mức độ chính xác bạn muốn test của mình. Trong trường hợp này, chuỗi con của thông báo panic là đủ để đảm bảo code trong hàm test thực thi nhánh else if value > 100.

Để xem chuyện gì xảy ra khi test should_panic với thông báo expected fail, hãy lại nhét bug vào code bằng cách đổi thân của khối if value < 1 và else if value > 100:

pub struct Guess {
    value: i32,
}

impl Guess {
    pub fn new(value: i32) -> Guess {
        if value < 1 {
            panic!(
                "Guess value must be less than or equal to 100, got {value}."
            );
        } else if value > 100 {
            panic!(
                "Guess value must be greater than or equal to 1, got {value}."
            );
        }

        Guess { value }
    }
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    #[should_panic(expected = "less than or equal to 100")]
    fn greater_than_100() {
        Guess::new(200);
    }
}

Dùng Result<T, E> Trong Test

Tất cả các test trước đây của ta đều panic khi fail. Ta cũng có thể viết test dùng Result<T, E>! Đây là test từ Listing 11-1, được viết lại để dùng Result<T, E> và trả về Err thay vì panic:

pub fn add(left: u64, right: u64) -> u64 {
    left + right
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn it_works() -> Result<(), String> {
        let result = add(2, 2);

        if result == 4 {
            Ok(())
        } else {
            Err(String::from("two plus two does not equal four"))
        }
    }
}

Hàm it_works giờ có kiểu trả về là Result<(), String>. Trong thân hàm, thay vì gọi macro assert_eq!, ta trả về Ok(()) khi test pass và Err với String bên trong khi test fail.

Viết test để trả về Result<T, E> cho phép bạn dùng toán tử dấu hỏi trong thân test, đây là cách thuận tiện để viết các test sẽ fail nếu bất kỳ thao tác nào bên trong chúng trả về biến thể Err.

Bạn không thể dùng annotation #[should_panic] trên các test dùng Result<T, E>. Để khẳng định rằng một thao tác trả về biến thể Err, đừng dùng toán tử dấu hỏi trên giá trị Result<T, E>. Thay vào đó, hãy dùng assert!(value.is_err()).

Giờ khi bạn đã biết nhiều cách viết test, hãy cùng xem điều gì đang xảy ra khi ta chạy test và khám phá các tùy chọn khác nhau ta có thể dùng với cargo test.
