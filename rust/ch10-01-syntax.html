Kiểu Dữ Liệu Generic

Ta dùng generics để tạo ra các định nghĩa cho những thứ như chữ ký hàm hay struct, rồi có thể dùng chúng với nhiều kiểu dữ liệu cụ thể khác nhau. Hãy cùng xem cách định nghĩa hàm, struct, enum, và method bằng generics. Sau đó chúng mình sẽ bàn xem generics ảnh hưởng tới hiệu suất code ra sao.

Trong Định Nghĩa Hàm

Khi định nghĩa một hàm dùng generics, ta đặt generics vào trong chữ ký hàm ở những chỗ thường ta ghi rõ kiểu dữ liệu của tham số và giá trị trả về. Làm vậy giúp code linh hoạt hơn và cung cấp nhiều chức năng hơn cho người gọi hàm, đồng thời tránh trùng lặp code.

Tiếp tục với hàm largest, Listing 10-4 cho thấy hai hàm đều tìm giá trị lớn nhất trong một slice. Sau đó ta sẽ gộp chúng lại thành một hàm duy nhất dùng generics.

Hàm largest_i32 chính là hàm ta đã tách ra ở Listing 10-3, tìm giá trị i32 lớn nhất trong slice. Hàm largest_char tìm giá trị char lớn nhất trong slice. Thân hai hàm hoàn toàn giống nhau, vậy ta hãy khử trùng lặp bằng cách đưa vào một tham số kiểu generic trong một hàm duy nhất.

Để tham số hóa kiểu trong một hàm mới duy nhất, ta cần đặt tên cho tham số kiểu, giống như ta làm với tham số giá trị của hàm. Bạn có thể dùng bất kỳ tên nào làm tên tham số kiểu. Nhưng theo quy ước, tên tham số kiểu trong Rust thường ngắn gọn — hay chỉ một chữ cái — và dùng UpperCamelCase. Viết tắt của type, T là lựa chọn mặc định của hầu hết dân Rust.

Khi ta dùng một tham số trong thân hàm, ta phải khai báo tên tham số đó trong chữ ký để trình biên dịch biết cái tên đó nghĩa là gì. Tương tự, khi ta dùng tên tham số kiểu trong chữ ký hàm, ta phải khai báo tên tham số kiểu trước khi dùng. Để định nghĩa hàm largest generic, ta đặt phần khai báo tên kiểu trong cặp dấu ngoặc nhọn <>, giữa tên hàm và danh sách tham số, như thế này:

fn largest<T>(list: &[T]) -> &T {

Ta đọc định nghĩa này là: "Hàm largest là generic trên một kiểu nào đó là T." Hàm này có một tham số tên là list, là một slice các giá trị kiểu T. Hàm largest sẽ trả về tham chiếu đến một giá trị cùng kiểu T.

Listing 10-5 cho thấy định nghĩa hàm largest đã được gộp lại dùng kiểu dữ liệu generic trong chữ ký. Listing cũng cho thấy ta có thể gọi hàm với slice i32 hoặc char. Lưu ý rằng code này chưa compile được.

Nếu compile code này ngay lúc này, ta sẽ thấy lỗi sau:

$ cargo run
   Compiling chapter10 v0.1.0 (file:///projects/chapter10)
error[E0369]: binary operation `>` cannot be applied to type `&T`
 --> src/main.rs:5:17
  |
5 |         if item > largest {
  |            ---- ^ ------- &T
  |            |
  |            &T
  |
help: consider restricting type parameter `T` with trait `PartialOrd`
  |
1 | fn largest<T: std::cmp::PartialOrd>(list: &[T]) -> &T {
  |             ++++++++++++++++++++++

For more information about this error, try `rustc --explain E0369`.
error: could not compile `chapter10` (bin "chapter10") due to 1 previous error

Dòng gợi ý nhắc đến std::cmp::PartialOrd, đó là một trait — ta sẽ nói kỹ hơn ở phần sau. Hiện tại cứ hiểu là lỗi này nói rằng thân hàm largest sẽ không hoạt động với tất cả các kiểu mà T có thể là. Vì ta muốn so sánh các giá trị kiểu T, ta chỉ có thể dùng những kiểu mà giá trị của chúng có thể được sắp xếp thứ tự. Để kích hoạt tính năng so sánh, standard library có trait std::cmp::PartialOrd mà bạn có thể implement trên các kiểu (xem Phụ lục C để biết thêm). Để sửa Listing 10-5, ta có thể làm theo gợi ý của trình biên dịch và ràng buộc các kiểu hợp lệ cho T chỉ bao gồm những kiểu implement PartialOrd. Lúc đó code sẽ compile được, vì standard library implement PartialOrd cho cả i32 lẫn char.

Trong Định Nghĩa Struct

Ta cũng có thể định nghĩa struct dùng tham số kiểu generic cho một hoặc nhiều trường bằng cú pháp <>. Listing 10-6 định nghĩa struct Point<T> để chứa tọa độ x và y với giá trị thuộc bất kỳ kiểu nào.

Cú pháp dùng generics trong định nghĩa struct tương tự cú pháp trong định nghĩa hàm. Đầu tiên, khai báo tên tham số kiểu bên trong cặp ngoặc nhọn ngay sau tên struct. Rồi dùng kiểu generic đó trong định nghĩa struct ở những chỗ mà lẽ ra ta phải chỉ rõ kiểu cụ thể.

Lưu ý rằng vì ta chỉ dùng một kiểu generic để định nghĩa Point<T>, định nghĩa này nói rằng struct Point<T> là generic trên kiểu T, và cả hai trường x lẫn y phải cùng một kiểu — dù kiểu đó là gì đi nữa. Nếu ta tạo một instance Point<T> với các giá trị khác kiểu nhau như ở Listing 10-7, code sẽ không compile.

Trong ví dụ này, khi ta gán giá trị integer 5 cho x, ta cho trình biên dịch biết kiểu generic T sẽ là integer cho instance Point<T> này. Sau đó khi ta chỉ định 4.0 cho y — mà ta đã định nghĩa phải cùng kiểu với x — ta sẽ nhận được lỗi kiểu không khớp:

$ cargo run
   Compiling chapter10 v0.1.0 (file:///projects/chapter10)
error[E0308]: mismatched types
 --> src/main.rs:7:38
  |
7 |     let wont_work = Point { x: 5, y: 4.0 };
  |                                      ^^^ expected integer, found floating-point number

For more information about this error, try `rustc --explain E0308`.
error: could not compile `chapter10` (bin "chapter10") due to 1 previous error

Để định nghĩa struct Point với x và y đều là generic nhưng có thể khác kiểu nhau, ta dùng nhiều tham số kiểu generic. Ví dụ ở Listing 10-8, ta thay đổi định nghĩa Point để generic trên kiểu T và U, trong đó x có kiểu T còn y có kiểu U.

Bây giờ tất cả các instance Point trên đều được chấp nhận! Ta có thể dùng bao nhiêu tham số kiểu generic tùy ý, nhưng dùng quá nhiều sẽ làm code khó đọc. Nếu thấy mình cần quá nhiều kiểu generic, đó có thể là dấu hiệu code cần được tái cấu trúc thành các mảnh nhỏ hơn.

Trong Định Nghĩa Enum

Cũng giống như struct, ta có thể định nghĩa enum chứa kiểu dữ liệu generic trong các variant của nó. Hãy nhìn lại enum Option<T> mà standard library cung cấp, ta đã dùng ở Chương 6:

#![allow(unused)]
fn main() {
enum Option<T> {
    Some(T),
    None,
}
}

Định nghĩa này giờ đây hẳn là rõ ràng hơn với bạn rồi. Như bạn thấy, enum Option<T> là generic trên kiểu T và có hai variant: Some chứa một giá trị kiểu T, và None không chứa giá trị gì cả. Nhờ dùng enum Option<T>, ta có thể diễn đạt khái niệm trừu tượng về một giá trị tùy chọn (có thể có hoặc không), và vì Option<T> là generic, ta có thể dùng abstraction này cho mọi kiểu dữ liệu.

Enum cũng có thể dùng nhiều kiểu generic. Định nghĩa enum Result ta đã dùng ở Chương 9 là một ví dụ:

#![allow(unused)]
fn main() {
enum Result<T, E> {
    Ok(T),
    Err(E),
}
}

Enum Result là generic trên hai kiểu, T và E, có hai variant: Ok chứa giá trị kiểu T, và Err chứa giá trị kiểu E. Định nghĩa này giúp dùng enum Result ở bất cứ đâu ta có một tác vụ có thể thành công (trả về giá trị kiểu T) hoặc thất bại (trả về lỗi kiểu E). Đây chính xác là thứ ta đã dùng để mở file ở Listing 9-3, khi T được điền bằng kiểu std::fs::File khi mở file thành công và E được điền bằng std::io::Error khi gặp sự cố.

Khi bạn nhận ra trong code có nhiều định nghĩa struct hoặc enum chỉ khác nhau ở kiểu của các giá trị chúng chứa, hãy tránh trùng lặp bằng cách dùng kiểu generic thay thế.

Trong Định Nghĩa Method

Ta có thể implement các method trên struct và enum (như ta đã làm ở Chương 5) và cũng dùng kiểu generic trong định nghĩa chúng. Listing 10-9 cho thấy struct Point<T> từ Listing 10-6 với một method tên x được implement trên nó.

Ở đây, ta đã định nghĩa method x trên Point<T> trả về tham chiếu đến dữ liệu trong trường x.

Lưu ý rằng ta phải khai báo T ngay sau impl để có thể dùng T chỉ rõ rằng ta đang implement các method trên kiểu Point<T>. Bằng cách khai báo T là kiểu generic sau impl, Rust có thể nhận diện rằng kiểu trong cặp ngoặc nhọn ở Point là kiểu generic chứ không phải kiểu cụ thể. Method được định nghĩa trong một impl khai báo kiểu generic sẽ có mặt trên mọi instance của kiểu đó, bất kể kiểu cụ thể nào được điền vào.

Ta cũng có thể chỉ định ràng buộc trên kiểu generic khi định nghĩa method. Ví dụ, ta có thể implement method chỉ trên các instance Point<f32> chứ không phải mọi instance Point<T> bất kỳ. Ở Listing 10-10, ta dùng kiểu cụ thể f32, nghĩa là ta không khai báo thêm kiểu nào sau impl.

Code này có nghĩa là kiểu Point<f32> sẽ có method distance_from_origin; các instance Point<T> khác với T không phải f32 sẽ không có method này. Method này tính khoảng cách từ điểm của ta đến tọa độ gốc (0.0, 0.0) và dùng các phép toán chỉ hoạt động với kiểu số thực.

Các tham số kiểu generic trong định nghĩa struct không nhất thiết phải giống với những cái dùng trong chữ ký method của struct đó. Listing 10-11 dùng các kiểu generic X1 và Y1 cho struct Point, còn X2 và Y2 cho chữ ký method mixup để làm rõ hơn. Method tạo ra một instance Point mới với giá trị x từ self Point (kiểu X1) và giá trị y từ Point được truyền vào (kiểu Y2).

Trong main, ta định nghĩa một Point với i32 cho x (giá trị 5) và f64 cho y (giá trị 10.4). Biến p2 là một Point struct với string slice cho x (giá trị "Hello") và char cho y (giá trị c). Gọi mixup trên p1 với đối số p2 cho ta p3, sẽ có i32 cho x vì x đến từ p1. Biến p3 sẽ có char cho y vì y đến từ p2. Macro println! sẽ in ra p3.x = 5, p3.y = c.

Mục đích của ví dụ này là để minh họa tình huống trong đó một số tham số generic được khai báo với impl còn một số khác được khai báo với định nghĩa method. Ở đây, tham số generic X1 và Y1 được khai báo sau impl vì chúng gắn với định nghĩa struct. Còn X2 và Y2 được khai báo sau fn mixup vì chúng chỉ liên quan đến method.

Hiệu Suất Code Dùng Generics

Bạn có thể đang thắc mắc liệu dùng tham số kiểu generic có tốn chi phí runtime không. Tin vui là dùng kiểu generic sẽ không làm chương trình của bạn chạy chậm hơn so với dùng kiểu cụ thể.

Rust thực hiện điều này bằng cách thực hiện monomorphization của code dùng generics lúc compile. Monomorphization là quá trình biến code generic thành code cụ thể bằng cách điền vào các kiểu cụ thể được dùng lúc biên dịch. Trong quá trình này, trình biên dịch làm ngược lại với các bước ta đã dùng để tạo hàm generic ở Listing 10-5: nó nhìn vào tất cả các chỗ code generic được gọi và sinh ra code cho các kiểu cụ thể mà code generic đó được gọi với.

Hãy xem cách này hoạt động bằng cách dùng enum generic Option<T> từ standard library:

#![allow(unused)]
fn main() {
let integer = Some(5);
let float = Some(5.0);
}

Khi Rust compile code này, nó thực hiện monomorphization. Trong quá trình đó, trình biên dịch đọc các giá trị đã dùng trong các instance Option<T> và nhận ra có hai loại Option<T>: một là i32 và một là f64. Vậy là nó mở rộng định nghĩa generic của Option<T> thành hai định nghĩa chuyên biệt cho i32 và f64, thay thế định nghĩa generic bằng các định nghĩa cụ thể.

Phiên bản monomorphized của code trông kiểu như sau (trình biên dịch dùng tên khác với ví dụ này):

Generic Option<T> được thay thế bằng các định nghĩa cụ thể do trình biên dịch tạo ra. Vì Rust compile code generic thành code chỉ rõ kiểu trong từng instance, ta không phải trả chi phí runtime nào khi dùng generics. Khi code chạy, nó hoạt động y như thể ta đã viết tay từng định nghĩa riêng. Quá trình monomorphization làm cho generics của Rust cực kỳ hiệu quả ở runtime.
