Kiểm Soát Cách Chạy Test

Cũng như cargo run biên dịch code rồi chạy file binary kết quả, cargo test biên dịch code ở chế độ test rồi chạy file binary test kết quả. Hành vi mặc định của file binary được tạo ra bởi cargo test là chạy tất cả các test song song và bắt output được sinh ra trong quá trình chạy test, ngăn không cho output hiển thị và giúp dễ đọc output liên quan đến kết quả test hơn. Tuy nhiên, bạn có thể chỉ định các tùy chọn dòng lệnh để thay đổi hành vi mặc định này.

Một số tùy chọn dòng lệnh dành cho cargo test, một số khác dành cho file binary test kết quả. Để phân biệt hai loại đối số này, bạn liệt kê các đối số cho cargo test trước, rồi dùng dấu phân tách --, rồi mới đến các đối số cho file binary test. Chạy cargo test --help để xem các tùy chọn bạn có thể dùng với cargo test, và chạy cargo test -- --help để xem các tùy chọn sau dấu phân tách. Các tùy chọn này cũng được ghi lại trong phần "Tests" của The rustc Book.

Chạy Test Song Song Hay Tuần Tự

Khi bạn chạy nhiều test, theo mặc định chúng chạy song song dùng các thread, có nghĩa là chúng kết thúc chạy nhanh hơn và bạn nhận phản hồi sớm hơn. Vì các test chạy cùng một lúc, bạn phải đảm bảo các test không phụ thuộc vào nhau hoặc vào bất kỳ trạng thái dùng chung nào, bao gồm môi trường dùng chung, chẳng hạn như thư mục làm việc hiện tại hay các biến môi trường.

Ví dụ, giả sử mỗi test chạy một đoạn code tạo một file trên đĩa tên test-output.txt và ghi một số dữ liệu vào file đó. Rồi mỗi test đọc dữ liệu trong file đó và khẳng định file chứa một giá trị cụ thể, khác nhau trong mỗi test. Vì các test chạy cùng lúc, một test có thể ghi đè file trong khoảng thời gian giữa lúc test khác đang viết và đọc file. Test thứ hai sẽ fail, không phải vì code sai mà vì các test đã xen vào nhau khi chạy song song. Một giải pháp là đảm bảo mỗi test viết vào một file khác; giải pháp khác là chạy test lần lượt từng cái một.

Nếu bạn không muốn chạy test song song hoặc muốn kiểm soát chính xác hơn số thread được dùng, bạn có thể truyền cờ --test-threads và số thread bạn muốn dùng cho file binary test. Hãy xem ví dụ sau:

$ cargo test -- --test-threads=1

Ta đặt số thread test thành 1, báo cho chương trình không dùng bất kỳ song song nào. Chạy test dùng một thread sẽ mất nhiều thời gian hơn so với chạy song song, nhưng các test sẽ không xen vào nhau nếu chúng chia sẻ trạng thái.

Hiển Thị Output Của Hàm

Theo mặc định, nếu test pass, thư viện test của Rust bắt bất kỳ thứ gì được in ra stdout. Ví dụ, nếu ta gọi println! trong test và test pass, ta sẽ không thấy output println! trong terminal; ta chỉ thấy dòng cho biết test đã pass. Nếu test fail, ta sẽ thấy bất kỳ thứ gì đã được in ra stdout cùng với phần còn lại của thông báo fail.

Ví dụ, Listing 11-10 có một hàm ngớ ngẩn in ra giá trị tham số của nó và trả về 10, cùng với một test sẽ pass và một test sẽ fail.

Khi ta chạy các test này bằng cargo test, ta sẽ thấy output sau:

$ cargo test

    Finished `test` profile [unoptimized + debuginfo] target(s) in 0.58s

Lưu ý rằng không có chỗ nào trong output này ta thấy I got the value 4, dòng được in khi test pass chạy. Output đó đã bị bắt giữ. Output từ test fail, I got the value 8, xuất hiện trong phần output tổng kết test, cùng với nguyên nhân fail của test.

Nếu ta muốn thấy các giá trị được in cho cả test pass, ta có thể báo cho Rust cũng hiển thị output của test thành công bằng --show-output:

$ cargo test -- --show-output

Khi ta chạy lại các test trong Listing 11-10 với cờ --show-output, ta thấy output sau:

Chạy Một Tập Hợp Con Các Test Theo Tên

Đôi khi chạy toàn bộ test suite có thể mất nhiều thời gian. Nếu bạn đang làm việc trên code trong một khu vực cụ thể, bạn có thể chỉ muốn chạy các test liên quan đến code đó. Bạn có thể chọn test nào chạy bằng cách truyền cho cargo test tên hoặc danh sách tên của test bạn muốn chạy làm đối số.

Để minh họa cách chạy một tập hợp con test, trước tiên ta sẽ tạo ba test cho hàm add_two như trong Listing 11-11, và chọn test nào muốn chạy.

Nếu ta chạy test mà không truyền đối số nào, như ta thấy trước đó, tất cả test sẽ chạy song song:

$ cargo test
   Compiling adder v0.1.0 (file:///projects/adder)
    Finished `test` profile [unoptimized + debuginfo] target(s) in 0.62s
     Running unittests src/lib.rs (target/debug/deps/adder-92948b65e88960b4)

running 3 tests
test tests::add_three_and_two ... ok
test tests::add_two_and_two ... ok
test tests::one_hundred ... ok

test result: ok. 3 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out; finished in 0.00s

   Doc-tests adder

running 0 tests

test result: ok. 0 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out; finished in 0.00s

Chạy Một Test Duy Nhất

Ta có thể truyền tên của bất kỳ hàm test nào cho cargo test để chỉ chạy test đó:

$ cargo test one_hundred
   Compiling adder v0.1.0 (file:///projects/adder)
    Finished `test` profile [unoptimized + debuginfo] target(s) in 0.69s
     Running unittests src/lib.rs (target/debug/deps/adder-92948b65e88960b4)

running 1 test
test tests::one_hundred ... ok

test result: ok. 1 passed; 0 failed; 0 ignored; 0 measured; 2 filtered out; finished in 0.00s

Chỉ test có tên one_hundred chạy; hai test kia không khớp với tên đó. Output test cho ta biết ta còn nhiều test không chạy bằng cách hiển thị 2 filtered out ở cuối.

Ta không thể chỉ định tên của nhiều test theo cách này; chỉ giá trị đầu tiên được truyền cho cargo test sẽ được dùng. Nhưng có cách để chạy nhiều test.

Lọc Để Chạy Nhiều Test

Ta có thể chỉ định một phần tên test, và bất kỳ test nào có tên khớp với giá trị đó sẽ được chạy. Ví dụ, vì hai test của ta có tên chứa add, ta có thể chạy cả hai bằng lệnh cargo test add:

$ cargo test add
   Compiling adder v0.1.0 (file:///projects/adder)
    Finished `test` profile [unoptimized + debuginfo] target(s) in 0.61s
     Running unittests src/lib.rs (target/debug/deps/adder-92948b65e88960b4)

running 2 tests
test tests::add_three_and_two ... ok
test tests::add_two_and_two ... ok

test result: ok. 2 passed; 0 failed; 0 ignored; 0 measured; 1 filtered out; finished in 0.00s

Lệnh này chạy tất cả test có add trong tên và lọc ra test tên one_hundred. Cũng lưu ý rằng module mà test xuất hiện trong đó trở thành một phần tên của test, nên ta có thể chạy tất cả các test trong một module bằng cách lọc theo tên module.

Bỏ Qua Test Trừ Khi Được Yêu Cầu Rõ Ràng

Đôi khi một số test cụ thể có thể mất rất nhiều thời gian để thực thi, nên bạn có thể muốn loại chúng ra trong hầu hết các lần chạy cargo test. Thay vì liệt kê tất cả test bạn muốn chạy làm đối số, bạn có thể đánh dấu các test tốn thời gian bằng attribute ignore để loại trừ chúng, như sau:

Tên file: src/lib.rs

pub fn add(left: u64, right: u64) -> u64 {
    left + right
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn it_works() {
        let result = add(2, 2);
        assert_eq!(result, 4);
    }

    #[test]
    #[ignore]
    fn expensive_test() {
        // code that takes an hour to run
    }
}

Sau #[test], ta thêm dòng #[ignore] vào test ta muốn loại trừ. Giờ khi ta chạy test, it_works chạy, nhưng expensive_test thì không:

$ cargo test
   Compiling adder v0.1.0 (file:///projects/adder)
    Finished `test` profile [unoptimized + debuginfo] target(s) in 0.60s
     Running unittests src/lib.rs (target/debug/deps/adder-92948b65e88960b4)

running 2 tests
test tests::expensive_test ... ignored
test tests::it_works ... ok

test result: ok. 1 passed; 0 failed; 1 ignored; 0 measured; 0 filtered out; finished in 0.00s

   Doc-tests adder

running 0 tests

test result: ok. 0 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out; finished in 0.00s

Hàm expensive_test được liệt kê là ignored. Nếu ta muốn chỉ chạy các test bị ignored, ta có thể dùng cargo test -- --ignored:

$ cargo test -- --ignored
   Compiling adder v0.1.0 (file:///projects/adder)
    Finished `test` profile [unoptimized + debuginfo] target(s) in 0.61s
     Running unittests src/lib.rs (target/debug/deps/adder-92948b65e88960b4)

running 1 test
test tests::expensive_test ... ok

test result: ok. 1 passed; 0 failed; 0 ignored; 0 measured; 1 filtered out; finished in 0.00s

   Doc-tests adder

running 0 tests

test result: ok. 0 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out; finished in 0.00s

Bằng cách kiểm soát test nào chạy, bạn có thể đảm bảo kết quả cargo test của mình được trả về nhanh chóng. Khi bạn ở điểm mà việc kiểm tra kết quả của các test ignored là hợp lý và bạn có thời gian chờ kết quả, bạn có thể chạy cargo test -- --ignored thay vào đó. Nếu bạn muốn chạy tất cả test dù có bị ignored hay không, bạn có thể chạy cargo test -- --include-ignored.
