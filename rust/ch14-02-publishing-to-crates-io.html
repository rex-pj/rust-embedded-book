Xuất bản Crate lên Crates.io

Chúng ta đã dùng các package từ crates.io như là dependencies của dự án — nhưng bạn cũng có thể chia sẻ code của mình với người khác bằng cách xuất bản package riêng. Registry crate tại crates.io phân phối source code của các package, nên về cơ bản nó chuyên host code mã nguồn mở.

Rust và Cargo có những tính năng giúp package bạn xuất bản dễ tìm và dễ dùng hơn. Chúng ta sẽ nói về một số tính năng đó rồi giải thích cách xuất bản package.

Viết Documentation Comments Hữu Ích

Tài liệu chính xác cho package của bạn giúp người dùng biết cách và khi nào nên dùng, vì vậy đầu tư thời gian viết documentation là điều rất xứng đáng. Ở Chương 3, chúng ta đã thảo luận về cách comment code Rust bằng hai dấu gạch chéo //. Rust còn có một loại comment đặc biệt dành cho tài liệu, được gọi một cách tiện lợi là documentation comment, giúp tạo ra tài liệu HTML. HTML này hiển thị nội dung documentation comments cho các public API — nhắm đến những lập trình viên muốn biết cách dùng crate của bạn, thay vì cách nó được triển khai.

Documentation comments dùng ba dấu gạch chéo /// thay vì hai, và hỗ trợ ký hiệu Markdown để định dạng văn bản. Đặt documentation comments ngay trước item mà chúng mô tả. Listing 14-1 minh họa documentation comments cho hàm add_one trong crate tên my_crate.

Ở đây, chúng ta mô tả hàm add_one làm gì, tạo section với tiêu đề Examples, rồi cung cấp code minh họa cách dùng. Chúng ta có thể tạo tài liệu HTML từ documentation comment này bằng cách chạy cargo doc. Lệnh này chạy công cụ rustdoc đi kèm với Rust và đặt tài liệu HTML được tạo ra vào thư mục target/doc.

Tiện hơn nữa, chạy cargo doc --open sẽ build HTML cho tài liệu của crate hiện tại (cùng với tài liệu của tất cả dependencies) và mở kết quả trong trình duyệt web. Điều hướng đến hàm add_one và bạn sẽ thấy nội dung trong documentation comments được hiển thị như thế nào, như trong Hình 14-1.

Hình 14-1: Tài liệu HTML cho hàm add_one

Các Section Thường Dùng

Chúng ta đã dùng tiêu đề Markdown # Examples trong Listing 14-1 để tạo section trong HTML với tiêu đề "Examples". Dưới đây là một số section khác mà tác giả crate hay dùng trong tài liệu:

  • Panics: Các tình huống mà hàm được mô tả có thể panic. Người gọi hàm mà không muốn chương trình panic cần đảm bảo họ không gọi hàm trong những tình huống này.
  • Errors: Nếu hàm trả về Result, mô tả các loại lỗi có thể xảy ra và điều kiện nào gây ra lỗi sẽ rất hữu ích cho người gọi, giúp họ viết code xử lý các loại lỗi khác nhau theo cách phù hợp.
  • Safety: Nếu hàm là unsafe để gọi (chúng ta sẽ thảo luận về unsafe ở Chương 20), cần có một section giải thích tại sao hàm là unsafe và liệt kê các invariant mà hàm mong đợi người gọi tuân thủ.

Hầu hết documentation comments không cần tất cả các section này, nhưng đây là danh sách kiểm tra tốt để nhắc bạn về những khía cạnh mà người dùng code của bạn muốn biết.

Documentation Comments như là Tests

Thêm code block ví dụ vào documentation comments giúp minh họa cách dùng thư viện, và còn có một bonus thú vị nữa: chạy cargo test sẽ chạy các code example trong tài liệu của bạn như test! Không gì tốt hơn tài liệu có ví dụ. Nhưng cũng không gì tệ hơn ví dụ không chạy được vì code đã thay đổi từ khi tài liệu được viết. Nếu ta chạy cargo test với tài liệu cho hàm add_one từ Listing 14-1, ta sẽ thấy section trong kết quả test trông như thế này:

   Doc-tests my_crate

running 1 test
test src/lib.rs - add_one (line 5) ... ok

test result: ok. 1 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out; finished in 0.27s

Giờ nếu ta thay đổi hàm hoặc ví dụ sao cho assert_eq! trong ví dụ panic, rồi chạy cargo test lại, ta sẽ thấy doc tests bắt được rằng ví dụ và code đã lệch nhau!

Comment cho Item Chứa

Kiểu doc comment //! thêm tài liệu vào item chứa các comment đó, thay vì vào các item theo sau comment. Thông thường ta dùng loại doc comment này bên trong file crate root (src/lib.rs theo quy ước) hoặc bên trong một module để tài liệu hóa crate hoặc module đó như một tổng thể.

Ví dụ, để thêm tài liệu mô tả mục đích của crate my_crate — crate chứa hàm add_one — ta thêm documentation comments bắt đầu bằng //! vào đầu file src/lib.rs, như trong Listing 14-2.

Chú ý rằng không có code nào sau dòng cuối cùng bắt đầu bằng //!. Vì ta bắt đầu comment bằng //! thay vì ///, ta đang tài liệu hóa item chứa comment này chứ không phải item theo sau nó. Trong trường hợp này, item đó là file src/lib.rs — crate root. Các comment này mô tả toàn bộ crate.

Khi chạy cargo doc --open, các comment này sẽ hiển thị trên trang chủ của tài liệu cho my_crate phía trên danh sách các public item trong crate, như trong Hình 14-2.

Documentation comments bên trong các item rất hữu ích khi mô tả crate và module. Dùng chúng để giải thích mục đích tổng thể của container, giúp người dùng hiểu cách tổ chức của crate.

Hình 14-2: Tài liệu được render cho my_crate, bao gồm comment mô tả toàn bộ crate

Xuất khẩu Public API Tiện Lợi

Cấu trúc public API của bạn là một yếu tố quan trọng khi xuất bản crate. Người dùng crate của bạn ít quen thuộc với cấu trúc hơn bạn, và có thể gặp khó khăn khi tìm kiếm các phần họ muốn dùng nếu crate của bạn có một hệ thống phân cấp module phức tạp.

Ở Chương 7, chúng ta đã đề cập đến cách làm item công khai bằng từ khóa pub, và cách đưa item vào scope với từ khóa use. Tuy nhiên, cấu trúc hợp lý với bạn trong quá trình phát triển crate có thể không tiện lợi cho người dùng. Bạn có thể muốn tổ chức struct theo hệ thống phân cấp nhiều cấp, nhưng những người muốn dùng một type được định nghĩa sâu trong phân cấp có thể khó tìm ra type đó tồn tại. Họ cũng có thể bực mình khi phải nhập use my_crate::some_module::another_module::UsefulType; thay vì use my_crate::UsefulType;.

Tin vui là nếu cấu trúc không tiện lợi cho người dùng khi dùng từ thư viện khác, bạn không cần phải sắp xếp lại tổ chức nội bộ: Thay vào đó, bạn có thể re-export các item để tạo ra một cấu trúc công khai khác với cấu trúc nội bộ bằng cách dùng pub use. Re-exporting lấy một public item ở một vị trí và làm nó công khai ở một vị trí khác, như thể nó được định nghĩa ở vị trí kia vậy.

Ví dụ, giả sử ta tạo một thư viện tên art để mô hình hóa các khái niệm nghệ thuật. Trong thư viện này có hai module: module kinds chứa hai enum tên PrimaryColor và SecondaryColor, và module utils chứa hàm tên mix, như trong Listing 14-3.

Hình 14-3 cho thấy trang chủ của tài liệu cho crate này được tạo bởi cargo doc trông như thế nào.

Hình 14-3: Trang chủ tài liệu cho art liệt kê module kinds và utils

Lưu ý rằng các type PrimaryColor và SecondaryColor không được liệt kê trên trang chủ, hàm mix cũng không. Phải click vào kinds và utils mới thấy được.

Một crate khác phụ thuộc vào thư viện này sẽ cần các câu lệnh use để đưa item từ art vào scope, chỉ định cấu trúc module hiện tại. Listing 14-4 minh họa một crate dùng item PrimaryColor và mix từ crate art.

Tác giả của code trong Listing 14-4 phải tìm ra rằng PrimaryColor nằm trong module kinds và mix nằm trong module utils. Cấu trúc module của crate art thiết thực hơn với các developer đang làm việc trên crate art hơn là những người dùng nó. Cấu trúc nội bộ không chứa thông tin hữu ích cho người muốn hiểu cách dùng crate art, mà còn gây bối rối vì developer phải tự tìm đúng chỗ và phải khai báo module name trong câu lệnh use.

Để loại bỏ tổ chức nội bộ khỏi public API, ta có thể sửa code crate art trong Listing 14-3 để thêm các câu lệnh pub use nhằm re-export các item lên top level, như trong Listing 14-5.

Tài liệu API mà cargo doc tạo ra cho crate này giờ sẽ liệt kê và liên kết các re-export trên trang chủ, như trong Hình 14-4, giúp các type PrimaryColor và SecondaryColor cùng hàm mix dễ tìm hơn nhiều.

Hình 14-4: Trang chủ tài liệu cho art liệt kê các re-export

Người dùng crate art vẫn có thể thấy và dùng cấu trúc nội bộ từ Listing 14-3 như trong Listing 14-4, hoặc họ có thể dùng cấu trúc tiện lợi hơn trong Listing 14-5, như trong Listing 14-6.

Trong những trường hợp có nhiều module lồng nhau, re-export các type lên top level bằng pub use có thể tạo ra sự khác biệt đáng kể cho trải nghiệm của người dùng crate. Một cách dùng phổ biến khác của pub use là re-export định nghĩa của một dependency trong crate hiện tại để làm các định nghĩa đó trở thành một phần của public API của crate bạn.

Tạo ra cấu trúc public API hữu ích là một nghệ thuật hơn là khoa học, và bạn có thể lặp đi lặp lại để tìm ra API phù hợp nhất với người dùng. Chọn pub use cho bạn sự linh hoạt trong cách cấu trúc crate nội bộ và tách biệt cấu trúc nội bộ đó với những gì bạn trình bày ra ngoài. Hãy xem code của một số crate bạn đã cài để xem liệu cấu trúc nội bộ của chúng có khác với public API không.

Thiết Lập Tài Khoản Crates.io

Trước khi có thể xuất bản bất kỳ crate nào, bạn cần tạo tài khoản trên crates.io và lấy API token. Để làm vậy, ghé trang chủ tại crates.io và đăng nhập bằng tài khoản GitHub. (Tài khoản GitHub hiện là yêu cầu bắt buộc, nhưng trang web có thể hỗ trợ các cách tạo tài khoản khác trong tương lai.) Sau khi đăng nhập, ghé cài đặt tài khoản tại https://crates.io/me/ và lấy API key. Sau đó chạy lệnh cargo login và dán API key của bạn khi được hỏi, như thế này:

$ cargo login
abcdefghijklmnopqrstuvwxyz012345

Lệnh này sẽ thông báo cho Cargo về API token của bạn và lưu nó cục bộ trong ~/.cargo/credentials.toml. Lưu ý rằng token này là bí mật: Đừng chia sẻ nó với bất kỳ ai. Nếu vì lý do nào đó bạn chia sẻ nó với ai, hãy thu hồi và tạo token mới trên crates.io.

Thêm Metadata vào Crate Mới

Giả sử bạn có crate muốn xuất bản. Trước khi xuất bản, bạn cần thêm một số metadata vào section [package] trong file Cargo.toml của crate.

Crate của bạn cần có tên duy nhất. Khi làm việc với crate cục bộ, bạn có thể đặt tên tùy thích. Tuy nhiên, tên crate trên crates.io được cấp theo nguyên tắc ai đến trước được trước. Một khi tên crate đã được đặt, không ai khác có thể xuất bản crate với tên đó. Trước khi thử xuất bản crate, hãy tìm kiếm tên bạn muốn dùng. Nếu tên đó đã được dùng, bạn phải tìm tên khác và chỉnh sửa trường name trong file Cargo.toml dưới section [package] để dùng tên mới khi xuất bản, như sau:

Tên file: Cargo.toml

[package]
name = "guessing_game"

Ngay cả khi bạn đã chọn tên duy nhất, khi chạy cargo publish để xuất bản crate tại thời điểm này, bạn sẽ nhận được cảnh báo rồi một lỗi:

$ cargo publish
    Updating crates.io index
warning: manifest has no description, license, license-file, documentation, homepage or repository.
See https://doc.rust-lang.org/cargo/reference/manifest.html#package-metadata for more info.
--snip--
error: failed to publish to registry at https://crates.io

Caused by:
  the remote server responded with an error (status 400 Bad Request): missing or empty metadata fields: description, license. Please see https://doc.rust-lang.org/cargo/reference/manifest.html for more information on configuring these fields

Lỗi xảy ra vì bạn thiếu một số thông tin quan trọng: Description và license là bắt buộc để người dùng biết crate của bạn làm gì và điều khoản sử dụng là gì. Trong Cargo.toml, thêm description ngắn gọn một vài câu, vì nó sẽ xuất hiện cùng crate trong kết quả tìm kiếm. Đối với trường license, bạn cần cung cấp giá trị định danh license. Linux Foundation's Software Package Data Exchange (SPDX) liệt kê các định danh bạn có thể dùng cho giá trị này. Ví dụ, để xác định bạn đã cấp phép crate bằng MIT License, thêm định danh MIT:

Tên file: Cargo.toml

[package]
name = "guessing_game"
license = "MIT"

Nếu bạn muốn dùng license không có trong SPDX, bạn cần đặt nội dung license đó vào một file, đưa file vào dự án, rồi dùng license-file để chỉ định tên file thay vì dùng key license.

Hướng dẫn về việc chọn license phù hợp cho dự án nằm ngoài phạm vi cuốn sách này. Nhiều người trong cộng đồng Rust cấp phép dự án theo cách tương tự Rust, dùng dual license MIT OR Apache-2.0. Cách này cũng cho thấy bạn có thể chỉ định nhiều định danh license cách nhau bằng OR để áp dụng nhiều license cho dự án.

Với tên duy nhất, version, description và license đã thêm, file Cargo.toml cho dự án sẵn sàng xuất bản có thể trông như thế này:

Tên file: Cargo.toml

[package]
name = "guessing_game"
version = "0.1.0"
edition = "2024"
description = "A fun game where you guess what number the computer has chosen."
license = "MIT OR Apache-2.0"

[dependencies]

Tài liệu của Cargo mô tả các metadata khác mà bạn có thể chỉ định để người khác dễ dàng khám phá và sử dụng crate của bạn hơn.

Xuất Bản lên Crates.io

Giờ bạn đã tạo tài khoản, lưu API token, chọn tên crate và chỉ định metadata cần thiết — bạn đã sẵn sàng xuất bản! Xuất bản crate tải lên một version cụ thể lên crates.io để người khác sử dụng.

Hãy cẩn thận, vì lần xuất bản là vĩnh viễn. Version không bao giờ có thể bị ghi đè, và code không thể bị xóa trừ trong một số trường hợp ngoại lệ. Một mục tiêu lớn của Crates.io là hoạt động như kho lưu trữ code vĩnh viễn, đảm bảo mọi build của dự án phụ thuộc vào crate từ crates.io sẽ tiếp tục hoạt động. Cho phép xóa version sẽ khiến mục tiêu đó bất khả thi. Tuy nhiên, không có giới hạn về số version crate bạn có thể xuất bản.

Xuất Bản Version Mới của Crate Hiện Có

Khi bạn đã thay đổi crate và sẵn sàng phát hành version mới, hãy thay đổi giá trị version trong file Cargo.toml và xuất bản lại. Dùng quy tắc Semantic Versioning để quyết định số version tiếp theo phù hợp dựa trên loại thay đổi bạn đã thực hiện. Sau đó chạy cargo publish để tải lên version mới.

Deprecating Version từ Crates.io

Dù bạn không thể xóa các version cũ của crate, bạn có thể ngăn các dự án tương lai thêm chúng làm dependency mới. Điều này hữu ích khi một version crate bị hỏng vì lý do nào đó. Trong những tình huống như vậy, Cargo hỗ trợ yanking một version crate.

Yanking một version ngăn các dự án mới phụ thuộc vào version đó trong khi vẫn cho phép tất cả dự án hiện tại đang phụ thuộc vào nó tiếp tục hoạt động. Về cơ bản, yank nghĩa là tất cả dự án có Cargo.lock sẽ không bị ảnh hưởng, và mọi file Cargo.lock được tạo trong tương lai sẽ không dùng version đã bị yank.

Để yank một version của crate, trong thư mục crate mà bạn đã xuất bản trước đó, chạy cargo yank và chỉ định version muốn yank. Ví dụ, nếu ta đã xuất bản crate tên guessing_game version 1.0.1 và muốn yank nó, ta chạy lệnh sau trong thư mục dự án guessing_game:

$ cargo yank --vers 1.0.1
    Updating crates.io index
        Yank guessing_game@1.0.1

Bằng cách thêm --undo vào lệnh, bạn cũng có thể hoàn tác yank và cho phép dự án bắt đầu phụ thuộc vào version đó trở lại:

$ cargo yank --vers 1.0.1 --undo
    Updating crates.io index
      Unyank guessing_game@1.0.1

Yank không xóa bất kỳ code nào. Nó không thể xóa, ví dụ, các secret vô tình được tải lên. Nếu điều đó xảy ra, bạn phải reset các secret đó ngay lập tức.
