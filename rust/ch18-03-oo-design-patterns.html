Triển khai Design Pattern Hướng Đối Tượng

State pattern (mẫu trạng thái) là một design pattern hướng đối tượng. Cốt lõi của pattern này là chúng ta định nghĩa một tập hợp các trạng thái mà một giá trị có thể có bên trong. Các trạng thái được đại diện bởi một tập hợp state objects, và hành vi của giá trị thay đổi dựa trên trạng thái của nó. Chúng ta sẽ làm việc qua một ví dụ về một struct blog post có trường để lưu trạng thái của nó, là một state object từ tập hợp "draft" (bản thảo), "review" (đang xem xét) hoặc "published" (đã xuất bản).

Các state objects chia sẻ chức năng: Trong Rust, chúng ta dùng struct và trait thay vì object và kế thừa. Mỗi state object chịu trách nhiệm cho hành vi của chính nó và kiểm soát khi nào nó nên chuyển sang trạng thái khác. Giá trị chứa một state object không biết gì về hành vi khác nhau của các trạng thái hay khi nào cần chuyển đổi giữa các trạng thái.

Ưu điểm của state pattern là khi các yêu cầu nghiệp vụ của chương trình thay đổi, chúng ta không cần thay đổi code của giá trị chứa trạng thái hay code dùng giá trị đó. Chúng ta chỉ cần cập nhật code bên trong một trong các state objects để thay đổi quy tắc của nó, hoặc thêm nhiều state objects hơn.

Định nghĩa Post và Tạo Instance ở Trạng thái Draft

Chúng ta sẽ bắt đầu triển khai thư viện. Chúng ta biết cần một struct Post công khai chứa nội dung, vì vậy chúng ta bắt đầu với định nghĩa struct và hàm new công khai liên quan để tạo một instance của Post:

Trait State định nghĩa hành vi được chia sẻ bởi các trạng thái post khác nhau. Các state objects là Draft, PendingReview và Published, và tất cả đều triển khai trait State. Hiện tại, trait không có phương thức nào, và chúng ta sẽ bắt đầu bằng cách chỉ định nghĩa trạng thái Draft vì đó là trạng thái mà chúng ta muốn một post bắt đầu từ đó.

Khi chúng ta tạo một Post mới, chúng ta đặt trường state của nó thành giá trị Some chứa một Box. Box này trỏ đến một instance mới của struct Draft. Điều này đảm bảo mỗi khi chúng ta tạo một instance mới của Post, nó sẽ bắt đầu như là một bản thảo.

Lưu trữ Text của Nội dung Post

impl Post {
    // --snip--
    pub fn add_text(&mut self, text: &str) {
        self.content.push_str(text);
    }
}

Phương thức add_text nhận một tham chiếu có thể thay đổi tới self vì chúng ta đang thay đổi instance Post mà chúng ta đang gọi add_text trên đó. Sau đó chúng ta gọi push_str trên content và truyền tham số text để thêm vào content đã lưu.

Đảm bảo Nội dung của Draft Post là Rỗng

impl Post {
    // --snip--
    pub fn content(&self) -> &str {
        self.state.as_ref().unwrap().content(self)
    }
}

Chúng ta gọi phương thức as_ref trên Option vì chúng ta muốn một tham chiếu đến giá trị bên trong Option thay vì ownership của giá trị. Vì state là Option<Box<dyn State>>, khi chúng ta gọi as_ref, một Option<&Box<dyn State>> được trả về. Nếu chúng ta không gọi as_ref, chúng ta sẽ gặp lỗi vì chúng ta không thể move state ra khỏi &self mượn của tham số hàm.

Yêu cầu Xem xét Thay đổi Trạng thái của Post

impl Post {
    // --snip--
    pub fn request_review(&mut self) {
        if let Some(s) = self.state.take() {
            self.state = Some(s.request_review())
        }
    }
}

trait State {
    fn request_review(self: Box<Self>) -> Box<dyn State>;
}

struct Draft {}

impl State for Draft {
    fn request_review(self: Box<Self>) -> Box<dyn State> {
        Box::new(PendingReview {})
    }
}

struct PendingReview {}

impl State for PendingReview {
    fn request_review(self: Box<Self>) -> Box<dyn State> {
        self
    }
}

Chúng ta thêm phương thức request_review vào trait State; tất cả các kiểu triển khai trait phải triển khai phương thức request_review. Lưu ý chữ ký của phương thức: self: Box<Self>. Cú pháp này có nghĩa là phương thức chỉ hợp lệ khi được gọi trên một Box chứa kiểu đó. Cú pháp này lấy ownership của Box<Self>, làm cho trạng thái cũ không hợp lệ để giá trị của Post có thể chuyển thành trạng thái mới.

Thêm Phương thức approve Để Thay đổi Hành vi của content

impl Post {
    // --snip--
    pub fn approve(&mut self) {
        if let Some(s) = self.state.take() {
            self.state = Some(s.approve())
        }
    }
}

trait State {
    fn request_review(self: Box<Self>) -> Box<dyn State>;
    fn approve(self: Box<Self>) -> Box<dyn State>;
    fn content<'a>(&self, post: &'a Post) -> &'a str {
        ""
    }
}

struct Published {}

impl State for Published {
    fn request_review(self: Box<Self>) -> Box<dyn State> {
        self
    }

    fn approve(self: Box<Self>) -> Box<dyn State> {
        self
    }

    fn content<'a>(&self, post: &'a Post) -> &'a str {
        &post.content
    }
}

Chúng ta thêm phương thức content vào trait State. Phương thức này trả về một slice chuỗi trống "" như là triển khai mặc định — có nghĩa là chúng ta không cần triển khai content trên các struct Draft và PendingReview. Struct Published sẽ ghi đè phương thức content và trả về giá trị trong post.content.

Sử dụng Đầy đủ

use blog::Post;

fn main() {
    let mut post = Post::new();

    post.add_text("I ate a salad for lunch today");
    assert_eq!("", post.content());

    post.request_review();
    assert_eq!("", post.content());

    post.approve();
    assert_eq!("I ate a salad for lunch today", post.content());
}

Đánh đổi của State Pattern

Chúng ta đã thấy rằng Rust có khả năng triển khai state pattern hướng đối tượng để đóng gói các loại hành vi khác nhau mà một post có thể có trong mỗi trạng thái. Các phương thức trên Post không biết gì về các hành vi khác nhau. Cách chúng ta tổ chức code, chúng ta chỉ cần nhìn vào một nơi để biết các cách khác nhau mà một post đã xuất bản có thể hoạt động: triển khai trait State trên struct Published.

Nếu chúng ta tạo một triển khai thay thế không dùng state pattern, chúng ta có thể dùng các kiểu khác nhau để đại diện cho các trạng thái khác nhau:

pub struct Post {
    content: String,
}

pub struct DraftPost {
    content: String,
}

impl Post {
    pub fn new() -> DraftPost {
        DraftPost {
            content: String::new(),
        }
    }

    pub fn content(&self) -> &str {
        &self.content
    }
}

impl DraftPost {
    pub fn add_text(&mut self, text: &str) {
        self.content.push_str(text);
    }

    pub fn request_review(self) -> PendingReviewPost {
        PendingReviewPost {
            content: self.content,
        }
    }
}

pub struct PendingReviewPost {
    content: String,
}

impl PendingReviewPost {
    pub fn approve(self) -> Post {
        Post {
            content: self.content,
        }
    }
}

Cả DraftPost và PendingReviewPost không có phương thức content, vì vậy code sử dụng chúng không thể gọi content cho đến khi post đạt trạng thái Published. Điều này mã hóa quy trình làm việc vào hệ thống kiểu: compiler sẽ báo lỗi nếu bạn cố lấy nội dung khi bài viết chưa được xuất bản.

Tóm tắt

Dù bạn nghĩ rằng Rust là ngôn ngữ hướng đối tượng hay không, bạn đã thấy rằng có thể dùng trait objects để có một số tính năng hướng đối tượng trong Rust. Dynamic dispatch có thể cho code của bạn một chút linh hoạt với một chút overhead về hiệu suất runtime. Bạn có thể dùng tính linh hoạt đó để triển khai các OOP patterns giúp khả năng bảo trì code của bạn. Rust cũng có các tính năng khác, như ownership, mà ngôn ngữ hướng đối tượng không có. Một OOP pattern không phải lúc nào cũng là cách tốt nhất để tận dụng điểm mạnh của Rust, nhưng nó là một tùy chọn có sẵn.
