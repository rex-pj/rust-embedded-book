Macros

Chúng ta đã dùng các macro như println! xuyên suốt cuốn sách này, nhưng chưa thực sự khám phá macro là gì và hoạt động ra sao. Thuật ngữ macro chỉ một họ tính năng trong Rust — gồm declarative macro với macro_rules! và ba loại procedural macro:

  • Custom #[derive] macro — chỉ định code được thêm vào thông qua attribute derive dùng trên struct và enum
  • Attribute-like macro — định nghĩa attribute tùy chỉnh có thể áp dụng lên bất kỳ item nào
  • Function-like macro — trông như lời gọi hàm nhưng thực ra thao tác trên các token được truyền vào

Chúng ta sẽ nói về từng loại, nhưng trước tiên, hãy cùng hiểu tại sao chúng ta cần macro khi đã có hàm rồi.

Sự khác biệt giữa Macros và Functions

Về bản chất, macro là cách viết code sinh ra code khác — thường được gọi là metaprogramming. Ở Phụ lục C, chúng ta bàn về attribute derive giúp tự động sinh implementation của nhiều trait. Chúng ta cũng đã dùng println! và vec! suốt cuốn sách. Tất cả các macro này đều mở rộng để tạo ra nhiều code hơn những gì bạn viết tay.

Metaprogramming hữu ích để giảm lượng code cần viết và bảo trì — đó cũng là một trong những vai trò của hàm. Nhưng macro còn có thêm những sức mạnh mà hàm không có.

Signature của hàm phải khai báo số lượng và kiểu tham số. Macro thì ngược lại — chúng có thể nhận số lượng tham số tùy ý: gọi println!("hello") với một đối số hay println!("hello {}", name) với hai đối số đều được. Hơn nữa, macro được mở rộng trước khi compiler hiểu ý nghĩa code, nên macro có thể, ví dụ, implement một trait trên một kiểu nhất định. Hàm thì không làm được điều này vì hàm được gọi lúc runtime còn trait cần được implement lúc compile time.

Mặt trái của việc dùng macro thay vì hàm là định nghĩa macro phức tạp hơn định nghĩa hàm — vì bạn đang viết code Rust sinh ra code Rust khác. Do lớp gián tiếp này, định nghĩa macro thường khó đọc, khó hiểu và khó bảo trì hơn.

Một điểm khác biệt quan trọng nữa: bạn phải định nghĩa macro hoặc đưa chúng vào scope trước khi gọi, khác với hàm có thể định nghĩa bất cứ đâu và gọi bất cứ đâu.

Declarative Macros cho Metaprogramming Tổng quát

Dạng macro được dùng phổ biến nhất trong Rust là declarative macro. Chúng đôi khi còn được gọi là "macros by example," "macro_rules! macros," hay đơn giản là "macros." Về cốt lõi, declarative macro cho phép bạn viết thứ gì đó giống biểu thức match của Rust. Như đã thảo luận ở Chương 6, biểu thức match là cấu trúc điều khiển nhận một biểu thức, so sánh giá trị kết quả với các pattern rồi chạy code tương ứng với pattern khớp. Macro cũng so sánh một giá trị với các pattern gắn với code cụ thể: ở đây giá trị là source code Rust literal truyền vào macro; các pattern so sánh với cấu trúc của source code đó; và code gắn với mỗi pattern, khi khớp, thay thế code truyền vào macro. Tất cả điều này xảy ra lúc compile time.

Để định nghĩa macro, bạn dùng construct macro_rules!. Hãy cùng khám phá cách dùng macro_rules! bằng cách xem macro vec! được định nghĩa thế nào. Chương 8 đã đề cập cách dùng vec! để tạo vector với các giá trị cụ thể. Ví dụ, macro sau tạo vector chứa ba số nguyên:

#![allow(unused)]
fn main() {
let v: Vec<u32> = vec![1, 2, 3];
}

Chúng ta cũng có thể dùng vec! để tạo vector hai số nguyên hay vector năm string slice. Không thể dùng hàm để làm điều tương tự vì chúng ta không biết trước số lượng hay kiểu giá trị.

Listing 20-35 là phiên bản đơn giản hóa của định nghĩa macro vec!.

Lưu ý: Định nghĩa thực tế của macro vec! trong thư viện chuẩn có thêm code để cấp phát trước lượng bộ nhớ chính xác. Code đó là tối ưu hóa mà chúng ta bỏ qua ở đây để ví dụ đơn giản hơn.

Annotation #[macro_export] chỉ ra rằng macro này nên được cung cấp bất cứ khi nào crate chứa nó được đưa vào scope. Không có annotation này, macro không thể đưa vào scope được.

Sau đó chúng ta bắt đầu định nghĩa macro với macro_rules! và tên macro không có dấu chấm than. Tên, ở đây là vec, tiếp theo là dấu ngoặc nhọn bao quanh thân macro.

Cấu trúc trong thân vec! giống cấu trúc biểu thức match. Ở đây chúng ta có một arm với pattern ( $( $x:expr ),* ), tiếp theo là => và khối code gắn với pattern này. Nếu pattern khớp, khối code đó sẽ được phát ra. Vì đây là pattern duy nhất trong macro này, chỉ có một cách hợp lệ để khớp; bất kỳ pattern nào khác sẽ gây lỗi. Các macro phức tạp hơn sẽ có nhiều arm hơn.

Cú pháp pattern hợp lệ trong định nghĩa macro khác với cú pháp pattern ở Chương 19 vì macro pattern khớp với cấu trúc code Rust chứ không phải giá trị. Hãy cùng xem qua các phần của pattern trong Listing 20-35; để xem đầy đủ cú pháp macro pattern, hãy tham khảo Rust Reference.

Đầu tiên, chúng ta dùng một cặp ngoặc đơn bao quanh toàn bộ pattern. Chúng ta dùng dấu dollar ($) để khai báo biến trong hệ thống macro sẽ chứa code Rust khớp với pattern. Dấu dollar làm rõ đây là biến macro chứ không phải biến Rust thông thường. Tiếp theo là cặp ngoặc đơn capture các giá trị khớp với pattern bên trong để dùng trong code thay thế. Bên trong $() là $x:expr, khớp với bất kỳ biểu thức Rust nào và đặt tên biểu thức đó là $x.

Dấu phẩy sau $() chỉ ra rằng một ký tự dấu phẩy phân tách literal phải xuất hiện giữa các lần code trong $() khớp. Dấu * chỉ ra pattern này khớp không hoặc nhiều lần với bất cứ thứ gì đứng trước *.

Khi gọi macro này với vec![1, 2, 3];, pattern $x khớp ba lần với ba biểu thức 1, 2, và 3.

Giờ hãy nhìn vào pattern trong thân code gắn với arm này: temp_vec.push() bên trong $()* được tạo ra cho mỗi phần khớp với $() trong pattern, không hoặc nhiều lần tùy theo số lần pattern khớp. $x được thay thế bởi mỗi biểu thức đã khớp. Khi gọi macro với vec![1, 2, 3];, code được tạo ra thay thế lời gọi macro này sẽ là:

{
    let mut temp_vec = Vec::new();
    temp_vec.push(1);
    temp_vec.push(2);
    temp_vec.push(3);
    temp_vec
}

Chúng ta đã định nghĩa một macro có thể nhận bất kỳ số lượng đối số của bất kỳ kiểu nào và tạo ra code để tạo vector chứa các phần tử đã chỉ định.

Để tìm hiểu thêm về cách viết macro, hãy tham khảo tài liệu online hoặc các tài nguyên khác như "The Little Book of Rust Macros" do Daniel Keep khởi xướng và Lukas Wirth tiếp nối.

Procedural Macros — Sinh Code từ Attributes

Dạng macro thứ hai là procedural macro, hoạt động giống hàm hơn (và thực chất là một loại thủ tục). Procedural macro nhận một số code làm đầu vào, thao tác trên code đó, và tạo ra code mới làm đầu ra — thay vì khớp pattern và thay code bằng code khác như declarative macro. Ba loại procedural macro là custom derive, attribute-like, và function-like, và tất cả đều hoạt động theo cách tương tự nhau.

Khi tạo procedural macro, định nghĩa phải nằm trong crate riêng với loại crate đặc biệt. Điều này vì những lý do kỹ thuật phức tạp mà chúng ta hy vọng sẽ được loại bỏ trong tương lai. Trong Listing 20-36, chúng ta minh họa cách định nghĩa procedural macro, trong đó some_attribute là placeholder cho một loại macro cụ thể.

Hàm định nghĩa procedural macro nhận TokenStream làm đầu vào và tạo ra TokenStream làm đầu ra. Kiểu TokenStream được định nghĩa bởi crate proc_macro đi kèm Rust và đại diện cho một chuỗi token. Đây là cốt lõi của macro: source code mà macro thao tác tạo thành TokenStream đầu vào, và code macro tạo ra là TokenStream đầu ra. Hàm còn có attribute gắn vào để chỉ định loại procedural macro đang tạo. Bạn có thể có nhiều loại procedural macro trong cùng một crate.

Hãy cùng xem các loại procedural macro khác nhau. Chúng ta sẽ bắt đầu với custom derive macro rồi giải thích những điểm khác biệt nhỏ giúp các loại khác trở nên đặc biệt.

Custom derive Macros

Hãy tạo một crate tên hello_macro định nghĩa trait HelloMacro với một associated function tên hello_macro. Thay vì bắt người dùng tự implement trait HelloMacro cho từng kiểu, chúng ta sẽ cung cấp procedural macro để người dùng có thể annotate kiểu của họ với #[derive(HelloMacro)] và nhận được implementation mặc định của hàm hello_macro. Implementation mặc định sẽ in ra Hello, Macro! My name is TypeName! trong đó TypeName là tên kiểu mà trait được định nghĩa trên đó. Tuy nhiên, họ sẽ phải viết khối implementation cho mỗi kiểu muốn dùng với hello_macro — chúng ta muốn giúp họ thoát khỏi công việc tẻ nhạt này.

Hơn nữa, chúng ta chưa thể cung cấp implementation mặc định cho hàm hello_macro để in tên kiểu: Rust không có khả năng reflection, vậy nên không thể tra cứu tên kiểu lúc runtime. Chúng ta cần macro để tạo code lúc compile time.

Bước tiếp theo là định nghĩa procedural macro. Tại thời điểm viết bài này, procedural macro cần nằm trong crate riêng. Cuối cùng hạn chế này có thể được gỡ bỏ. Quy ước đặt tên crate và macro crate như sau: với crate tên foo, crate custom derive procedural macro được gọi là foo_derive. Hãy tạo crate mới hello_macro_derive bên trong dự án hello_macro:

$ cargo new hello_macro_derive --lib

Hai crate của chúng ta liên quan chặt chẽ nên chúng ta tạo crate procedural macro bên trong thư mục của crate hello_macro. Nếu thay đổi định nghĩa trait trong hello_macro, chúng ta cũng phải thay đổi implementation của procedural macro trong hello_macro_derive. Hai crate sẽ cần publish riêng biệt, và lập trình viên dùng chúng cần thêm cả hai làm dependency và đưa cả hai vào scope. Thay vào đó chúng ta có thể để crate hello_macro dùng hello_macro_derive như dependency và re-export code procedural macro. Tuy nhiên, cách chúng ta cấu trúc dự án cho phép lập trình viên dùng hello_macro ngay cả khi họ không cần chức năng derive.

Chúng ta đã giới thiệu ba crate mới: proc_macro, syn, và quote. Crate proc_macro đi kèm Rust nên không cần thêm vào dependencies trong Cargo.toml. Crate proc_macro là API của compiler cho phép chúng ta đọc và thao tác code Rust từ code của mình.

Crate syn parse code Rust từ string thành cấu trúc dữ liệu mà chúng ta có thể thực hiện các thao tác. Crate quote chuyển cấu trúc dữ liệu syn trở lại thành code Rust. Các crate này giúp việc parse bất kỳ loại code Rust nào trở nên đơn giản hơn rất nhiều — viết đầy đủ một parser cho code Rust là điều không hề dễ dàng.

Hàm hello_macro_derive sẽ được gọi khi người dùng thư viện chỉ định #[derive(HelloMacro)] trên một kiểu. Điều này được thực hiện vì chúng ta đã annotate hàm hello_macro_derive với proc_macro_derive và chỉ định tên HelloMacro khớp với tên trait; đây là quy ước mà hầu hết procedural macro đều theo.

Hàm hello_macro_derive đầu tiên chuyển input từ TokenStream thành cấu trúc dữ liệu để chúng ta có thể diễn giải và thao tác. Đây là lúc syn phát huy tác dụng. Hàm parse trong syn nhận TokenStream và trả về struct DeriveInput đại diện cho code Rust đã được parse. Listing 20-41 hiển thị các phần liên quan của struct DeriveInput khi parse chuỗi struct Pancakes;.

Các trường của struct này cho thấy code Rust chúng ta đã parse là một unit struct với ident (identifier, tức là tên) là Pancakes. Struct này có thêm nhiều trường để mô tả đủ loại code Rust; hãy xem tài liệu syn cho DeriveInput để biết thêm.

Chúng ta sắp định nghĩa hàm impl_hello_macro — nơi chúng ta xây dựng code Rust mới muốn đưa vào. Nhưng trước khi làm vậy, hãy lưu ý rằng đầu ra của derive macro cũng là TokenStream. TokenStream được trả về sẽ được thêm vào code người dùng viết, vậy nên khi họ biên dịch crate, họ sẽ nhận được chức năng bổ sung mà chúng ta cung cấp trong TokenStream đã chỉnh sửa.

Bạn có thể đã để ý chúng ta gọi unwrap để khiến hàm hello_macro_derive panic nếu lời gọi hàm syn::parse thất bại. Điều này cần thiết vì hàm proc_macro_derive phải trả về TokenStream chứ không phải Result để phù hợp với API procedural macro. Chúng ta đã đơn giản hóa ví dụ này bằng unwrap; trong code production, bạn nên cung cấp thông báo lỗi cụ thể hơn bằng panic! hoặc expect.

Struct trong Listing 20-41 cho thấy khi chạy hàm impl_hello_macro trên code trong Listing 20-37, ident thu được sẽ có trường ident với giá trị "Pancakes". Do đó, biến name trong Listing 20-42 sẽ chứa instance struct Ident, khi in ra sẽ là chuỗi "Pancakes" — tên struct trong Listing 20-37.

Macro quote! cho phép chúng ta định nghĩa code Rust muốn trả về. Compiler cần thứ gì đó khác với kết quả trực tiếp của macro quote!, nên chúng ta cần chuyển đổi nó thành TokenStream bằng cách gọi phương thức into, tiêu thụ biểu diễn trung gian này và trả về giá trị kiểu TokenStream cần thiết.

Macro quote! còn cung cấp các cơ chế template cực kỳ thú vị: chúng ta có thể nhập #name và quote! sẽ thay thế nó bằng giá trị trong biến name. Bạn thậm chí có thể làm lặp lại tương tự như cách macro thông thường hoạt động. Xem tài liệu crate quote để biết thêm chi tiết.

Chúng ta muốn procedural macro tạo ra implementation của trait HelloMacro cho kiểu người dùng annotate, mà chúng ta lấy được bằng #name. Implementation trait có một hàm hello_macro, thân hàm chứa chức năng muốn cung cấp: in Hello, Macro! My name is rồi đến tên của kiểu được annotate.

Macro stringify! dùng ở đây là macro có sẵn trong Rust. Nó nhận biểu thức Rust, ví dụ 1 + 2, và lúc compile time biến biểu thức thành string literal, ví dụ "1 + 2". Khác với format! hay println! là những macro evaluate biểu thức rồi biến kết quả thành String. Có khả năng đầu vào #name là biểu thức cần in theo nghĩa đen nên chúng ta dùng stringify!. Dùng stringify! cũng tiết kiệm một lần cấp phát bằng cách chuyển #name thành string literal lúc compile time.

Lúc này, cargo build nên hoàn thành thành công trong cả hello_macro và hello_macro_derive. Hãy kết nối các crate này với code trong Listing 20-37 để xem procedural macro hoạt động! Tạo project binary mới trong thư mục projects bằng cargo new pancakes. Chúng ta cần thêm hello_macro và hello_macro_derive làm dependencies trong Cargo.toml của crate pancakes. Nếu bạn đang publish các phiên bản hello_macro và hello_macro_derive lên crates.io, chúng sẽ là dependencies thông thường; nếu không, bạn có thể chỉ định chúng như path dependency:

[dependencies]
hello_macro = { path = "../hello_macro" }
hello_macro_derive = { path = "../hello_macro/hello_macro_derive" }

Đặt code trong Listing 20-37 vào src/main.rs và chạy cargo run: nó sẽ in Hello, Macro! My name is Pancakes!. Implementation của trait HelloMacro từ procedural macro đã được đưa vào mà crate pancakes không cần tự implement; #[derive(HelloMacro)] đã thêm implementation trait.

Tiếp theo, hãy khám phá cách các loại procedural macro khác nhau như thế nào so với custom derive macro.

Attribute-Like Macros

Attribute-like macro tương tự custom derive macro, nhưng thay vì tạo code cho attribute derive, chúng cho phép bạn tạo attribute mới. Chúng cũng linh hoạt hơn: derive chỉ hoạt động với struct và enum; còn attribute có thể áp dụng cho các item khác như hàm. Ví dụ, bạn có attribute tên route annotate hàm khi dùng một web application framework:

#[route(GET, "/")]
fn index() {

Attribute #[route] này sẽ được framework định nghĩa như một procedural macro. Signature của hàm định nghĩa macro trông như thế này:

#[proc_macro_attribute]
pub fn route(attr: TokenStream, item: TokenStream) -> TokenStream {

Ở đây chúng ta có hai tham số kiểu TokenStream. Tham số đầu tiên là nội dung của attribute: phần GET, "/". Tham số thứ hai là thân của item mà attribute được gắn vào: trong trường hợp này là fn index() {} và phần còn lại của thân hàm.

Ngoài điều đó, attribute-like macro hoạt động giống custom derive macro: bạn tạo một crate với loại crate proc-macro và implement hàm tạo ra code bạn muốn!

Function-Like Macros

Function-like macro định nghĩa macro trông như lời gọi hàm. Tương tự macro_rules! macro, chúng linh hoạt hơn hàm; ví dụ, chúng có thể nhận số lượng đối số không xác định. Tuy nhiên, macro_rules! macro chỉ có thể được định nghĩa bằng cú pháp match-like đã thảo luận ở phần "Declarative Macros for General Metaprogramming" trước đó. Function-like macro nhận tham số TokenStream và định nghĩa của chúng thao tác TokenStream đó bằng code Rust giống như hai loại procedural macro kia. Ví dụ về function-like macro là macro sql! có thể được gọi như sau:

let sql = sql!(SELECT * FROM posts WHERE id=1);

Macro này sẽ parse câu SQL bên trong nó và kiểm tra cú pháp đúng đắn của nó — quá trình xử lý phức tạp hơn nhiều so với macro_rules! macro. Macro sql! sẽ được định nghĩa như sau:

#[proc_macro]
pub fn sql(input: TokenStream) -> TokenStream {

Định nghĩa này tương tự signature của custom derive macro: chúng ta nhận các token bên trong ngoặc đơn và trả về code muốn tạo ra.
