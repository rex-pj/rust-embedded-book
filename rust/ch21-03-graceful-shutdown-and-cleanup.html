Graceful Shutdown và Dọn dẹp

Code trong Listing 21-20 đang phản hồi các request bất đồng bộ thông qua thread pool, đúng như ý định của chúng ta. Chúng ta nhận được một số cảnh báo về các trường workers, id, và thread mà chúng ta không dùng trực tiếp — nhắc nhở rằng chúng ta chưa dọn dẹp gì cả. Khi dùng cách kém tinh tế là Ctrl+C để dừng thread chính, tất cả các thread khác cũng bị dừng ngay lập tức, ngay cả khi chúng đang giữa quá trình xử lý request.

Tiếp theo, chúng ta sẽ triển khai trait Drop để gọi join trên mỗi thread trong pool để chúng có thể hoàn thành các request đang xử lý trước khi đóng. Sau đó, chúng ta sẽ triển khai cách báo cho các thread biết chúng nên ngừng chấp nhận request mới và tắt máy. Để thấy code này hoạt động, chúng ta sẽ sửa server để chỉ chấp nhận hai request trước khi gracefully tắt thread pool.

Một điều cần lưu ý: Không có gì trong phần này ảnh hưởng đến các phần code xử lý thực thi closure, vì vậy mọi thứ ở đây sẽ giống nhau nếu chúng ta đang dùng thread pool cho async runtime.

Báo hiệu cho Threads Ngừng Lắng nghe Job

Với tất cả các thay đổi đã thực hiện, code biên dịch không có cảnh báo. Tuy nhiên, tin xấu là code này chưa hoạt động như ý muốn. Vấn đề nằm ở logic trong closure do các thread của Worker chạy: Hiện tại chúng ta gọi join, nhưng điều đó sẽ không tắt thread vì chúng loop mãi để tìm job. Nếu thử drop ThreadPool với triển khai drop hiện tại, thread chính sẽ bị block mãi mãi, chờ thread đầu tiên hoàn thành.

Để khắc phục, chúng ta cần thay đổi triển khai drop của ThreadPool và sau đó thay đổi vòng lặp Worker.

Method take được định nghĩa trong trait Iterator và giới hạn iteration đến hai item đầu tiên nhiều nhất. ThreadPool sẽ ra khỏi scope ở cuối main, và triển khai drop sẽ chạy.

Khởi động server với cargo run và thực hiện ba request. Request thứ ba sẽ báo lỗi, và trong terminal bạn sẽ thấy output tương tự:

$ cargo run
   Compiling hello v0.1.0 (file:///projects/hello)
    Finished `dev` profile [unoptimized + debuginfo] target(s) in 1.0s
     Running `target/debug/hello`
Worker 0 got a job; executing.
Shutting down.
Shutting down worker 0
Worker 3 got a job; executing.
Worker 1 disconnected; shutting down.
Worker 2 disconnected; shutting down.
Worker 3 disconnected; shutting down.
Worker 0 disconnected; shutting down.
Shutting down worker 1
Shutting down worker 2
Shutting down worker 3

Bạn có thể thấy thứ tự khác nhau khi chạy. Chúng ta thấy các Worker với ID 0 và 3 nhận được các request. Sau hai request, server ngừng chấp nhận kết nối mới. Khi ThreadPool ra khỏi scope cuối main, triển khai Drop của nó bắt đầu, và pool báo cho tất cả Worker tắt máy. Các Worker in ra thông báo khi chúng thấy thông điệp ngắt kết nối, sau đó thread pool gọi join để chờ từng thread hoàn thành.

Hãy chú ý một điều thú vị về thứ tự trong output này: Worker 0 và 3 nhận được các job, rồi server ngừng. Worker 1 và 2 nhận thấy ngắt kết nối và tắt. Worker 0 và 3 hoàn thành job của mình và cũng nhận thấy ngắt kết nối, rồi tắt. Cuối cùng, thread chính gọi join trên mỗi Worker, chờ chúng hoàn thành trước khi thoát.

Chúc mừng! Chúng ta đã hoàn thành dự án; giờ chúng ta có một web server cơ bản sử dụng thread pool để phản hồi bất đồng bộ. Chúng ta có thể thực hiện graceful shutdown của server, dọn dẹp tất cả các thread trong pool.
