Tách Rời Các Module Ra Thành Từng File Khác Nhau

Từ nãy tới giờ, toàn bộ các ví dụ trong chương này đều định nghĩa nhiều module gom chung trong một file duy nhất. Khi mà các module dần bành trướng lớn hơn, bạn có lẽ sẽ muốn dời các phần định nghĩa của chúng sang một file riêng biệt để giúp đoạn code dễ dàng định hướng đọc hơn.

Chẳng hạn, hãy bắt đầu lại bộ code ở Listing 7-17 chứa đựng nhiều loại module nhà hàng khác nhau đan xen. Chúng ta sẽ chiết xuất từng module ra các file riêng lẻ thay vì cứ để tất cả rúc vào chung đụng trong rễ file (crate root file). Ở ví dụ này, rễ file (crate root) đang là src/lib.rs, thế nhưng thủ thuật này bạn vẫn áp xài ngon lành cho binary crates có bộ rễ file gốc khởi điểm tại src/main.rs.

Đầu tiên, chúng ta sẽ bưng cái module front_of_house sang một file độc lập của chính nó. Hãy xóa bỏ toàn bộ code nằm lọt trong cặp dấu ngoặc nhọn của module front_of_house, chỉ để chừa lại duy nhất câu lệnh khai báo mod front_of_house;, nên src/lib.rs lúc này sẽ chỉ chứa mã như thể hiện tại Listing 7-21. Chú ý rằng mớ này sẽ quăng lỗi trình biên dịch chừng nào bạn phải tạo cho được file src/front_of_house.rs như thể hiện ở Listing 7-22 thì nó mới chạy.

Tiếp theo, hãy nhồi trọn đoạn code bạn vừa xóa từ trong lòng dấu ngoặc nhọn kia sang hẳn một tập tin mới gắn tên src/front_of_house.rs, giống Listing 7-22 làm mẫu. Tụi compiler tự khắc biết cách là phải đánh hơi tìm cọc file này do lúc nãy chúng lướt trúng phải dòng khai module ở cọc root cái tên gọi front_of_house.

Lưu ý rằng bạn chỉ cần khai nạp load một file bằng từ khóa khai báo mod duy nhất một lần trong suốt trọn nhành module tree của project. Một khi trình biên dịch ngửi ra được tập tin đó là một phần của project (và hóng được nó vọc vạch chui rúc nằm ở đâu trong mớ module tree từ vị trí bạn găm đặt lệnh statement mod), là các tập tin khác trong project của bạn chỉ cần lấy chỉ dẫn lôi code ở cái file đã nạp bằng cách chiếu khai path đến gốc ban đầu nơi nó được khai báo ra, như đã mổ xẻ mục "Dùng Đường Dẫn Path Để Ngắm Gọi Một Cục Item bên trong Module Tree". Đổi cách diễn đạt: cái từ mod vạch vòi đây sạch bách chẳng phải là cái phép nhúng include operation mà các thể loại coders hay múa máy dùng bên mấy bến đỗ ngôn ngữ lập trình xứ khác.

Tiếp theo đội xếp hàng, ta sẽ đôn cái module hosting thảy vọt sang ổ file độc ác của riêng hắn. Trình tự này hẵng có nới lóng chút gợn khác biệt bởi cái thân hosting đính mác danh phận là một child module (module con cưng) dính vào mẹ front_of_house, chứ không rẽ ngoặc cắm thân chướng về root module (module gốc). Chúng mình sẽ đặt tên cái file của hosting chui tọt vào tạo một thư mục mớ mới toanh lấy tên xưng danh y xì đúc với nguyên lão ancestors module bọc rễ nó, ở trường hợp này là chui dzô src/front_of_house.

Cái đống quy luật mớ rẽ lối rình rập vạch định tìm code đi khấp tìm tệp trùng đúng ổ nòng bọc của nhóm compiler sẽ xích đám tệp tin cùng thư mục ôm ấp chặt khắng khít vừa hệt như khuôn đúc sát xịt với nhóm module tree ngóc rễ vác kia hơn.

Các Đường Dẫn Tệp Tin Khác

Cho đến nay, chúng ta đã đề cập đến các đường dẫn (file paths) chuẩn mực nhất mà trình biên dịch Rust sử dụng, nhưng Rust cũng hỗ trợ một định dạng đường dẫn cũ hơn. Đối với một module tên là front_of_house được khai báo trong crate root, trình biên dịch sẽ tìm kiếm đoạn code của module đó ở:

  • src/front_of_house.rs (chuẩn mực mà chúng ta vừa phân tích)
  • src/front_of_house/mod.rs (định dạng cũ, vẫn còn được hỗ trợ)

Đối với một module tên là hosting và là một module con (submodule) của front_of_house, trình biên dịch sẽ tìm phần code của module này ở:

  • src/front_of_house/hosting.rs (chuẩn mực mà chúng ta vừa phân tích)
  • src/front_of_house/hosting/mod.rs (định dạng cũ, vẫn còn được hỗ trợ)

Nếu bạn sử dụng cả hai định dạng cho cùng một module, bạn sẽ nhận được thông báo lỗi từ phía trình biên dịch. Việc trộn lẫn hai định dạng cho các module khác nhau trong cùng một dự án tuy được cho phép nhưng lại có nguy cơ gây nhầm lẫn cho những ai đọc qua dự án của bạn.

Nhược điểm lớn nhất khi xài định dạng sử dụng tên tệp mod.rs là dự án của bạn sẽ kết thúc với một rổ các tệp tin có cùng tên mod.rs, làm dấy lên sự khó chịu khi bạn lỡ mở chúng cùng lúc trên code editor của mình.

Chúng ta đã chuyển mỗi module vào từng file riêng biệt và thành công duy trì sự ổn định của nguyên module tree ban đầu. Lời gọi hàm trong eat_at_restaurant sẽ hoạt động trơn tru mà không cần chỉnh sửa bất cứ thứ gì, dù cho các định nghĩa bây giờ đã sống ở những file hoàn toàn khác biệt. Kỹ thuật này tiếp sức cho bạn chuyển các modules sang các files độc lập khi chúng dần trở nên phức tạp về kích cỡ.

Hãy cùng lưu tâm rằng dòng lệnh pub use crate::front_of_house::hosting tại src/lib.rs cũng chả chịu sức mẻ xê dịch gì, và kể cả lệnh use cũng không bao biện tác động đến bất cứ quá trình tệp nào được biên dịch cho cái hộp crate. Từ khóa mod khai mở mác modules, và gã thợ Rust lân la tìm tệp tin đồng tên với danh tính module nọ để bê code vào cho module đó.

Tóm Lược

Rust cho phép bạn chia một package thành một chùm crates nối với nhau, và bẻ một crate xuống thành các modules để tiếp sức cho quá trình truy xuất các item được cấu hình từ module bên bờ này sang phía rạch của tay module bên kia. Bạn có thể châm ngòi điều này thông qua việc phân chia hai bộ paths: tuyệt đối (absolute) hay vòng vo tương đối (relative). Những khối paths này được tống thẳng vô scope bằng chiêu thức use để cho bạn hưởng sái shortcut một cái path đi tắt lấp lỉm khi cần vọc đi vọc lại đống item lọt trong cái scope nội bộ nói trên. Đóng vai trò mặc định, tất thảy bộ code trong ruột một module hoàn toàn là đồ private (thâm cung), nhưng bạn chỉ cần phù phép chèn mác từ khóa pub để xưng tụng phần công bố ra chỗ public sính ngoại.

Đến chặn dừng ở chương sau, chúng ta sẽ bắt tay săm soi vài loại cấu trúc dữ liệu collections lọt bọc trong thư viện chuẩn, mà bạn sẽ phải có duyên mượn tay tới chèo lái trong bộ code được dọn thẳng dán chỉn chu này.
