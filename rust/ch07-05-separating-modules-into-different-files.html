Tách Module ra các File khác nhau

Cho đến nay, tất cả ví dụ trong chương này đều định nghĩa nhiều module trong một file duy nhất. Khi các module trở nên lớn hơn, bạn có thể muốn chuyển định nghĩa của chúng sang một file riêng để code dễ điều hướng hơn.

Ví dụ, hãy bắt đầu từ code trong Listing 7-17 có nhiều module về nhà hàng. Ta sẽ tách các module ra các file thay vì giữ tất cả trong file crate root. Trong trường hợp này, file crate root là src/lib.rs, nhưng quy trình này cũng hoạt động với binary crate có file crate root là src/main.rs.

Trước tiên, ta sẽ tách module front_of_house ra file riêng của nó. Xóa phần code nằm trong cặp ngoặc nhọn của module front_of_house, chỉ giữ lại khai báo mod front_of_house;, sao cho src/lib.rs chứa code như trong Listing 7-21. Lưu ý rằng code này sẽ chưa biên dịch được cho đến khi ta tạo file src/front_of_house.rs trong Listing 7-22.

Tiếp theo, đặt phần code từ trong cặp ngoặc nhọn vào một file mới tên là src/front_of_house.rs, như trong Listing 7-22. Compiler biết cần tìm ở file này vì nó gặp khai báo module trong crate root với tên front_of_house.

Lưu ý rằng bạn chỉ cần load một file bằng khai báo mod một lần duy nhất trong module tree của mình. Một khi compiler biết file đó là một phần của dự án (và biết code nằm ở đâu trong module tree dựa vào vị trí bạn đặt câu lệnh mod), các file khác trong dự án nên tham chiếu đến code của file đã được load bằng cách dùng path đến nơi nó được khai báo, như đã trình bày trong phần "Paths for Referring to an Item in the Module Tree". Nói cách khác, mod không phải là thao tác "include" như bạn có thể thấy trong các ngôn ngữ lập trình khác.

Tiếp theo, ta sẽ tách module hosting ra file riêng của nó. Quy trình này hơi khác một chút vì hosting là child module của front_of_house, không phải của root module. Ta sẽ đặt file cho hosting trong một thư mục mới được đặt tên theo các module cha của nó trong module tree, trong trường hợp này là src/front_of_house.

Quy tắc của compiler về việc kiểm tra file nào cho module nào đồng nghĩa với việc cấu trúc thư mục và file phản ánh chặt chẽ module tree.

Các đường dẫn file thay thế

Cho đến nay ta đã đề cập đến cách đặt đường dẫn file theo phong cách chuẩn nhất mà Rust compiler sử dụng, nhưng Rust cũng hỗ trợ một phong cách cũ hơn. Với module tên là front_of_house được khai báo trong crate root, compiler sẽ tìm code của module trong:

  • src/front_of_house.rs (cách ta đã đề cập)
  • src/front_of_house/mod.rs (phong cách cũ, vẫn còn được hỗ trợ)

Với module tên là hosting là submodule của front_of_house, compiler sẽ tìm code của module trong:

  • src/front_of_house/hosting.rs (cách ta đã đề cập)
  • src/front_of_house/hosting/mod.rs (phong cách cũ, vẫn còn được hỗ trợ)

Nếu bạn dùng cả hai phong cách cho cùng một module, bạn sẽ gặp lỗi biên dịch. Dùng hỗn hợp hai phong cách cho các module khác nhau trong cùng một dự án được phép, nhưng có thể gây khó hiểu cho người điều hướng dự án.

Nhược điểm chính của phong cách dùng file có tên mod.rs là dự án của bạn có thể có nhiều file cùng tên mod.rs, điều này dễ gây nhầm lẫn khi bạn mở nhiều file như vậy trong editor cùng một lúc.

Ta đã chuyển code của mỗi module sang một file riêng, và module tree vẫn giữ nguyên. Các lời gọi hàm trong eat_at_restaurant sẽ hoạt động bình thường mà không cần sửa đổi gì, dù các định nghĩa nằm ở các file khác nhau. Kỹ thuật này cho phép bạn chuyển module sang file mới khi module phát triển về kích thước.

Lưu ý rằng câu lệnh pub use crate::front_of_house::hosting trong src/lib.rs cũng không thay đổi, và use không ảnh hưởng đến việc file nào được biên dịch như là một phần của crate. Từ khóa mod khai báo các module, và Rust tìm trong file có cùng tên với module để lấy code đưa vào module đó.

Tổng kết

Rust cho phép bạn chia một package thành nhiều crate và chia một crate thành các module, nhờ đó bạn có thể tham chiếu đến các item được định nghĩa trong một module từ module khác. Bạn có thể làm điều này bằng cách chỉ định absolute path hoặc relative path. Các path này có thể được đưa vào scope bằng câu lệnh use để bạn dùng path ngắn hơn cho nhiều lần sử dụng item trong scope đó. Code của module mặc định là private, nhưng bạn có thể công khai các định nghĩa bằng cách thêm từ khóa pub.

Trong chương tiếp theo, ta sẽ xem xét một số cấu trúc dữ liệu collection trong thư viện chuẩn mà bạn có thể sử dụng trong code được tổ chức gọn gàng của mình.
