Xây dựng Web Server Đơn luồng

Chúng ta sẽ bắt đầu bằng cách làm cho một web server đơn luồng hoạt động. Trước khi bắt đầu, hãy cùng xem qua tổng quan nhanh về các giao thức liên quan khi xây dựng web server. Chi tiết về các giao thức này nằm ngoài phạm vi cuốn sách, nhưng một cái nhìn tổng quan sẽ cung cấp đủ thông tin cần thiết.

Hai giao thức chính liên quan đến web server là Hypertext Transfer Protocol (HTTP) và Transmission Control Protocol (TCP). Cả hai đều là giao thức request-response, nghĩa là client khởi tạo request và server lắng nghe request rồi cung cấp response cho client. Nội dung của các request và response đó được định nghĩa bởi các giao thức.

TCP là giao thức cấp thấp hơn, mô tả chi tiết cách thông tin được truyền từ server này sang server khác nhưng không chỉ định thông tin đó là gì. HTTP xây dựng trên TCP bằng cách định nghĩa nội dung của các request và response. Về mặt kỹ thuật, có thể dùng HTTP với các giao thức khác, nhưng trong phần lớn trường hợp, HTTP gửi dữ liệu qua TCP. Chúng ta sẽ làm việc với các byte thô của TCP và HTTP request, response.

📌
Sơ đồ: Luồng xử lý Server Đơn luồng

Single-threaded server flow:
TCP Listener → Nhận stream → handle_connection() → Đọc HTTP request → Trả về HTTP response → Đóng kết nối → (lặp lại — xử lý tuần tự)

Hạn chế: Mỗi lần chỉ xử lý được 1 request. Nếu một request chậm, toàn bộ các request sau phải chờ.

Lắng nghe kết nối TCP

Web server của chúng ta cần lắng nghe kết nối TCP, đó là phần đầu tiên chúng ta sẽ làm. Thư viện chuẩn cung cấp module std::net để thực hiện điều này. Hãy tạo một dự án mới theo cách thông thường:

$ cargo new hello
     Created binary (application) `hello` project
$ cd hello

Bây giờ hãy nhập code ở Listing 21-1 vào src/main.rs để bắt đầu. Code này sẽ lắng nghe ở địa chỉ cục bộ 127.0.0.1:7878 cho các TCP stream đến. Khi nhận được stream, nó sẽ in Connection established!.

Dùng TcpListener, chúng ta có thể lắng nghe kết nối TCP tại địa chỉ 127.0.0.1:7878. Trong địa chỉ, phần trước dấu hai chấm là địa chỉ IP đại diện cho máy tính của bạn (giống nhau trên mọi máy và không đại diện cho máy của các tác giả), và 7878 là port. Chúng ta chọn port này vì hai lý do: HTTP thường không được chấp nhận trên port này nên server của chúng ta khó xung đột với bất kỳ web server nào khác đang chạy trên máy, và 7878 là chữ rust gõ trên bàn phím điện thoại.

Hàm bind trong kịch bản này hoạt động như hàm new ở chỗ nó trả về một TcpListener instance mới. Hàm được gọi là bind vì trong mạng, kết nối với một port để lắng nghe được gọi là "binding to a port" (gắn vào port).

Hàm bind trả về Result<T, E>, cho biết việc binding có thể thất bại, ví dụ như khi chạy hai instance của chương trình và có hai chương trình lắng nghe trên cùng port. Vì chúng ta đang viết server cơ bản cho mục đích học, chúng ta sẽ không xử lý các lỗi này; thay vào đó dùng unwrap để dừng chương trình nếu xảy ra lỗi.

Phương thức incoming trên TcpListener trả về iterator cung cấp một chuỗi stream (cụ thể hơn là stream kiểu TcpStream). Một stream đại diện cho một kết nối mở giữa client và server. Connection là tên cho toàn bộ quá trình request và response khi client kết nối với server, server tạo response và server đóng kết nối. Như vậy, chúng ta sẽ đọc từ TcpStream để xem client gửi gì rồi ghi response vào stream để gửi dữ liệu trở lại client. Tổng thể, vòng lặp for này sẽ xử lý từng kết nối lần lượt và tạo ra một chuỗi stream để chúng ta xử lý.

Hiện tại, việc xử lý stream bao gồm gọi unwrap để kết thúc chương trình nếu stream có lỗi; nếu không có lỗi, chương trình in thông báo. Chúng ta sẽ thêm nhiều chức năng hơn cho trường hợp thành công ở listing tiếp theo.

Hãy thử chạy code này! Gọi cargo run trong terminal rồi tải 127.0.0.1:7878 trong trình duyệt web. Trình duyệt sẽ hiển thị thông báo lỗi như "Connection reset" vì server hiện chưa gửi dữ liệu nào trở lại. Nhưng khi nhìn vào terminal, bạn sẽ thấy nhiều thông báo được in khi trình duyệt kết nối với server!

     Running `target/debug/hello`
Connection established!
Connection established!
Connection established!

Đôi khi bạn sẽ thấy nhiều thông báo được in cho một request từ trình duyệt; lý do có thể là trình duyệt đang thực hiện request cho trang cũng như request cho các tài nguyên khác, như icon favicon.ico xuất hiện trong tab trình duyệt.

Cũng có thể trình duyệt đang cố kết nối nhiều lần vì server không phản hồi dữ liệu. Khi stream ra ngoài phạm vi và bị drop ở cuối vòng lặp, kết nối bị đóng như một phần của drop implementation. Trình duyệt đôi khi xử lý kết nối bị đóng bằng cách thử lại vì vấn đề có thể là tạm thời.

Điều quan trọng là chúng ta đã nhận được một handle cho kết nối TCP thành công!

Nhớ dừng chương trình bằng cách nhấn ctrl-C khi xong. Sau đó, khởi động lại chương trình bằng lệnh cargo run sau mỗi bộ thay đổi code để đảm bảo bạn đang chạy code mới nhất.

Đọc Request

Hãy triển khai chức năng đọc request từ trình duyệt! Để tách biệt mối quan tâm giữa việc nhận kết nối và xử lý kết nối, chúng ta sẽ tạo một hàm mới để xử lý kết nối. Trong hàm handle_connection mới này, chúng ta sẽ đọc dữ liệu từ TCP stream và in ra để xem dữ liệu được gửi từ trình duyệt. Thay đổi code như Listing 21-2.

Chúng ta đưa std::io::BufReader và std::io::prelude vào phạm vi để truy cập trait và kiểu cho phép đọc và ghi vào stream. Trong vòng lặp for trong hàm main, thay vì in thông báo kết nối thành công, chúng ta gọi hàm handle_connection mới và truyền stream vào.

Trong hàm handle_connection, chúng ta tạo instance BufReader mới bao quanh tham chiếu đến stream. BufReader thêm buffering bằng cách quản lý các lần gọi phương thức của trait std::io::Read cho chúng ta.

Chúng ta tạo biến http_request để thu thập các dòng của request mà trình duyệt gửi đến server. Chúng ta chỉ ra muốn thu thập các dòng này vào một vector bằng cách thêm annotation kiểu Vec<_>.

BufReader implement trait std::io::BufRead, cung cấp phương thức lines. Phương thức lines trả về iterator của Result<String, std::io::Error> bằng cách tách stream dữ liệu khi gặp byte newline. Để lấy từng String, chúng ta map và unwrap từng Result.

Trình duyệt báo hiệu kết thúc HTTP request bằng cách gửi hai ký tự newline liên tiếp, do đó để lấy một request từ stream, chúng ta lấy các dòng cho đến khi gặp dòng rỗng. Sau khi thu thập các dòng vào vector, chúng ta in chúng bằng pretty debug formatting.

Xem xét kỹ hơn về HTTP Request

HTTP là giao thức dựa trên text, và request có định dạng sau:

Method Request-URI HTTP-Version CRLF
headers CRLF
message-body

Dòng đầu tiên là request line chứa thông tin về những gì client đang yêu cầu. Phần đầu tiên của request line cho biết method đang được dùng, như GET hay POST, mô tả cách client thực hiện request này. Client của chúng ta dùng GET request, nghĩa là đang yêu cầu thông tin.

Phần tiếp theo của request line là /, cho biết uniform resource identifier (URI) mà client đang yêu cầu. Phần cuối là phiên bản HTTP mà client dùng, và request line kết thúc bằng chuỗi CRLF. (CRLF là viết tắt của carriage return và line feed, thuật ngữ từ thời máy đánh chữ!) Chuỗi CRLF phân tách request line với phần còn lại của dữ liệu request.

Sau request line, các dòng còn lại bắt đầu từ Host: trở đi là header. GET request không có body.

Viết Response

Chúng ta sẽ triển khai việc gửi dữ liệu để phản hồi client request. Response có định dạng sau:

HTTP-Version Status-Code Reason-Phrase CRLF
headers CRLF
message-body

Dòng đầu tiên là status line chứa phiên bản HTTP dùng trong response, status code số tóm tắt kết quả của request, và reason phrase mô tả text của status code. Sau chuỗi CRLF là bất kỳ header nào, một chuỗi CRLF khác, và body của response.

Đây là ví dụ response dùng HTTP version 1.1 với status code 200, reason phrase OK, không có header và không có body:

HTTP/1.1 200 OK\r\n\r\n

Status code 200 là response thành công chuẩn. Hãy ghi điều này vào stream như response cho request thành công! Từ hàm handle_connection, xóa println! in dữ liệu request và thay bằng code ở Listing 21-3.

Dòng mới đầu tiên định nghĩa biến response chứa dữ liệu thông báo thành công. Sau đó, chúng ta gọi as_bytes trên response để chuyển đổi dữ liệu chuỗi thành byte. Phương thức write_all trên stream nhận &[u8] và gửi các byte đó trực tiếp qua kết nối.
