Lỗi Không Thể Phục Hồi với panic!

Đôi khi trong code của bạn có những thứ hỏng hoàn toàn và chẳng còn cách nào cứu vãn. Trong những tình huống bi đát đó, Rust trang bị macro panic!. Có hai cách để gây ra panic trong thực tế: hoặc là thực hiện một hành động khiến code banh chành (ví dụ truy cập mảng vượt quá giới hạn cuối), hoặc là gọi macro panic! một cách tường minh. Cả hai cách đều làm cho chương trình của bạn rơi vào trạng thái panic. Theo mặc định, những cơn panic này sẽ in ra thông báo lỗi, unwind (cuộn ngược) stack, dọn dẹp tài nguyên, rồi thoát. Thông qua một biến môi trường, bạn còn có thể bảo Rust hiển thị cả call stack khi panic xảy ra, giúp bạn truy tìm nguyên nhân dễ hơn.

Unwind Stack hay Abort Khi Gặp Panic?

Mặc định, khi panic xảy ra, chương trình bắt đầu unwind — nghĩa là Rust leo ngược lên stack và dọn dẹp dữ liệu của từng hàm mà nó đi qua. Tuy nhiên, việc leo ngược và dọn dẹp như vậy tốn khá nhiều công sức. Vì thế Rust cho phép bạn chọn phương án thay thế là abort ngay lập tức — tức là kết thúc chương trình mà không thèm dọn dẹp gì hết.

Phần bộ nhớ mà chương trình đang dùng khi đó sẽ do hệ điều hành dọn dẹp. Nếu trong dự án bạn cần file binary nhỏ gọn nhất có thể, bạn có thể chuyển từ unwinding sang aborting khi panic bằng cách thêm panic = 'abort' vào mục [profile] phù hợp trong file Cargo.toml. Ví dụ, nếu bạn muốn abort khi panic ở chế độ release, thêm dòng này:

[profile.release]
panic = 'abort'

Thử gọi panic! trong một chương trình đơn giản xem sao:

Khi chạy chương trình này, bạn sẽ thấy kết quả kiểu như thế này:

$ cargo run
   Compiling panic v0.1.0 (file:///projects/panic)
    Finished `dev` profile [unoptimized + debuginfo] target(s) in 0.25s
     Running `target/debug/panic`

thread 'main' panicked at src/main.rs:2:5:
crash and burn
note: run with `RUST_BACKTRACE=1` environment variable to display a backtrace

Lời gọi panic! gây ra thông báo lỗi xuất hiện ở hai dòng cuối. Dòng đầu tiên cho thấy thông điệp panic và vị trí trong source code nơi panic xảy ra: src/main.rs:2:5 nghĩa là dòng thứ hai, ký tự thứ năm của file src/main.rs.

Trong trường hợp này, dòng được chỉ ra là một phần trong code của ta, và nếu ta tới dòng đó, ta sẽ thấy lời gọi macro panic!. Trong những trường hợp khác, lời gọi panic! có thể nằm trong code của thư viện mà code ta đang gọi, và tên file cùng số dòng được thông báo lỗi chỉ ra sẽ là code của người khác — nơi macro panic! được gọi — chứ không phải dòng code của ta mà rốt cuộc dẫn đến lời gọi panic! đó.

Ta có thể dùng backtrace của các hàm mà lời gọi panic! xuất phát từ đó để tìm ra phần code của mình đang gây ra vấn đề. Để hiểu cách dùng backtrace của panic!, hãy xem một ví dụ khác và coi thử điều gì xảy ra khi lời gọi panic! đến từ một thư viện do bug trong code của ta thay vì từ code ta gọi macro trực tiếp. Listing 9-1 có đoạn code cố truy cập một chỉ mục trong vector vượt ra ngoài phạm vi chỉ mục hợp lệ.

Ở đây, ta đang cố truy cập phần tử thứ 100 của vector (nằm ở chỉ mục 99 vì đánh số từ 0), nhưng vector chỉ có đúng ba phần tử. Trong tình huống này, Rust sẽ panic. Dùng [] đáng ra phải trả về một phần tử, nhưng nếu bạn truyền vào chỉ mục không hợp lệ thì chẳng có phần tử nào để Rust trả về cả.

Trong C, cố đọc dữ liệu ngoài biên của một cấu trúc là hành vi không xác định (undefined behavior). Bạn có thể lấy được bất cứ thứ gì nằm ở vùng nhớ tương ứng với phần tử đó trong cấu trúc, dù vùng nhớ đó không thuộc về cấu trúc đó. Đây gọi là buffer overread và có thể dẫn đến lỗ hổng bảo mật nếu kẻ tấn công có thể thao túng chỉ mục để đọc dữ liệu mà chúng không được phép đọc, vốn nằm sau cấu trúc dữ liệu.

Để bảo vệ chương trình khỏi loại lỗ hổng này, nếu bạn cố đọc phần tử ở chỉ mục không tồn tại, Rust sẽ dừng thực thi và từ chối tiếp tục. Hãy thử và xem:

$ cargo run
   Compiling panic v0.1.0 (file:///projects/panic)
    Finished `dev` profile [unoptimized + debuginfo] target(s) in 0.27s
     Running `target/debug/panic`

thread 'main' panicked at src/main.rs:4:6:
index out of bounds: the len is 3 but the index is 99
note: run with `RUST_BACKTRACE=1` environment variable to display a backtrace

Thông báo lỗi này trỏ vào dòng 4 của file main.rs, nơi ta cố truy cập chỉ mục 99 của vector trong v.

Dòng note: cho ta biết có thể đặt biến môi trường RUST_BACKTRACE để lấy backtrace về chính xác điều gì đã xảy ra và gây ra lỗi. Backtrace là danh sách tất cả các hàm đã được gọi để đến điểm hiện tại. Backtraces trong Rust hoạt động giống như các ngôn ngữ khác: Bí quyết để đọc backtrace là bắt đầu từ trên cùng và đọc xuống cho đến khi thấy các file mà bạn đã viết. Đó chính là điểm xuất phát của vấn đề. Các dòng phía trên điểm đó là code mà code của bạn đã gọi; các dòng bên dưới là code đã gọi code của bạn. Những dòng trước-và-sau này có thể bao gồm core Rust code, standard library code, hoặc các crate bạn đang dùng. Hãy thử lấy backtrace bằng cách đặt biến môi trường RUST_BACKTRACE thành bất kỳ giá trị nào khác 0. Listing 9-2 cho thấy output tương tự với những gì bạn sẽ thấy.

Đó là một đống output! Output chính xác mà bạn thấy có thể khác tùy thuộc vào hệ điều hành và phiên bản Rust. Để lấy bactraces với thông tin này, debug symbols phải được bật. Debug symbols được bật mặc định khi dùng cargo build hoặc cargo run không có cờ --release, như ở đây.

Trong output ở Listing 9-2, dòng 6 của backtrace trỏ vào dòng trong dự án của ta đang gây ra vấn đề: dòng 4 của src/main.rs. Nếu ta không muốn chương trình panic, ta nên bắt đầu điều tra tại vị trí được trỏ đến bởi dòng đầu tiên đề cập đến file ta đã viết. Trong Listing 9-1, nơi ta cố tình viết code sẽ panic, cách để sửa là không yêu cầu phần tử ngoài phạm vi chỉ mục của vector. Khi code của bạn panic trong tương lai, bạn cần tìm hiểu hành động nào code đang thực hiện với những giá trị nào khiến panic xảy ra và code nên làm gì thay thế.

Ta sẽ quay lại panic! và bàn về khi nào nên và không nên dùng panic! để xử lý các điều kiện lỗi trong phần "panic! hay Không panic!?" sau trong chương này. Tiếp theo, ta sẽ xem cách phục hồi lỗi bằng Result.
