Lưu trữ văn bản UTF-8 với String

Ta đã đề cập đến string ở Chương 4, nhưng giờ mình sẽ đào sâu hơn một chút. Nhiều Rustacean mới hay bị vấp ở phần string vì ba lý do kết hợp với nhau: Rust hay lộ ra các lỗi tiềm ẩn, string thực ra là một cấu trúc dữ liệu phức tạp hơn nhiều người nghĩ, và cuối cùng là UTF-8. Ba thứ này cộng lại dễ làm bạn choáng nếu bạn vừa chuyển sang từ ngôn ngữ khác.

Ta bàn về string trong ngữ cảnh collections vì string thực chất được cài đặt như một tập hợp các byte, kèm theo một số method giúp xử lý dữ liệu khi các byte đó được hiểu là văn bản. Trong phần này, ta sẽ nói về các thao tác trên String mà mọi collection đều có, như tạo mới, cập nhật và đọc. Ta cũng sẽ bàn về những điểm khiến String khác các collection khác — cụ thể là tại sao việc truy cập bằng index vào String lại phức tạp do sự khác biệt giữa cách người và máy tính hiểu dữ liệu String.

Định nghĩa String

Trước hết, hãy làm rõ ta đang nói đến cái gì khi nhắc tới string. Rust chỉ có một kiểu string duy nhất ở tầng ngôn ngữ lõi, đó là string slice str, thường xuất hiện dưới dạng mượn là &str. Ở Chương 4, ta đã nói về string slice — chúng là các tham chiếu đến dữ liệu string được mã hóa UTF-8 lưu ở nơi khác. Ví dụ, string literal được lưu trong binary của chương trình, vì thế chúng là string slice.

Còn kiểu String thì do thư viện chuẩn của Rust cung cấp chứ không phải là một phần của ngôn ngữ lõi — đây là kiểu string có thể mở rộng, mutable, có ownership, và được mã hóa UTF-8. Khi Rustacean nói đến "string" trong Rust, họ có thể đang nhắc đến cả String lẫn &str, không chỉ một trong hai. Mặc dù phần này chủ yếu nói về String, cả hai kiểu đều được dùng nhiều trong thư viện chuẩn và đều được mã hóa UTF-8.

Tạo một String mới

Nhiều thao tác dùng được với Vec<T> cũng dùng được với String, bởi vì String thực ra được cài đặt như một wrapper quanh vector byte với một số đảm bảo, hạn chế và khả năng thêm. Một ví dụ là hàm new để tạo một instance, như Listing 8-11 dưới đây.

Dòng này tạo ra một string rỗng tên là s, sau đó ta có thể nạp dữ liệu vào. Thường thì ta sẽ có sẵn dữ liệu ban đầu để khởi tạo string. Khi đó, ta dùng method to_string, có sẵn trên bất kỳ kiểu nào implement trait Display, trong đó có string literal. Listing 8-12 cho thấy hai cách dùng.

Đoạn code này tạo ra một string chứa initial contents.

Ta cũng có thể dùng hàm String::from để tạo String từ string literal. Đoạn code ở Listing 8-13 tương đương với Listing 8-12 dùng to_string.

Vì string được dùng cho quá nhiều thứ nên ta có rất nhiều API generic cho string, dẫn đến nhiều lựa chọn. Một số trông có vẻ dư thừa, nhưng chúng đều có chỗ đứng! Trong trường hợp này, String::from và to_string làm cùng một việc, bạn chọn cái nào là tùy phong cách và dễ đọc.

Nhớ rằng string được mã hóa UTF-8, nên ta có thể chứa bất kỳ dữ liệu nào được mã hóa đúng trong đó, như Listing 8-14 minh họa.

Tất cả những cái trên đều là giá trị String hợp lệ.

Cập nhật một String

Một String có thể tăng kích thước và nội dung có thể thay đổi, giống như nội dung của Vec<T> khi bạn nhét thêm dữ liệu vào. Ngoài ra, bạn còn có thể dùng toán tử + hoặc macro format! để nối các String lại với nhau.

Thêm vào cuối với push_str hoặc push

Ta có thể mở rộng String bằng cách dùng method push_str để thêm một string slice, như Listing 8-15 dưới đây.

Sau hai dòng này, s sẽ chứa foobar. Method push_str nhận vào string slice vì ta không nhất thiết muốn lấy ownership của tham số. Ví dụ, trong đoạn code ở Listing 8-16, ta muốn vẫn dùng được s2 sau khi đã thêm nội dung của nó vào s1.

Nếu push_str lấy ownership của s2, ta sẽ không in được giá trị của nó ở dòng cuối. Nhưng code này hoạt động đúng như kỳ vọng!

Method push nhận vào một ký tự đơn và thêm nó vào String. Listing 8-17 thêm chữ l vào một String bằng method push.

Kết quả là s sẽ chứa lol.

Nối chuỗi với + hoặc format!

Thường thì bạn sẽ muốn kết hợp hai string có sẵn lại. Một cách là dùng toán tử +, như Listing 8-18 dưới đây.

String s3 sẽ chứa Hello, world!. Lý do s1 không còn hợp lệ sau phép cộng, và tại sao ta dùng tham chiếu đến s2, liên quan đến signature của method được gọi khi ta dùng toán tử +. Toán tử + dùng method add, có signature trông đại khái như thế này:

fn add(self, s: &str) -> String {

Trong thư viện chuẩn, bạn sẽ thấy add được định nghĩa với generics và associated types. Ở đây ta đã thay bằng kiểu cụ thể, tức là điều xảy ra khi ta gọi method này với giá trị String. Ta sẽ nói về generics ở Chương 10. Signature này cung cấp những manh mối để hiểu những điểm kỳ lạ của toán tử +.

Đầu tiên, s2 có dấu &, nghĩa là ta đang thêm một tham chiếu đến string thứ hai vào string thứ nhất. Điều này bởi vì tham số s trong hàm add: ta chỉ có thể thêm một string slice vào một String; không thể cộng hai String trực tiếp. Nhưng khoan — kiểu của &s2 là &String, không phải &str như tham số thứ hai của add yêu cầu. Vậy tại sao Listing 8-18 vẫn compile được?

Lý do ta dùng được &s2 trong lời gọi add là vì compiler có thể ép kiểu &String thành &str. Khi ta gọi method add, Rust dùng deref coercion, để biến đổi &s2 thành &s2[..]. Ta sẽ bàn sâu hơn về deref coercion ở Chương 15. Vì add không lấy ownership của tham số s, s2 vẫn là một String hợp lệ sau thao tác này.

Thứ hai, ta thấy trong signature rằng add lấy ownership của self vì self không có dấu &. Điều này có nghĩa s1 trong Listing 8-18 sẽ bị move vào lời gọi add và không còn hợp lệ sau đó. Vì vậy, mặc dù let s3 = s1 + &s2; trông như sẽ copy cả hai string rồi tạo mới, câu lệnh này thực ra lấy ownership của s1, thêm bản copy nội dung của s2, rồi trả về ownership của kết quả. Nói cách khác, trông như copy nhiều thứ nhưng thực ra không phải; cách cài đặt này hiệu quả hơn copy.

Nếu ta cần nối nhiều string, hành vi của toán tử + trở nên rắc rối:

fn main() {
    let s1 = String::from("tic");
    let s2 = String::from("tac");
    let s3 = String::from("toe");

    let s = s1 + "-" + &s2 + "-" + &s3;
}

Lúc này s sẽ là tic-tac-toe. Với đống dấu + và " lộn xộn, khó mà đọc hiểu code. Khi cần nối string theo cách phức tạp hơn, ta có thể dùng macro format! thay thế:

fn main() {
    let s1 = String::from("tic");
    let s2 = String::from("tac");
    let s3 = String::from("toe");

    let s = format!("{s1}-{s2}-{s3}");
}

Code này cũng gán s bằng tic-tac-toe. Macro format! hoạt động giống println!, nhưng thay vì in ra màn hình, nó trả về một String với nội dung tương ứng. Phiên bản dùng format! dễ đọc hơn nhiều, và code do macro format! tạo ra dùng tham chiếu nên không lấy ownership của bất kỳ tham số nào.

Truy cập String bằng index

Trong nhiều ngôn ngữ lập trình khác, truy cập từng ký tự trong string bằng index là thao tác bình thường và phổ biến. Tuy nhiên, nếu bạn thử truy cập các phần của String bằng cú pháp index trong Rust, bạn sẽ gặp lỗi. Hãy xem đoạn code không hợp lệ ở Listing 8-19.

Đoạn code này sẽ gây ra lỗi sau:

Lỗi đã nói lên tất cả: Rust string không hỗ trợ index. Nhưng tại sao? Để trả lời câu hỏi đó, ta cần bàn về cách Rust lưu string trong bộ nhớ.

Biểu diễn nội tại

Một String là wrapper quanh Vec<u8>. Hãy nhìn lại một số string UTF-8 được mã hóa đúng từ Listing 8-14. Đầu tiên, cái này:

fn main() {
    let hello = String::from("السلام عليكم");
    let hello = String::from("Dobrý den");
    let hello = String::from("Hello");
    let hello = String::from("שלום");
    let hello = String::from("नमस्ते");
    let hello = String::from("こんにちは");
    let hello = String::from("안녕하세요");
    let hello = String::from("你好");
    let hello = String::from("Olá");
    let hello = String::from("Здравствуйте");
    let hello = String::from("Hola");
}

Trong trường hợp này, len sẽ là 4, nghĩa là vector lưu string "Hola" dài 4 byte. Mỗi chữ cái này chiếm 1 byte khi mã hóa UTF-8. Tuy nhiên, dòng tiếp theo có thể khiến bạn ngạc nhiên (lưu ý string này bắt đầu bằng chữ hoa Cyrillic Ze, không phải số 3):

fn main() {
    let hello = String::from("السلام عليكم");
    let hello = String::from("Dobrý den");
    let hello = String::from("Hello");
    let hello = String::from("שלום");
    let hello = String::from("नमस्ते");
    let hello = String::from("こんにちは");
    let hello = String::from("안녕하세요");
    let hello = String::from("你好");
    let hello = String::from("Olá");
    let hello = String::from("Здравствуйте");
    let hello = String::from("Hola");
}

Nếu bạn được hỏi chuỗi này dài bao nhiêu, bạn có thể đáp là 12. Nhưng thực ra Rust trả lời là 24: đó là số byte cần để mã hóa "Здравствуйте" trong UTF-8, vì mỗi Unicode scalar value trong chuỗi đó chiếm 2 byte. Vì thế, một index vào các byte của string không phải lúc nào cũng tương ứng với một Unicode scalar value hợp lệ. Để minh họa, hãy xem đoạn code Rust không hợp lệ này:

let hello = "Здравствуйте";
let answer = &hello[0];

Bạn đã biết answer sẽ không phải là З, chữ cái đầu tiên. Khi mã hóa UTF-8, byte đầu tiên của З là 208 và byte thứ hai là 151, vậy nên trông như answer phải là 208, nhưng 208 không phải là một ký tự hợp lệ đứng một mình. Trả về 208 chắc chắn không phải điều người dùng muốn khi họ hỏi chữ cái đầu tiên của string; tuy nhiên, đó là dữ liệu duy nhất Rust có ở byte index 0. Người dùng thường không muốn nhận về giá trị byte, kể cả khi string chỉ chứa chữ Latin: nếu &"hi"[0] là code hợp lệ trả về giá trị byte, nó sẽ trả về 104, không phải h.

Câu trả lời là: để tránh trả về giá trị không mong muốn và gây ra bug khó phát hiện, Rust không compile code này từ đầu, ngăn chặn hiểu nhầm ngay trong quá trình phát triển.

Bytes, Scalar Values, và Grapheme Clusters

Một điểm nữa về UTF-8 là có ba cách nhìn khác nhau về string từ góc độ Rust: dưới dạng byte, scalar values, và grapheme clusters (cái gần nhất với điều ta gọi là chữ cái).

Nếu ta nhìn vào chữ Hindi "नमस्ते" viết theo chữ Devanagari, nó được lưu dưới dạng vector các giá trị u8 trông như thế này:

[224, 164, 168, 224, 164, 174, 224, 164, 184, 224, 165, 141, 224, 164, 164,
224, 165, 135]

Đó là 18 byte và là cách máy tính thực sự lưu dữ liệu này. Nếu ta nhìn chúng dưới dạng Unicode scalar values — tức là kiểu char của Rust — các byte đó trông như thế này:

['न', 'म', 'स', '्', 'त', 'े']

Có sáu giá trị char ở đây, nhưng cái thứ tư và thứ sáu không phải chữ cái: Chúng là dấu phụ không có nghĩa khi đứng một mình. Cuối cùng, nếu ta nhìn chúng dưới dạng grapheme clusters, ta sẽ thấy những gì mà người ta gọi là bốn chữ cái tạo nên chữ Hindi đó:

["न", "म", "स्", "ते"]

Rust cung cấp các cách khác nhau để diễn giải dữ liệu string thô mà máy tính lưu, để mỗi chương trình có thể chọn cách diễn giải phù hợp với mình, bất kể dữ liệu thuộc ngôn ngữ nào.

Một lý do cuối cùng khiến Rust không cho phép ta index vào String để lấy ký tự là các thao tác index được kỳ vọng luôn có thời gian hằng số (O(1)). Nhưng không thể đảm bảo hiệu suất đó với String, vì Rust sẽ phải duyệt qua nội dung từ đầu đến index để xác định có bao nhiêu ký tự hợp lệ.

Cắt lát String

Truy cập string bằng index thường là ý tưởng tồi vì không rõ kiểu trả về của thao tác đó nên là gì: một giá trị byte, một ký tự, một grapheme cluster, hay một string slice. Vì vậy nếu bạn thực sự cần dùng index để tạo string slice, Rust yêu cầu bạn phải chỉ rõ hơn.

Thay vì index bằng [] với một số duy nhất, bạn có thể dùng [] với một range để tạo string slice chứa các byte cụ thể:

#![allow(unused)]
fn main() {
let hello = "Здравствуйте";

let s = &hello[0..4];
}

Ở đây, s sẽ là một &str chứa 4 byte đầu tiên của string. Trước đó ta đã đề cập mỗi ký tự này chiếm 2 byte, nên s sẽ là Зд.

Nếu ta thử cắt chỉ một phần byte của một ký tự, ví dụ như &hello[0..1], Rust sẽ panic lúc runtime giống như khi truy cập index không hợp lệ trong vector:

$ cargo run
   Compiling collections v0.1.0 (file:///projects/collections)
    Finished `dev` profile [unoptimized + debuginfo] target(s) in 0.43s
     Running `target/debug/collections`

thread 'main' panicked at src/main.rs:4:19:
byte index 1 is not a char boundary; it is inside 'З' (bytes 0..2) of `Здравствуйте`
note: run with `RUST_BACKTRACE=1` environment variable to display a backtrace

Bạn cần cẩn thận khi tạo string slice bằng range, vì làm vậy có thể crash chương trình.

Duyệt qua String

Cách tốt nhất để thao tác trên từng phần của string là nói rõ bạn muốn ký tự hay byte. Với từng Unicode scalar value, dùng method chars. Gọi chars trên "Зд" sẽ tách ra và trả về hai giá trị kiểu char, và bạn có thể duyệt qua kết quả để truy cập từng phần tử:

#![allow(unused)]
fn main() {
for c in "Зд".chars() {
    println!("{c}");
}
}

Đoạn code này sẽ in ra:

З
д

Ngoài ra, method bytes trả về từng byte thô, có thể phù hợp với trường hợp của bạn:

#![allow(unused)]
fn main() {
for b in "Зд".bytes() {
    println!("{b}");
}
}

Đoạn code này sẽ in ra 4 byte tạo nên string này:

208
151

Xử lý sự phức tạp của String

Tóm lại, string thật sự phức tạp. Các ngôn ngữ lập trình khác nhau đưa ra các lựa chọn khác nhau về cách trình bày sự phức tạp này cho lập trình viên. Rust chọn cách xử lý đúng đắn dữ liệu String là hành vi mặc định cho mọi chương trình Rust, nghĩa là lập trình viên phải suy nghĩ nhiều hơn về việc xử lý dữ liệu UTF-8 ngay từ đầu. Sự đánh đổi này lộ ra nhiều sự phức tạp của string hơn so với các ngôn ngữ khác, nhưng bù lại giúp bạn tránh phải xử lý các lỗi liên quan đến ký tự không phải ASCII ở giai đoạn sau của vòng đời phát triển.

Tin vui là thư viện chuẩn cung cấp rất nhiều chức năng xây dựng trên String và &str để giúp xử lý các tình huống phức tạp này một cách chính xác. Hãy tra tài liệu để tìm các method hữu ích như contains để tìm kiếm trong string và replace để thay thế một phần string bằng một string khác.
