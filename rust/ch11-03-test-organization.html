Tổ chức Test

Như đã đề cập ở đầu chương, testing là một lĩnh vực phức tạp, và mỗi người có cách dùng thuật ngữ và tổ chức khác nhau. Cộng đồng Rust chia test thành hai loại chính: unit test và integration test. Unit test thì nhỏ và tập trung hơn, kiểm thử từng module một cách riêng lẻ, và có thể test cả các interface private. Integration test hoàn toàn đứng ngoài thư viện của bạn và dùng code của bạn theo đúng cách mà code bên ngoài sẽ dùng — chỉ thông qua public interface, và có thể chạm đến nhiều module trong một lần test.

Viết cả hai loại test đều quan trọng để đảm bảo rằng các phần của thư viện hoạt động đúng như kỳ vọng, cả khi chạy độc lập lẫn khi phối hợp với nhau.

Unit Test

Mục đích của unit test là kiểm thử từng đơn vị code một cách tách biệt với phần còn lại, để nhanh chóng xác định chỗ nào đang hoạt động đúng, chỗ nào chưa đúng. Bạn sẽ đặt unit test vào thư mục src, trong cùng file với code mà chúng test. Quy ước là tạo một module tên tests trong mỗi file để chứa các hàm test, và đánh dấu module đó bằng cfg(test).

Module tests và #[cfg(test)]

Annotation #[cfg(test)] trên module tests báo cho Rust biết chỉ compile và chạy code test khi bạn chạy cargo test, chứ không phải khi chạy cargo build. Điều này giúp tiết kiệm thời gian compile khi bạn chỉ muốn build thư viện, đồng thời giảm dung lượng file kết quả vì test không được đưa vào. Bạn sẽ thấy rằng vì integration test nằm ở thư mục khác, chúng không cần annotation #[cfg(test)]. Nhưng với unit test — nằm chung file với code — bạn cần dùng #[cfg(test)] để chỉ rõ rằng chúng không được đưa vào bản build thông thường.

Nhớ lại khi chúng ta tạo project adder ở phần đầu chương, Cargo đã tự sinh ra đoạn code này:

Tên file: src/lib.rs

pub fn add(left: u64, right: u64) -> u64 {
    left + right
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn it_works() {
        let result = add(2, 2);
        assert_eq!(result, 4);
    }
}

Trong module tests được tự sinh ra đó, attribute cfg là viết tắt của configuration — nó báo cho Rust biết rằng item tiếp theo chỉ được đưa vào khi có một tùy chọn cấu hình cụ thể. Trong trường hợp này, tùy chọn đó là test, do Rust cung cấp để compile và chạy test. Nhờ attribute cfg, Cargo chỉ compile code test khi ta chạy lệnh cargo test. Điều này áp dụng cho cả các hàm helper trong module đó, ngoài những hàm được đánh dấu #[test].

Test Hàm Private

Cộng đồng testing có nhiều quan điểm về việc có nên test trực tiếp các hàm private hay không, và một số ngôn ngữ thậm chí không cho phép làm điều đó. Dù bạn theo trường phái nào, các quy tắc về privacy của Rust vẫn cho phép bạn test hàm private. Hãy xem đoạn code trong Listing 11-12 với hàm private internal_adder.

Lưu ý rằng hàm internal_adder không được đánh dấu là pub. Test cũng chỉ là code Rust thông thường, và module tests cũng chỉ là một module bình thường. Như đã thảo luận trong phần "Paths for Referring to an Item in the Module Tree", các item trong module con có thể dùng các item từ module cha. Trong test này, ta đưa tất cả item từ module cha của tests vào scope bằng use super::*, và sau đó test có thể gọi internal_adder. Nếu bạn không muốn test hàm private, Rust không ép buộc bạn làm vậy.

Integration Test

Trong Rust, integration test hoàn toàn đứng ngoài thư viện của bạn. Chúng dùng thư viện theo đúng cách mà code bên ngoài sẽ dùng — tức là chỉ có thể gọi các hàm thuộc public API. Mục đích của chúng là kiểm tra xem nhiều phần của thư viện có phối hợp hoạt động đúng cách không. Những đơn vị code hoạt động tốt riêng lẻ vẫn có thể gặp vấn đề khi tích hợp lại, vì vậy test coverage cho code tích hợp cũng rất quan trọng. Để tạo integration test, trước tiên bạn cần có thư mục tests.

Thư mục tests

Chúng ta tạo thư mục tests ở cấp cao nhất của project, ngang hàng với src. Cargo biết để tìm các file integration test trong thư mục này. Chúng ta có thể tạo bao nhiêu file test tùy thích, và Cargo sẽ compile mỗi file như một crate độc lập.

Hãy tạo một integration test. Với code trong Listing 11-12 vẫn còn trong file src/lib.rs, tạo thư mục tests và một file mới tên tests/integration_test.rs. Cấu trúc thư mục lúc này trông như thế này:

adder
├── Cargo.lock
├── Cargo.toml
├── src
│   └── lib.rs
└── tests
    └── integration_test.rs

Nhập code trong Listing 11-13 vào file tests/integration_test.rs.

Mỗi file trong thư mục tests là một crate riêng biệt, vì vậy chúng ta cần đưa thư viện vào scope của từng test crate. Đó là lý do chúng ta thêm use adder::add_two; ở đầu — điều mà trong unit test chúng ta không cần làm.

Chúng ta không cần đánh dấu bất kỳ code nào trong tests/integration_test.rs bằng #[cfg(test)]. Cargo xử lý thư mục tests một cách đặc biệt và chỉ compile các file trong thư mục này khi chúng ta chạy cargo test. Giờ hãy chạy cargo test:

$ cargo test
   Compiling adder v0.1.0 (file:///projects/adder)
    Finished `test` profile [unoptimized + debuginfo] target(s) in 1.31s
     Running unittests src/lib.rs (target/debug/deps/adder-1082c4b063a8fbe6)

running 1 test
test tests::internal ... ok

test result: ok. 1 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out; finished in 0.00s

     Running tests/integration_test.rs (target/debug/deps/integration_test-1082c4b063a8fbe6)

running 1 test
test it_adds_two ... ok

test result: ok. 1 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out; finished in 0.00s

   Doc-tests adder

running 0 tests

test result: ok. 0 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out; finished in 0.00s

Ba phần output bao gồm unit test, integration test và doc test. Lưu ý rằng nếu có test nào trong một phần bị fail, các phần tiếp theo sẽ không được chạy. Ví dụ: nếu một unit test fail, sẽ không có output cho integration test và doc test, vì chúng chỉ chạy khi tất cả unit test đều pass.

Phần đầu tiên dành cho unit test vẫn như những gì chúng ta đã thấy: một dòng cho mỗi unit test (ở đây là internal từ Listing 11-12) và một dòng tóm tắt.

Phần integration test bắt đầu với dòng Running tests/integration_test.rs. Tiếp theo là một dòng cho mỗi hàm test trong file integration test đó, và một dòng tóm tắt kết quả, trước khi phần Doc-tests adder bắt đầu.

Mỗi file integration test có phần output riêng của nó, vì vậy nếu bạn thêm nhiều file vào thư mục tests, sẽ có thêm nhiều phần integration test tương ứng.

Bạn vẫn có thể chạy một hàm test integration cụ thể bằng cách truyền tên hàm như một tham số cho cargo test. Để chạy tất cả test trong một file integration test cụ thể, dùng argument --test của cargo test theo sau là tên file:

$ cargo test --test integration_test
   Compiling adder v0.1.0 (file:///projects/adder)
    Finished `test` profile [unoptimized + debuginfo] target(s) in 0.64s
     Running tests/integration_test.rs (target/debug/deps/integration_test-82e7799c1bc62298)

running 1 test
test it_adds_two ... ok

test result: ok. 1 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out; finished in 0.00s

Lệnh này chỉ chạy các test trong file tests/integration_test.rs.

Submodule trong Integration Test

Khi bạn thêm nhiều integration test hơn, bạn có thể muốn tổ chức chúng thành nhiều file trong thư mục tests — ví dụ, nhóm các hàm test theo chức năng mà chúng kiểm thử. Như đã đề cập, mỗi file trong thư mục tests được compile như một crate riêng biệt, điều này hữu ích để tạo các scope tách biệt, mô phỏng cách người dùng cuối sẽ dùng crate của bạn. Tuy nhiên, điều này cũng có nghĩa là các file trong thư mục tests không có cùng hành vi như các file trong src, như bạn đã học trong Chương 7 về cách tách code thành module và file.

Sự khác biệt về hành vi của thư mục tests thể hiện rõ nhất khi bạn có một bộ hàm helper dùng chung cho nhiều file integration test, và bạn muốn làm theo các bước trong phần "Separating Modules into Different Files" của Chương 7 để tách chúng ra module chung. Ví dụ: nếu chúng ta tạo file tests/common.rs và đặt hàm setup vào đó, chúng ta có thể thêm code vào setup để gọi từ nhiều hàm test trong nhiều file test:

Tên file: tests/common.rs

pub fn setup() {
    // code setup riêng cho test của thư viện sẽ đặt ở đây
}

Khi chạy test lại, chúng ta sẽ thấy một phần mới trong output cho file common.rs, dù file này không chứa hàm test nào và chúng ta cũng chưa gọi hàm setup từ bất kỳ đâu:

$ cargo test
   Compiling adder v0.1.0 (file:///projects/adder)
    Finished `test` profile [unoptimized + debuginfo] target(s) in 0.89s
     Running unittests src/lib.rs (target/debug/deps/adder-92948b65e88960b4)

running 1 test
test tests::internal ... ok

test result: ok. 1 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out; finished in 0.00s

     Running tests/common.rs (target/debug/deps/common-92948b65e88960b4)

running 0 tests

test result: ok. 0 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out; finished in 0.00s

     Running tests/integration_test.rs (target/debug/deps/integration_test-92948b65e88960b4)

running 1 test
test it_adds_two ... ok

test result: ok. 1 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out; finished in 0.00s

   Doc-tests adder

running 0 tests

test result: ok. 0 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out; finished in 0.00s

Việc common hiện ra trong kết quả test với dòng running 0 tests không phải là điều chúng ta muốn. Chúng ta chỉ muốn chia sẻ code với các file integration test khác. Để tránh common xuất hiện trong output test, thay vì tạo tests/common.rs, hãy tạo tests/common/mod.rs. Cấu trúc thư mục project lúc này trông như thế này:

├── Cargo.lock
├── Cargo.toml
├── src
│   └── lib.rs
└── tests
    ├── common
    │   └── mod.rs
    └── integration_test.rs

Đây là quy ước đặt tên cũ hơn mà Rust vẫn hiểu được, đã được đề cập trong phần "Alternate File Paths" của Chương 7. Đặt tên file theo cách này báo cho Rust biết không nên coi module common là file integration test. Khi chúng ta di chuyển code hàm setup vào tests/common/mod.rs và xóa file tests/common.rs, phần đó sẽ không còn xuất hiện trong output test nữa. Các file trong thư mục con của tests không được compile thành các crate riêng biệt và cũng không có phần output trong kết quả test.

Sau khi tạo tests/common/mod.rs, chúng ta có thể dùng nó từ bất kỳ file integration test nào như một module. Đây là ví dụ gọi hàm setup từ test it_adds_two trong tests/integration_test.rs:

Tên file: tests/integration_test.rs

use adder::add_two;

mod common;

#[test]
fn it_adds_two() {
    common::setup();

    let result = add_two(2);
    assert_eq!(result, 4);
}

Lưu ý rằng khai báo mod common; cũng giống như khai báo module mà chúng ta đã thấy trong Listing 7-21. Sau đó, trong hàm test, chúng ta có thể gọi hàm common::setup().

Integration Test cho Binary Crate

Nếu project của bạn là một binary crate chỉ chứa file src/main.rs mà không có file src/lib.rs, bạn không thể tạo integration test trong thư mục tests và đưa các hàm được định nghĩa trong src/main.rs vào scope bằng câu lệnh use. Chỉ có library crate mới có thể expose hàm cho crate khác dùng; binary crate được thiết kế để chạy độc lập.

Đây là một trong những lý do tại sao các project Rust cung cấp binary thường có file src/main.rs rất đơn giản, chỉ gọi logic nằm trong file src/lib.rs. Với cấu trúc đó, integration test có thể test library crate bằng use để truy cập các chức năng quan trọng. Nếu các chức năng quan trọng hoạt động đúng, thì phần code nhỏ trong src/main.rs cũng sẽ hoạt động tốt, và phần code nhỏ đó không nhất thiết phải được test riêng.

Tóm tắt

Các tính năng testing của Rust cung cấp cách chỉ định code nên hoạt động như thế nào để đảm bảo nó tiếp tục hoạt động đúng như kỳ vọng, ngay cả khi bạn thực hiện thay đổi. Unit test kiểm tra từng phần của thư viện một cách riêng biệt và có thể test các chi tiết implementation private. Integration test kiểm tra xem nhiều phần của thư viện có phối hợp hoạt động đúng không, và chúng dùng public API của thư viện để test code theo đúng cách mà code bên ngoài sẽ dùng. Dù hệ thống kiểu và các quy tắc ownership của Rust giúp ngăn chặn một số loại bug, nhưng test vẫn quan trọng để giảm thiểu các logic bug liên quan đến hành vi kỳ vọng của code.

Hãy kết hợp kiến thức bạn đã học trong chương này và các chương trước để cùng làm một dự án thực tế!
