Cải thiện dự án I/O của chúng ta

Với kiến thức mới về iterator, chúng ta có thể cải thiện dự án I/O ở Chương 12 bằng cách dùng iterator để làm cho code rõ ràng và súc tích hơn. Hãy xem cách iterator có thể cải thiện implementation của hàm Config::build và hàm search.

Loại bỏ clone bằng cách dùng iterator

Ở Listing 12-6, chúng ta đã thêm code lấy một slice các giá trị String và tạo một instance của struct Config bằng cách index vào slice và clone các giá trị, cho phép struct Config sở hữu những giá trị đó. Trong Listing 13-17, chúng ta tái hiện lại implementation của hàm Config::build như trong Listing 12-23.

Hồi đó, chúng ta đã nói đừng lo về các lời gọi clone không hiệu quả vì sẽ loại bỏ chúng sau. Và lúc đó đây rồi!

Chúng ta cần clone ở đây vì tham số args là một slice chứa các phần tử String, nhưng hàm build không sở hữu args. Để trả về ownership của một instance Config, chúng ta phải clone các giá trị từ các trường query và file_path của Config để instance Config có thể sở hữu các giá trị của nó.

Với kiến thức mới về iterator, chúng ta có thể thay đổi hàm build để lấy ownership của một iterator làm tham số thay vì borrow một slice. Chúng ta sẽ dùng chức năng của iterator thay vì code kiểm tra độ dài của slice và index vào các vị trí cụ thể. Điều này làm rõ hơn hàm Config::build đang làm gì vì iterator sẽ truy cập trực tiếp vào các giá trị.

Khi Config::build lấy ownership của iterator và dừng dùng các thao tác index mượn, chúng ta có thể di chuyển các giá trị String từ iterator vào Config thay vì gọi clone và tạo bộ nhớ mới.

Dùng trực tiếp iterator được trả về

Mở file src/main.rs của dự án I/O của bạn, trông nó sẽ như thế này:

Tên file: src/main.rs

use std::env;
use std::error::Error;
use std::fs;
use std::process;

use minigrep::{search, search_case_insensitive};

fn main() {
    let args: Vec<String> = env::args().collect();

    let config = Config::build(&args).unwrap_or_else(|err| {
        eprintln!("Problem parsing arguments: {err}");
        process::exit(1);
    });

    // --snip--

    if let Err(e) = run(config) {
        eprintln!("Application error: {e}");
        process::exit(1);
    }
}

pub struct Config {
    pub query: String,
    pub file_path: String,
    pub ignore_case: bool,
}

impl Config {
    fn build(args: &[String]) -> Result<Config, &'static str> {
        if args.len() < 3 {
            return Err("not enough arguments");
        }

        let query = args[1].clone();
        let file_path = args[2].clone();

        let ignore_case = env::var("IGNORE_CASE").is_ok();

        Ok(Config {
            query,
            file_path,
            ignore_case,
        })
    }
}

fn run(config: Config) -> Result<(), Box<dyn Error>> {
    let contents = fs::read_to_string(config.file_path)?;

    let results = if config.ignore_case {
        search_case_insensitive(&config.query, &contents)
    } else {
        search(&config.query, &contents)
    };

    for line in results {
        println!("{line}");
    }

    Ok(())
}

Chúng ta sẽ đầu tiên thay đổi phần đầu hàm main mà chúng ta có trong Listing 12-24 thành code trong Listing 13-18, lần này dùng iterator. Code này sẽ không compile cho đến khi chúng ta cũng cập nhật Config::build.

Hàm env::args trả về một iterator! Thay vì thu thập các giá trị iterator vào một vector rồi truyền một slice cho Config::build, giờ chúng ta truyền trực tiếp ownership của iterator được trả về từ env::args cho Config::build.

Tiếp theo, chúng ta cần cập nhật định nghĩa của Config::build. Hãy thay đổi signature của Config::build như trong Listing 13-19. Code này vẫn chưa compile vì chúng ta cần cập nhật thân hàm.

Tài liệu thư viện chuẩn của hàm env::args cho thấy kiểu iterator mà nó trả về là std::env::Args, và kiểu đó implement trait Iterator và trả về các giá trị String.

Chúng ta đã cập nhật signature của hàm Config::build để tham số args có generic type với trait bound impl Iterator<Item = String> thay vì &[String]. Cách dùng cú pháp impl Trait mà chúng ta đã thảo luận trong phần "Dùng Trait làm tham số" của Chương 10 có nghĩa là args có thể là bất kỳ kiểu nào implement trait Iterator và trả về các item String.

Vì chúng ta đang lấy ownership của args và sẽ mutate args bằng cách duyệt qua nó, ta có thể thêm từ khóa mut vào khai báo tham số args để làm nó mutable.

Dùng các method của trait Iterator

Tiếp theo, chúng ta sẽ sửa thân hàm Config::build. Vì args implement trait Iterator, chúng ta biết mình có thể gọi method next trên nó! Listing 13-20 cập nhật code từ Listing 12-23 để dùng method next.

Hãy nhớ rằng giá trị đầu tiên trong giá trị trả về của env::args là tên chương trình. Chúng ta muốn bỏ qua phần đó và đến giá trị tiếp theo, vì vậy đầu tiên ta gọi next và không làm gì với giá trị trả về. Sau đó, ta gọi next để lấy giá trị ta muốn đặt vào trường query của Config. Nếu next trả về Some, ta dùng match để lấy ra giá trị. Nếu nó trả về None, tức là chưa đủ tham số, và ta return sớm với giá trị Err. Tương tự cho giá trị file_path.

Làm rõ code với iterator adapter

Chúng ta cũng có thể tận dụng iterator trong hàm search của dự án I/O, được tái hiện ở đây trong Listing 13-21 như trong Listing 12-19.

Chúng ta có thể viết code này theo cách súc tích hơn bằng cách dùng các iterator adapter method. Làm vậy cũng giúp chúng ta tránh phải có một vector trung gian results mutable. Phong cách lập trình hàm ưu tiên tối thiểu hóa lượng trạng thái mutable để code rõ ràng hơn. Loại bỏ trạng thái mutable còn có thể tạo điều kiện cho một cải tiến trong tương lai: tìm kiếm song song vì chúng ta sẽ không phải quản lý truy cập đồng thời vào vector results. Listing 13-22 cho thấy sự thay đổi này.

Hãy nhớ mục đích của hàm search là trả về tất cả các dòng trong contents có chứa query. Tương tự ví dụ filter trong Listing 13-16, code này dùng adapter filter để chỉ giữ lại những dòng mà line.contains(query) trả về true. Sau đó chúng ta thu thập các dòng khớp vào một vector khác với collect. Gọn gàng hơn nhiều! Hãy thoải mái áp dụng thay đổi tương tự để dùng iterator method trong hàm search_case_insensitive.

Để cải tiến thêm, hãy trả về một iterator từ hàm search bằng cách bỏ lời gọi collect và thay đổi kiểu trả về thành impl Iterator<Item = &'a str> để hàm trở thành một iterator adapter. Lưu ý rằng bạn cũng cần cập nhật các test! Thử tìm kiếm qua một file lớn bằng công cụ minigrep trước và sau khi thực hiện thay đổi này để quan sát sự khác biệt trong hành vi. Trước thay đổi, chương trình sẽ không in kết quả nào cho đến khi thu thập được tất cả kết quả, nhưng sau thay đổi, kết quả sẽ được in ra ngay khi tìm thấy từng dòng khớp vì vòng lặp for trong hàm run có thể tận dụng tính lazy của iterator.

Chọn giữa vòng lặp và iterator

Câu hỏi logic tiếp theo là bạn nên chọn phong cách nào trong code của mình và tại sao: implementation ban đầu trong Listing 13-21 hay phiên bản dùng iterator trong Listing 13-22 (giả sử chúng ta thu thập tất cả kết quả trước khi trả về thay vì trả về iterator). Hầu hết lập trình viên Rust đều thích dùng phong cách iterator. Ban đầu có hơi khó nắm bắt một chút, nhưng một khi bạn quen với các iterator adapter khác nhau và chức năng của chúng, iterator có thể dễ hiểu hơn. Thay vì loay hoay với các chi tiết của vòng lặp và xây dựng các vector mới, code tập trung vào mục tiêu cấp cao của vòng lặp. Điều này trừu tượng hóa đi những đoạn code quen thuộc, giúp ta dễ nhìn thấy các khái niệm đặc thù trong code này hơn, chẳng hạn như điều kiện lọc mà mỗi phần tử trong iterator phải thỏa mãn.

Nhưng liệu hai implementation có thực sự tương đương không? Giả định trực giác có thể là vòng lặp cấp thấp sẽ nhanh hơn. Hãy nói về hiệu suất.
