Lỗi Có Thể Phục Hồi với Result

Hầu hết các lỗi không đến mức nghiêm trọng cần phải dừng hẳn chương trình. Đôi khi khi một hàm thất bại, đó là vì một lý do mà bạn có thể hiểu được và xử lý dễ dàng. Ví dụ, nếu bạn cố mở một file nhưng thao tác thất bại vì file không tồn tại, bạn có thể muốn tạo file đó thay vì ngắt quá trình.

Nhớ lại từ "Xử Lý Thất Bại Tiềm Năng với Result" trong Chương 2 rằng enum Result được định nghĩa với hai biến thể, Ok và Err, như sau:

#![allow(unused)]
fn main() {
enum Result<T, E> {
    Ok(T),
    Err(E),
}
}

Cặp T và E là các tham số generic: Ta sẽ thảo luận chi tiết hơn về generics ở Chương 10. Điều bạn cần biết ngay bây giờ là T đại diện cho kiểu của giá trị sẽ được trả về trong trường hợp thành công bên trong biến thể Ok, và E đại diện cho kiểu của lỗi sẽ được trả về trong trường hợp thất bại bên trong biến thể Err. Vì Result có các tham số generic này, ta có thể dùng kiểu Result và các hàm được định nghĩa trên nó trong nhiều tình huống khác nhau nơi giá trị thành công và giá trị lỗi mà ta muốn trả về có thể khác nhau.

Hãy gọi một hàm trả về giá trị Result vì hàm đó có thể thất bại. Trong Listing 9-3, ta thử mở một file.

Kiểu trả về của File::open là Result<T, E>. Tham số generic T đã được implementation của File::open điền vào với kiểu của giá trị thành công, std::fs::File, là một file handle. Kiểu E dùng trong giá trị lỗi là std::io::Error. Kiểu trả về này có nghĩa là lời gọi File::open có thể thành công và trả về một file handle mà ta có thể đọc hoặc ghi. Lời gọi hàm cũng có thể thất bại: Ví dụ, file có thể không tồn tại, hoặc ta có thể không có quyền truy cập file. Hàm File::open cần có cách để cho ta biết nó thành công hay thất bại và đồng thời cho ta hoặc file handle hoặc thông tin lỗi. Chính xác đây là điều mà enum Result truyền đạt.

Trong trường hợp File::open thành công, giá trị trong biến greeting_file_result sẽ là một instance của Ok chứa file handle. Trong trường hợp thất bại, giá trị trong greeting_file_result sẽ là một instance của Err chứa thêm thông tin về loại lỗi đã xảy ra.

Ta cần thêm vào code trong Listing 9-3 để thực hiện các hành động khác nhau tùy thuộc vào giá trị mà File::open trả về. Listing 9-4 cho thấy một cách xử lý Result dùng công cụ cơ bản là biểu thức match mà ta đã thảo luận trong Chương 6.

Lưu ý rằng, giống như enum Option, enum Result và các biến thể của nó đã được đưa vào scope bởi prelude, nên ta không cần ghi Result:: trước Ok và Err trong các nhánh match.

Khi kết quả là Ok, code này sẽ trả về giá trị file bên trong biến thể Ok, và ta gán giá trị file handle đó vào biến greeting_file. Sau lệnh match, ta có thể dùng file handle để đọc hoặc ghi.

Nhánh còn lại của match xử lý trường hợp ta nhận được giá trị Err từ File::open. Trong ví dụ này, ta chọn gọi macro panic!. Nếu không có file nào tên hello.txt trong thư mục hiện tại và ta chạy code này, ta sẽ thấy output sau từ macro panic!:

$ cargo run
   Compiling error-handling v0.1.0 (file:///projects/error-handling)
    Finished `dev` profile [unoptimized + debuginfo] target(s) in 0.73s
     Running `target/debug/error-handling`

thread 'main' panicked at src/main.rs:8:23:
Problem opening the file: Os { code: 2, kind: NotFound, message: "No such file or directory" }
note: run with `RUST_BACKTRACE=1` environment variable to display a backtrace

Như thường lệ, output này cho ta biết chính xác điều gì đã xảy ra.

Khớp Với Các Loại Lỗi Khác Nhau

Code trong Listing 9-4 sẽ panic! bất kể lý do tại sao File::open thất bại. Tuy nhiên, ta muốn thực hiện các hành động khác nhau cho các lý do thất bại khác nhau. Nếu File::open thất bại vì file không tồn tại, ta muốn tạo file và trả về handle của file mới. Nếu File::open thất bại vì bất kỳ lý do nào khác — ví dụ vì ta không có quyền mở file — ta vẫn muốn code panic! theo cách như trong Listing 9-4. Để làm điều này, ta thêm một biểu thức match bên trong, như thể hiện trong Listing 9-5.

Kiểu của giá trị mà File::open trả về bên trong biến thể Err là io::Error, một struct được cung cấp bởi standard library. Struct này có một method là kind mà ta có thể gọi để lấy giá trị io::ErrorKind. Enum io::ErrorKind được cung cấp bởi standard library và có các biến thể đại diện cho các loại lỗi khác nhau có thể phát sinh từ một thao tác io. Biến thể ta muốn dùng là ErrorKind::NotFound, cho biết file ta đang cố mở chưa tồn tại. Vậy ta match trên greeting_file_result, nhưng cũng có một match bên trong trên error.kind().

Điều kiện ta muốn kiểm tra trong match bên trong là liệu giá trị trả về bởi error.kind() có phải là biến thể NotFound của enum ErrorKind không. Nếu có, ta thử tạo file bằng File::create. Tuy nhiên, vì File::create cũng có thể thất bại, ta cần một nhánh thứ hai trong biểu thức match bên trong. Khi file không thể được tạo, một thông báo lỗi khác sẽ được in ra. Nhánh thứ hai của match bên ngoài giữ nguyên, vậy chương trình panic với bất kỳ lỗi nào ngoài lỗi thiếu file.

Thay Thế Cho match với Result<T, E>

Nhiều match quá nhỉ! Biểu thức match rất hữu ích nhưng cũng khá nguyên thủy. Trong Chương 13, bạn sẽ học về closures, được dùng với nhiều method được định nghĩa trên Result<T, E>. Các method này có thể ngắn gọn hơn so với dùng match khi xử lý các giá trị Result<T, E> trong code của bạn.

Ví dụ, đây là cách khác để viết logic tương tự như Listing 9-5, lần này dùng closures và method unwrap_or_else:

use std::fs::File;
use std::io::ErrorKind;

fn main() {
    let greeting_file = File::open("hello.txt").unwrap_or_else(|error| {
        if error.kind() == ErrorKind::NotFound {
            File::create("hello.txt").unwrap_or_else(|error| {
                panic!("Problem creating the file: {error:?}");
            })
        } else {
            panic!("Problem opening the file: {error:?}");
        }
    });
}

Mặc dù code này có cùng hành vi như Listing 9-5, nó không chứa bất kỳ biểu thức match nào và dễ đọc hơn. Quay lại ví dụ này sau khi bạn đã đọc Chương 13 và tra cứu method unwrap_or_else trong tài liệu standard library. Nhiều method hơn nữa có thể dọn dẹp các biểu thức match lồng nhau khổng lồ khi bạn xử lý lỗi.

Phím Tắt Để Panic Khi Có Lỗi

Dùng match hoạt động tốt, nhưng có thể hơi dài dòng và không phải lúc nào cũng truyền đạt ý định rõ ràng. Kiểu Result<T, E> có nhiều helper method được định nghĩa trên nó để thực hiện các tác vụ cụ thể hơn, đa dạng hơn. Method unwrap là một shortcut method được implement giống y chang biểu thức match ta đã viết trong Listing 9-4. Nếu giá trị Result là biến thể Ok, unwrap sẽ trả về giá trị bên trong Ok. thread 'main' panicked at src/main.rs:4:49:

Ta dùng expect theo cách tương tự như unwrap: để trả về file handle hoặc gọi macro panic!. Thông báo lỗi được dùng bởi expect trong lời gọi panic! của nó sẽ là tham số mà ta truyền vào expect, thay vì thông báo panic! mặc định mà unwrap dùng. Kết quả trông như thế này:

Trong code production-quality, hầu hết Rustaceans chọn expect thay vì unwrap và đưa ra thêm ngữ cảnh về lý do tại sao thao tác được mong đợi luôn thành công. Như vậy, nếu các giả định của bạn từng bị chứng minh là sai, bạn sẽ có nhiều thông tin hơn để dùng khi debug.

Lan Truyền Lỗi

Khi implementation của một hàm gọi thứ gì đó có thể thất bại, thay vì xử lý lỗi ngay bên trong hàm đó, bạn có thể trả lỗi về cho code gọi nó để code đó quyết định phải làm gì. Đây được gọi là propagating (lan truyền) lỗi và trao thêm quyền kiểm soát cho code gọi, nơi có thể có nhiều thông tin hoặc logic hơn để quyết định cách xử lý lỗi so với những gì bạn có trong ngữ cảnh code của mình.

Ví dụ, Listing 9-6 cho thấy một hàm đọc tên người dùng từ một file. Nếu file không tồn tại hoặc không thể đọc được, hàm này sẽ trả những lỗi đó về cho code đã gọi hàm.

Hàm này có thể được viết theo cách ngắn hơn nhiều, nhưng ta sẽ bắt đầu bằng cách làm nhiều thứ thủ công để khám phá error handling; cuối cùng ta sẽ chỉ cách ngắn hơn. Hãy nhìn vào kiểu trả về của hàm trước: Result<String, io::Error>. Điều này có nghĩa là hàm đang trả về một giá trị kiểu Result<T, E>, nơi tham số generic T đã được điền bằng kiểu cụ thể String và generic E đã được điền bằng kiểu cụ thể io::Error.

Nếu hàm này thành công không có vấn đề gì, code gọi hàm sẽ nhận được giá trị Ok chứa String — tên người dùng mà hàm này đọc từ file. Nếu hàm gặp bất kỳ vấn đề nào, code gọi sẽ nhận được giá trị Err chứa một instance của io::Error có thêm thông tin về những vấn đề gì đã xảy ra. Ta chọn io::Error làm kiểu trả về của hàm này vì đó là kiểu của giá trị lỗi được trả về từ cả hai thao tác ta gọi trong thân hàm có thể thất bại: hàm File::open và method read_to_string.

Thân của hàm bắt đầu bằng cách gọi hàm File::open. Sau đó, ta xử lý giá trị Result bằng một match tương tự như match trong Listing 9-4. Nếu File::open thành công, file handle trong biến pattern file trở thành giá trị trong biến mutable username_file và hàm tiếp tục. Trong trường hợp Err, thay vì gọi panic!, ta dùng từ khóa return để thoát sớm ra khỏi hàm hoàn toàn và truyền giá trị lỗi từ File::open, hiện ở biến pattern e, về cho code gọi như là giá trị lỗi của hàm này.

Vậy, nếu ta có file handle trong username_file, hàm sau đó tạo một String mới trong biến username và gọi method read_to_string trên file handle trong username_file để đọc nội dung của file vào username. Method read_to_string cũng trả về một Result vì nó có thể thất bại, dù File::open đã thành công. Vậy ta cần một match khác để xử lý Result đó: Nếu read_to_string thành công, hàm của ta đã thành công, và ta trả về tên người dùng từ file hiện đang ở username, được bọc trong một Ok. Nếu read_to_string thất bại, ta trả về giá trị lỗi theo cách tương tự như ta đã trả về giá trị lỗi trong match xử lý giá trị trả về của File::open. Tuy nhiên, ta không cần nói rõ return, vì đây là biểu thức cuối cùng trong hàm.

Code gọi hàm này sau đó sẽ xử lý việc nhận được giá trị Ok chứa tên người dùng hoặc giá trị Err chứa một io::Error. Code gọi quyết định phải làm gì với những giá trị đó. Nếu code gọi nhận được giá trị Err, nó có thể gọi panic! và crash chương trình, dùng tên người dùng mặc định, hoặc tra cứu tên người dùng từ nơi khác thay vì từ file, chẳng hạn. Ta không có đủ thông tin về những gì code gọi thực sự đang cố làm, nên ta propagate tất cả thông tin thành công hoặc lỗi lên trên để nó xử lý phù hợp.

Pattern lan truyền lỗi này rất phổ biến trong Rust đến mức Rust cung cấp toán tử dấu hỏi ? để làm điều này dễ dàng hơn.

Toán Tử ? — Phím Tắt Siêu Đỉnh

Listing 9-7 cho thấy một implementation của read_username_from_file có cùng chức năng như trong Listing 9-6, nhưng implementation này dùng toán tử ?.

Dấu ? đặt sau giá trị Result được định nghĩa để hoạt động gần giống như các biểu thức match ta đã định nghĩa để xử lý các giá trị Result trong Listing 9-6. Nếu giá trị của Result là Ok, giá trị bên trong Ok sẽ được trả về từ biểu thức này, và chương trình tiếp tục. Nếu giá trị là Err, Err sẽ được trả về từ toàn bộ hàm như thể ta đã dùng từ khóa return để giá trị lỗi được propagate về cho code gọi.

Có một điểm khác biệt giữa những gì biểu thức match từ Listing 9-6 làm và những gì toán tử ? làm: Các giá trị lỗi có toán tử ? được gọi trên đó sẽ đi qua hàm from, được định nghĩa trong trait From của standard library, dùng để chuyển đổi giá trị từ kiểu này sang kiểu khác. Khi toán tử ? gọi hàm from, kiểu lỗi nhận được sẽ được chuyển đổi thành kiểu lỗi được định nghĩa trong kiểu trả về của hàm hiện tại. Điều này hữu ích khi một hàm trả về một kiểu lỗi để đại diện cho tất cả các cách một hàm có thể thất bại, dù các phần có thể thất bại vì nhiều lý do khác nhau.

Ví dụ, ta có thể thay đổi hàm read_username_from_file trong Listing 9-7 để trả về một kiểu lỗi custom tên OurError mà ta định nghĩa. Nếu ta cũng định nghĩa impl From<io::Error> for OurError để xây dựng một instance của OurError từ một io::Error, thì các lời gọi toán tử ? trong thân hàm read_username_from_file sẽ gọi from và chuyển đổi các kiểu lỗi mà không cần thêm bất kỳ code nào vào hàm.

Trong ngữ cảnh của Listing 9-7, dấu ? ở cuối lời gọi File::open sẽ trả về giá trị bên trong một Ok vào biến username_file. Nếu có lỗi xảy ra, toán tử ? sẽ thoát sớm ra khỏi toàn bộ hàm và đưa bất kỳ giá trị Err nào về cho code gọi. Điều tương tự áp dụng cho dấu ? ở cuối lời gọi read_to_string.

Toán tử ? loại bỏ rất nhiều boilerplate và làm cho implementation của hàm này đơn giản hơn. Ta thậm chí có thể rút ngắn code này hơn nữa bằng cách nối chuỗi các lời gọi method ngay sau ?, như thể hiện trong Listing 9-8.

Ta đã chuyển việc tạo String mới trong username lên đầu hàm; phần đó không thay đổi. Thay vì tạo biến username_file, ta đã nối chuỗi lời gọi read_to_string trực tiếp vào kết quả của File::open("hello.txt")?. Ta vẫn có dấu ? ở cuối lời gọi read_to_string, và ta vẫn trả về giá trị Ok chứa username khi cả File::open và read_to_string đều thành công thay vì trả lỗi. Chức năng vẫn giống như trong Listing 9-6 và Listing 9-7; đây chỉ là cách viết khác, ergonomic hơn.

Listing 9-9 cho thấy cách rút ngắn hơn nữa bằng cách dùng fs::read_to_string.

Đọc file vào một string là thao tác khá phổ biến, vậy nên standard library cung cấp hàm tiện lợi fs::read_to_string mở file, tạo một String mới, đọc nội dung file, đặt nội dung vào String đó, và trả về. Tất nhiên, dùng fs::read_to_string không cho ta cơ hội giải thích toàn bộ error handling, vậy nên ta đã làm theo cách dài hơn trước.

Dùng Toán Tử ? Ở Đâu Được?

Toán tử ? chỉ có thể được dùng trong các hàm có kiểu trả về tương thích với giá trị mà ? được dùng trên đó. Điều này là vì toán tử ? được định nghĩa để thực hiện early return của một giá trị ra khỏi hàm, theo cách tương tự như biểu thức match ta đã định nghĩa trong Listing 9-6. Trong Listing 9-6, match đang dùng giá trị Result, và nhánh early return đã trả về giá trị Err(e). Kiểu trả về của hàm phải là một Result để tương thích với return này.

Trong Listing 9-10, hãy xem lỗi ta sẽ nhận được nếu dùng toán tử ? trong hàm main với kiểu trả về không tương thích với kiểu của giá trị ta dùng ? trên đó.

Code này mở một file, thao tác có thể thất bại. Toán tử ? theo sau giá trị Result được trả về bởi File::open, nhưng hàm main này có kiểu trả về là (), không phải Result. Khi compile code này, ta nhận được thông báo lỗi sau:

Thông báo lỗi này chỉ ra rằng ta chỉ được phép dùng toán tử ? trong hàm trả về Result, Option, hoặc kiểu khác implement FromResidual.

Để sửa lỗi, bạn có hai lựa chọn. Một là thay đổi kiểu trả về của hàm để tương thích với giá trị bạn đang dùng toán tử ? trên đó, miễn là không có ràng buộc nào ngăn cản điều đó. Lựa chọn kia là dùng match hoặc một trong các method của Result<T, E> để xử lý Result<T, E> theo cách phù hợp.

Thông báo lỗi cũng đề cập rằng ? có thể được dùng với các giá trị Option<T>. Cũng như dùng ? trên Result, bạn chỉ có thể dùng ? trên Option trong hàm trả về Option. Hành vi của toán tử ? khi được gọi trên Option<T> tương tự như hành vi của nó khi được gọi trên Result<T, E>: Nếu giá trị là None, None sẽ được trả về sớm từ hàm tại điểm đó. Nếu giá trị là Some, giá trị bên trong Some là giá trị kết quả của biểu thức, và hàm tiếp tục. Listing 9-11 có ví dụ về hàm tìm ký tự cuối cùng của dòng đầu tiên trong text đã cho.

Hàm này trả về Option<char> vì có thể có một ký tự ở đó, nhưng cũng có thể không. Code này lấy string slice argument text và gọi method lines trên nó, method này trả về một iterator qua các dòng trong string. Vì hàm này muốn kiểm tra dòng đầu tiên, nó gọi next trên iterator để lấy giá trị đầu tiên từ iterator. Nếu text là string rỗng, lời gọi next này sẽ trả về None, và khi đó ta dùng ? để dừng và trả về None từ last_char_of_first_line. Nếu text không phải string rỗng, next sẽ trả về giá trị Some chứa một string slice của dòng đầu tiên trong text.

Dấu ? trích xuất string slice, và ta có thể gọi chars trên string slice đó để lấy iterator của các ký tự. Ta quan tâm đến ký tự cuối cùng trong dòng đầu tiên này, vậy ta gọi last để trả về item cuối cùng trong iterator. Đây là một Option vì có thể dòng đầu tiên là string rỗng; ví dụ, nếu text bắt đầu với dòng trống nhưng có ký tự ở các dòng khác, như trong "\nhi". Tuy nhiên, nếu có ký tự cuối cùng ở dòng đầu tiên, nó sẽ được trả về trong biến thể Some. Toán tử ? ở giữa cung cấp cho ta cách ngắn gọn để biểu diễn logic này, cho phép ta implement hàm trong một dòng. Nếu ta không thể dùng toán tử ? trên Option, ta sẽ phải implement logic này dùng nhiều lời gọi method hơn hoặc một biểu thức match.

Lưu ý rằng bạn có thể dùng toán tử ? trên Result trong hàm trả về Result, và bạn có thể dùng toán tử ? trên Option trong hàm trả về Option, nhưng bạn không thể pha trộn và kết hợp. Toán tử ? sẽ không tự động chuyển đổi Result thành Option hoặc ngược lại; trong những trường hợp đó, bạn có thể dùng các method như method ok trên Result hoặc method ok_or trên Option để thực hiện chuyển đổi một cách tường minh.

Cho đến nay, tất cả các hàm main ta đã dùng trả về (). Hàm main đặc biệt vì nó là điểm vào và điểm ra của một chương trình executable, và có các ràng buộc về kiểu trả về của nó để chương trình hoạt động như mong đợi.

May mắn thay, main cũng có thể trả về Result<(), E>. Listing 9-12 có code từ Listing 9-10, nhưng ta đã thay đổi kiểu trả về của main thành Result<(), Box<dyn Error>> và thêm giá trị trả về Ok(()) vào cuối. Code này bây giờ sẽ compile được.

Kiểu Box<dyn Error> là một trait object, ta sẽ nói về nó trong "Dùng Trait Objects để Trừu Tượng Hóa Hành Vi Chung" trong Chương 18. Hiện tại, bạn có thể đọc Box<dyn Error> là "bất kỳ loại lỗi nào." Dùng ? trên giá trị Result trong hàm main với kiểu lỗi Box<dyn Error> được phép vì nó cho phép bất kỳ giá trị Err nào được trả về sớm. Dù thân hàm main này chỉ trả về lỗi kiểu std::io::Error, bằng cách chỉ định Box<dyn Error>, signature này sẽ tiếp tục đúng ngay cả khi thêm nhiều code trả về lỗi khác vào thân main.

Khi hàm main trả về Result<(), E>, executable sẽ thoát với giá trị 0 nếu main trả về Ok(()) và sẽ thoát với giá trị khác không nếu main trả về giá trị Err. Các executable viết bằng C trả về integers khi thoát: Các chương trình thoát thành công trả về integer 0, và các chương trình có lỗi trả về một integer khác 0. Rust cũng trả về integers từ executables để tương thích với convention này.

Hàm main có thể trả về bất kỳ kiểu nào implement trait std::process::Termination, chứa hàm report trả về một ExitCode. Tham khảo tài liệu standard library để biết thêm thông tin về cách implement trait Termination cho các kiểu của riêng bạn.

Bây giờ ta đã thảo luận xong chi tiết về việc gọi panic! hoặc trả về Result, hãy quay lại chủ đề làm thế nào để quyết định cái nào phù hợp để dùng trong tình huống nào.
