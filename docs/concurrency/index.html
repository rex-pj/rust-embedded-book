Concurrency

Xử lý đồng thời an toàn trong embedded Rust

Concurrency xảy ra bất cứ khi nào các phần khác nhau của chương trình bạn có thể thực thi vào các thời điểm khác nhau hoặc theo thứ tự khác nhau. Trong ngữ cảnh embedded, điều này bao gồm:

  • Interrupt handlers — chạy bất cứ khi nào interrupt tương ứng xảy ra,
  • Đa luồng — vi xử lý thường xuyên hoán đổi giữa các phần của chương trình,
  • Đa lõi — một số hệ thống có nhiều lõi, mỗi lõi có thể chạy một phần khác nhau của chương trình cùng lúc.

Vì nhiều chương trình embedded cần xử lý interrupt, concurrency thường xuất hiện sớm hay muộn, và đây cũng là nơi nhiều bug tinh vi và khó phát hiện có thể xảy ra. May mắn thay, Rust cung cấp một số abstraction và bảo đảm an toàn để giúp chúng ta viết code đúng đắn.

Các nguồn gốc của Concurrency trong Embedded

Không có Concurrency

Dạng concurrency đơn giản nhất cho chương trình embedded là không có concurrency: phần mềm của bạn gồm một vòng lặp main duy nhất cứ chạy mãi mãi, và không có interrupt nào cả. Đôi khi đây là giải pháp hoàn toàn phù hợp! Thông thường vòng lặp của bạn sẽ đọc một số input, thực hiện xử lý nào đó, và ghi một số output. 

#[entry]
fn main() {
    let peripherals = setup_peripherals();
    loop {
        let inputs = read_inputs(&peripherals);
        let outputs = process(inputs);
        write_outputs(&peripherals, outputs);
    }
}

Vì không có concurrency, không cần lo lắng về việc chia sẻ dữ liệu giữa các phần của chương trình hay đồng bộ hóa truy cập vào peripherals. Nếu bạn có thể dùng cách tiếp cận đơn giản như vậy, đây có thể là một giải pháp tuyệt vời.

Dữ liệu Mutable Toàn cục (Global Mutable Data)

Khác với Rust thông thường, trong embedded thường chúng ta không có điều kiện thoải mái là cấp phát heap và truyền reference đến dữ liệu đó vào một thread mới được tạo. Thay vào đó, interrupt handler của chúng ta có thể được gọi bất cứ lúc nào và phải biết cách truy cập bộ nhớ dùng chung mà chúng ta đang sử dụng. Ở mức thấp nhất, điều này có nghĩa là chúng ta phải có bộ nhớ mutable được cấp phát tĩnh, mà cả interrupt handler lẫn code chính đều có thể tham chiếu đến.

Trong Rust, các biến static mut như vậy luôn không an toàn khi đọc hoặc ghi, bởi vì nếu không cẩn thận, bạn có thể gây ra một race condition — truy cập vào biến bị interrupt giữa chừng bởi một interrupt handler cũng truy cập vào biến đó.

Ví dụ về cách hành vi này có thể gây ra lỗi tinh vi — hãy xét một chương trình embedded đếm rising edges của tín hiệu input trong mỗi chu kỳ một giây (bộ đếm tần số): 

static mut COUNTER: u32 = 0;

#[entry]
fn main() -> ! {
    set_timer_1hz();
    let mut last_state = false;
    loop {
        let state = read_signal_level();
        if state && !last_state {
            // NGUY HIỂM - Thực ra không an toàn! Có thể gây data race.
            unsafe { COUNTER += 1 };
        }
        last_state = state;
    }
}

#[interrupt]
fn timer() {
    unsafe { COUNTER = 0; }
}

Mỗi giây, timer interrupt đặt lại counter về 0. Trong khi đó, vòng lặp main liên tục đo tín hiệu và tăng counter khi thấy thay đổi từ low sang high. Chúng ta phải dùng unsafe để truy cập COUNTER vì nó là static mut, và điều đó có nghĩa là chúng ta đang hứa với compiler rằng chúng ta sẽ không gây ra undefined behaviour. Bạn có thấy race condition không?

Race Condition: Tại sao COUNTER += 1 không an toàn

Phép tăng trên COUNTER không được đảm bảo là atomic — thực tế, trên hầu hết các nền tảng embedded, nó sẽ được chia thành một lệnh load, sau đó tăng, rồi store. Nếu interrupt kích hoạt sau khi load nhưng trước khi store, việc reset về 0 sẽ bị bỏ qua sau khi interrupt return — và chúng ta sẽ đếm gấp đôi số lần transition trong chu kỳ đó.

Critical Sections (Vùng Nguy Cơ)

Vậy chúng ta có thể làm gì về data races? Một cách tiếp cận đơn giản là dùng critical sections — một ngữ cảnh nơi interrupts bị vô hiệu hóa. Bằng cách bọc việc truy cập COUNTER trong main trong một critical section, chúng ta có thể chắc chắn rằng timer interrupt sẽ không kích hoạt cho đến khi chúng ta tăng xong COUNTER: 

static mut COUNTER: u32 = 0;

#[entry]
fn main() -> ! {
    set_timer_1hz();
    let mut last_state = false;
    loop {
        let state = read_signal_level();
        if state && !last_state {
            // Critical section mới đảm bảo truy cập đồng bộ vào COUNTER
            cortex_m::interrupt::free(|_| {
                unsafe { COUNTER += 1 };
            });
        }
        last_state = state;
    }
}

#[interrupt]
fn timer() {
    unsafe { COUNTER = 0; }
}

Trong ví dụ này, chúng ta dùng cortex_m::interrupt::free, nhưng các nền tảng khác sẽ có cơ chế tương tự để thực thi code trong critical section. Về bản chất đây giống như disable interrupts, chạy code nào đó, rồi re-enable interrupts.

Lưu ý rằng chúng ta không cần đặt critical section bên trong timer interrupt, vì hai lý do:

  • Ghi 0 vào COUNTER không bị ảnh hưởng bởi race vì chúng ta không đọc nó
  • Timer interrupt sẽ không bao giờ bị ngắt bởi thread main

Nếu COUNTER được chia sẻ bởi nhiều interrupt handler có thể preempt nhau, thì mỗi handler cũng sẽ cần một critical section.

Critical section giải quyết vấn đề trước mắt, nhưng chúng ta vẫn phải viết nhiều code unsafe cần suy nghĩ cẩn thận. Mỗi critical section tạm thời tạm dừng xử lý interrupt, có chi phí code size lớn hơn và interrupt latency cao hơn. Ngoài ra, critical section không đảm bảo truy cập độc quyền trên hệ thống đa lõi!

Atomic Access

Trên một số nền tảng, các lệnh atomic đặc biệt có sẵn, cung cấp bảo đảm về các thao tác read-modify-write. Cụ thể với Cortex-M: thumbv6 (Cortex-M0, M0+) chỉ cung cấp load và store atomic, trong khi thumbv7 (Cortex-M3 trở lên) cung cấp đầy đủ lệnh Compare and Swap (CAS). Những thao tác atomic này an toàn ngay cả trên nhiều lõi. 

use core::sync::atomic::{AtomicUsize, Ordering};

static COUNTER: AtomicUsize = AtomicUsize::new(0);

#[entry]
fn main() -> ! {
    set_timer_1hz();
    let mut last_state = false;
    loop {
        let state = read_signal_level();
        if state && !last_state {
            // Dùng `fetch_add` để cộng 1 vào COUNTER một cách atomic
            COUNTER.fetch_add(1, Ordering::Relaxed);
        }
        last_state = state;
    }
}

#[interrupt]
fn timer() {
    // Dùng `store` để ghi 0 trực tiếp vào COUNTER
    COUNTER.store(0, Ordering::Relaxed)
}

Lần này COUNTER là một biến static an toàn. Nhờ kiểu AtomicUsize, COUNTER có thể được sửa đổi an toàn từ cả interrupt handler lẫn thread chính mà không cần disable interrupts.

Một lưu ý về Ordering: tham số này ảnh hưởng đến cách compiler và hardware có thể sắp xếp lại lệnh, và cũng có tác động đến khả năng hiển thị của cache. Giả sử target là nền tảng single-core, Relaxed là đủ và là lựa chọn hiệu quả nhất trong trường hợp cụ thể này. Ordering nghiêm ngặt hơn sẽ khiến compiler emit memory barriers xung quanh các thao tác atomic.

Abstraction, Send và Sync

Không có giải pháp nào ở trên thực sự thỏa mãn. Chúng đều yêu cầu các block unsafe phải được kiểm tra rất cẩn thận và không ergonomic. Chắc chắn chúng ta có thể làm tốt hơn trong Rust!

Chúng ta có thể abstract counter của mình thành một interface an toàn có thể dùng an toàn ở bất kỳ đâu trong code. Trong ví dụ này, chúng ta sẽ dùng critical-section counter, nhưng bạn có thể làm điều gì đó rất tương tự với atomics. 

use core::cell::UnsafeCell;
use cortex_m::interrupt;

// Counter của chúng ta chỉ là một wrapper xung quanh UnsafeCell<u32>,
// đây là trung tâm của interior mutability trong Rust. Bằng cách dùng
// interior mutability, chúng ta có thể để COUNTER là `static` thay vì
// `static mut`, nhưng vẫn có thể thay đổi giá trị counter của nó.
struct CSCounter(UnsafeCell<u32>);

const CS_COUNTER_INIT: CSCounter = CSCounter(UnsafeCell::new(0));

impl CSCounter {
    pub fn reset(&self, _cs: &interrupt::CriticalSection) {
        // Bằng cách yêu cầu CriticalSection được truyền vào, chúng ta biết
        // mình đang ở bên trong một CriticalSection và có thể dùng unsafe block này.
        unsafe { *self.0.get() = 0 };
    }

    pub fn increment(&self, _cs: &interrupt::CriticalSection) {
        unsafe { *self.0.get() += 1 };
    }
}

// Cần thiết để cho phép static CSCounter. Xem giải thích bên dưới.
unsafe impl Sync for CSCounter {}

// COUNTER không còn là `mut` vì nó dùng interior mutability;
// do đó cũng không cần unsafe blocks để truy cập nữa.
static COUNTER: CSCounter = CS_COUNTER_INIT;

#[entry]
fn main() -> ! {
    set_timer_1hz();
    let mut last_state = false;
    loop {
        let state = read_signal_level();
        if state && !last_state {
            // Không có unsafe ở đây!
            interrupt::free(|cs| COUNTER.increment(cs));
        }
        last_state = state;
    }
}

#[interrupt]
fn timer() {
    // Chúng ta cần enter một critical section ở đây chỉ để lấy một token cs hợp lệ,
    // dù chúng ta biết không có interrupt nào khác có thể preempt cái này.
    interrupt::free(|cs| COUNTER.reset(cs));
}

Chúng ta đã chuyển code unsafe của mình vào bên trong abstraction được lên kế hoạch cẩn thận, và giờ code ứng dụng của chúng ta không chứa bất kỳ block unsafe nào.

Thiết kế này yêu cầu application truyền một token CriticalSection vào: những token này chỉ được tạo ra an toàn bởi interrupt::free, vì vậy bằng cách yêu cầu một cái được truyền vào, chúng ta đảm bảo rằng chúng ta đang hoạt động bên trong một critical section, mà không phải thực sự tự làm lock. Bảo đảm này được cung cấp tĩnh bởi compiler — sẽ không có runtime overhead nào liên quan đến cs.

Điều này cũng đề cập đến một chủ đề quan trọng cho concurrency trong Rust: các trait Send và Sync. Tóm tắt: một kiểu là Send khi nó có thể được an toàn chuyển sang thread khác, trong khi nó là Sync khi nó có thể được chia sẻ an toàn giữa nhiều thread. Trong ngữ cảnh embedded, chúng ta coi interrupts là đang thực thi trong một thread riêng biệt so với code ứng dụng, vì vậy các biến được truy cập bởi cả interrupt lẫn code chính phải là Sync.

Vì CSCounter chứa một UnsafeCell, nó không phải Sync, và do đó chúng ta không thể tạo một static CSCounter: biến static phải là Sync. Để nói với compiler rằng chúng ta đã đảm bảo CSCounter thực sự an toàn khi chia sẻ giữa các thread, chúng ta implement trait Sync một cách tường minh.

Mutex

Chúng ta đã tạo một abstraction hữu ích dành riêng cho vấn đề counter của mình, nhưng có nhiều abstraction phổ biến được dùng cho concurrency.

Một synchronisation primitive như vậy là mutex (mutual exclusion). Những cấu trúc này đảm bảo truy cập độc quyền vào một biến. Một thread có thể thử lock (acquire) mutex, và hoặc thành công ngay lập tức, hoặc block chờ lock được acquire, hoặc trả về lỗi. Khi thread đó giữ lock, nó được cấp quyền truy cập vào dữ liệu được bảo vệ. Khi thread xong, nó unlock (release) mutex.

Sử dụng mutex với interrupt handler có thể phức tạp: thông thường không chấp nhận được việc interrupt handler block, và sẽ thực sự thảm họa nếu nó block chờ thread chính release lock, vì khi đó chúng ta sẽ deadlock. Deadlock không được coi là unsafe — nó có thể xảy ra ngay cả trong safe Rust.

Để tránh hoàn toàn hành vi này, chúng ta có thể implement một mutex yêu cầu critical section để lock. Điều này đã được làm sẵn trong crate cortex_m: 

use core::cell::Cell;
use cortex_m::interrupt::Mutex;

static COUNTER: Mutex<Cell<u32>> = Mutex::new(Cell::new(0));

#[entry]
fn main() -> ! {
    set_timer_1hz();
    let mut last_state = false;
    loop {
        let state = read_signal_level();
        if state && !last_state {
            interrupt::free(|cs|
                COUNTER.borrow(cs).set(COUNTER.borrow(cs).get() + 1));
        }
        last_state = state;
    }
}

#[interrupt]
fn timer() {
    // Chúng ta vẫn cần enter critical section để thỏa mãn Mutex.
    interrupt::free(|cs| COUNTER.borrow(cs).set(0));
}

Chúng ta dùng Cell để cung cấp interior mutability an toàn. Cell cho phép lấy một bản copy của giá trị hiện tại hoặc thay thế nó, nhưng không lấy reference — và vì nó không phải Sync nên không thể chia sẻ giữa các thread.

Mutex<T> implement Sync cho bất kỳ T nào là Send — chẳng hạn như Cell. Nó có thể làm điều này một cách an toàn vì nó chỉ cho phép truy cập nội dung của mình trong một critical section. Kết quả là chúng ta có được một counter an toàn hoàn toàn không có unsafe code!

Chia sẻ Peripherals

Device crates được tạo bằng svd2rust và các abstraction tương tự cung cấp truy cập an toàn vào peripherals bằng cách đảm bảo chỉ tồn tại một instance của peripheral struct tại một thời điểm. Điều này đảm bảo an toàn, nhưng gây khó khăn khi truy cập peripheral từ cả thread chính lẫn interrupt handler.

Để chia sẻ peripheral một cách an toàn, chúng ta có thể dùng Mutex như đã thấy. Chúng ta cũng cần dùng RefCell để đảm bảo chỉ có một reference đến peripheral được cấp ra tại một thời điểm. Cuối cùng, chúng ta dùng kiểu Option để khởi tạo với None và sau đó set thành peripheral. 

use core::cell::RefCell;
use cortex_m::interrupt::{self, Mutex};
use stm32f4::stm32f405;

static MY_GPIO: Mutex<RefCell<Option<stm32f405::GPIOA>>> =
    Mutex::new(RefCell::new(None));

#[entry]
fn main() -> ! {
    // Lấy peripheral singletons và cấu hình nó.
    let dp = stm32f405::Peripherals::take().unwrap();
    let gpioa = &dp.GPIOA;

    // Một hàm cấu hình nào đó.
    // Giả sử nó đặt PA0 là input và PA1 là output.
    configure_gpio(gpioa);

    // Lưu GPIOA vào mutex (di chuyển vào).
    interrupt::free(|cs| MY_GPIO.borrow(cs).replace(Some(dp.GPIOA)));
    // Chúng ta không thể dùng `gpioa` hay `dp.GPIOA` nữa, thay vào đó
    // phải truy cập qua mutex.

    // Hãy cẩn thận bật interrupt chỉ SAU KHI đặt MY_GPIO:
    // nếu không, interrupt có thể kích hoạt khi nó vẫn chứa None,
    // và với unwrap(), nó sẽ panic.
    set_timer_1hz();
    let mut last_state = false;
    loop {
        // Bây giờ chúng ta đọc state như một digital input, qua mutex
        let state = interrupt::free(|cs| {
            let gpioa = MY_GPIO.borrow(cs).borrow();
            gpioa.as_ref().unwrap().idr.read().idr0().bit_is_set()
        });

        if state && !last_state {
            // Đặt PA1 high nếu chúng ta thấy rising edge trên PA0.
            interrupt::free(|cs| {
                let gpioa = MY_GPIO.borrow(cs).borrow();
                gpioa.as_ref().unwrap().odr.modify(|_, w| w.odr1().set_bit());
            });
        }
        last_state = state;
    }
}

#[interrupt]
fn timer() {
    // Lần này trong interrupt chúng ta chỉ clear PA0.
    interrupt::free(|cs| {
        let gpioa = MY_GPIO.borrow(cs).borrow();
        gpioa.as_ref().unwrap().odr.modify(|_, w| w.odr1().clear_bit());
    });
}

Đó là khá nhiều thứ cần tiếp thu, hãy phân tích các dòng quan trọng:

static MY_GPIO: Mutex<RefCell<Option<stm32f405::GPIOA>>> =
    Mutex::new(RefCell::new(None));
Tại sao cần cả ba lớp: Mutex<RefCell<Option<T>>>?

Shared variable của chúng ta giờ là một Mutex bao quanh một RefCell chứa một Option. Mutex đảm bảo chúng ta chỉ có quyền truy cập trong critical section, và do đó khiến biến Sync. RefCell cung cấp interior mutability với references. Option cho phép khởi tạo biến này với giá trị rỗng, và chỉ sau đó thực sự move biến vào.

RTIC

Một lựa chọn thay thế là RTIC framework (Real Time Interrupt-driven Concurrency). Nó thực thi các priority tĩnh và theo dõi truy cập vào các biến static mut ("resources") để đảm bảo tĩnh rằng các resource dùng chung luôn được truy cập an toàn, mà không cần overhead của việc luôn phải vào critical section và dùng reference counting (như RefCell).

Điều này có nhiều ưu điểm như đảm bảo không deadlock và overhead thời gian và bộ nhớ cực thấp.

Framework cũng bao gồm các tính năng khác như message passing, giúp giảm nhu cầu shared state tường minh, và khả năng lên lịch task chạy vào thời điểm cụ thể, dùng để implement periodic tasks. Xem tài liệu RTIC để biết thêm!

Real-Time Operating Systems (RTOS)

Một mô hình phổ biến khác cho embedded concurrency là real-time operating system (RTOS). Dù hiện tại ít được khám phá hơn trong Rust, chúng được dùng rộng rãi trong phát triển embedded truyền thống. Các ví dụ open source bao gồm FreeRTOS và ChibiOS. Những RTOS này cung cấp hỗ trợ chạy nhiều application thread mà CPU hoán đổi giữa chúng, hoặc khi thread yield quyền kiểm soát (cooperative multitasking) hoặc dựa trên timer hoặc interrupt đều đặn (preemptive multitasking).

Đa lõi (Multiple Cores)

Ngày càng phổ biến có hai hoặc nhiều lõi trong vi xử lý embedded, điều này thêm một lớp phức tạp cho concurrency. Tất cả các ví dụ dùng critical section (bao gồm cả cortex_m::interrupt::Mutex) giả định rằng execution thread duy nhất khác là interrupt thread, nhưng trên hệ thống đa lõi điều đó không còn đúng nữa. Thay vào đó, chúng ta sẽ cần các synchronisation primitive được thiết kế cho nhiều lõi (còn gọi là SMP — symmetric multi-processing).

Những primitive này thường dùng các lệnh atomic mà chúng ta đã thấy trước đó, vì hệ thống xử lý sẽ đảm bảo tính atomic được duy trì trên tất cả các lõi.

Đề cập chi tiết các chủ đề này hiện vượt quá phạm vi của cuốn sách này, nhưng các pattern chung đều giống như cho trường hợp single-core.
