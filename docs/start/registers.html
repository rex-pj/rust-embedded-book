Memory-mapped Registers

Điều khiển phần cứng thông qua địa chỉ bộ nhớ đặc biệt

Các hệ thống nhúng chỉ có thể làm được nhiều việc bằng cách thực thi code Rust thông thường và di chuyển dữ liệu trong RAM. Nếu chúng ta muốn đưa thông tin vào hoặc ra khỏi hệ thống (dù là nhấp nháy LED, phát hiện nhấn nút hay giao tiếp với peripheral bên ngoài chip qua bus nào đó) chúng ta cần phải đi vào thế giới của Peripherals và 'memory-mapped registers'.

Bạn có thể thấy rằng code cần để truy cập peripherals trong vi điều khiển của bạn đã được viết sẵn, ở một trong các mức sau:

Bốn tầng trừu tượng trong hệ sinh thái Rust Embedded

Board Crate

Một board crate là điểm khởi đầu hoàn hảo nếu bạn mới làm quen với embedded Rust. Chúng trừu tượng hóa các chi tiết HW có thể gây ngợp khi bắt đầu học, và làm cho các tác vụ chuẩn trở nên dễ dàng, như bật hoặc tắt LED.

Nếu bạn muốn thử nghiệm với board STM32F3DISCOVERY, rất khuyến nghị bạn xem crate stm32f3-discovery, cung cấp tính năng nhấp nháy LED, truy cập compass, bluetooth và nhiều hơn nữa. Cuốn sách Discovery cũng cung cấp phần giới thiệu tuyệt vời về việc dùng board crate.

Nhưng nếu bạn đang làm việc trên hệ thống chưa có board crate riêng, hoặc bạn cần tính năng không có trong các crate hiện có, hãy đọc tiếp khi chúng ta bắt đầu từ tầng dưới cùng — micro-architecture crates.

Micro-architecture Crate

Hãy xét peripheral SysTick có trong tất cả các vi điều khiển dựa trên Cortex-M. Chúng ta có thể tìm thấy một API khá low-level trong crate cortex-m, và dùng nó như thế này: 

#![no_std]
#![no_main]
use cortex_m::peripheral::{syst, Peripherals};
use cortex_m_rt::entry;
use panic_halt as _;

#[entry]
fn main() -> ! {
    let peripherals = Peripherals::take().unwrap();
    let mut systick = peripherals.SYST;
    systick.set_clock_source(syst::SystClkSource::Core);
    systick.set_reload(1_000);
    systick.clear_current();
    systick.enable_counter();
    while !systick.has_wrapped() {
        // Loop
    }

    loop {}
}

Các hàm trên struct SYST ánh xạ khá sát với tính năng được định nghĩa bởi ARM Technical Reference Manual cho peripheral này. Không có gì trong API này về 'delay X milliseconds' — chúng ta phải tự implement thủ công bằng vòng lặp while.

Lưu ý rằng chúng ta không thể truy cập struct SYST cho đến khi gọi Peripherals::take() — đây là một routine đặc biệt đảm bảo rằng chỉ có một struct SYST trong toàn bộ chương trình. Đây là Singleton Pattern được thực thi tại runtime — nếu bạn gọi take() lần thứ hai, nó trả về None, ngăn ngừa aliasing không an toàn.

Dùng PAC (Peripheral Access Crate)

Chúng ta sẽ không đi được xa với phát triển phần mềm embedded nếu chỉ giới hạn bản thân với các peripheral cơ bản có trong mọi Cortex-M. Tại một thời điểm nào đó, chúng ta sẽ cần viết code dành riêng cho vi điều khiển cụ thể đang dùng.

Trong ví dụ này, giả sử chúng ta có Texas Instruments TM4C123 — một Cortex-M4 80MHz trung cấp với 256 KiB Flash. Chúng ta sẽ kéo vào crate tm4c123x: 

#![no_std]
#![no_main]

use panic_halt as _;
use cortex_m_rt::entry;
use tm4c123x;

#[entry]
pub fn init() {
    let cp = cortex_m::Peripherals::take().unwrap();
    let p = tm4c123x::Peripherals::take().unwrap();

    let pwm = p.PWM0;
    pwm.ctl.write(|w| w.globalsync0().clear_bit());
    // Mode = 1 => Count up/down mode
    pwm._2_ctl.write(|w| w.enable().set_bit().mode().set_bit());
    pwm._2_gena.write(|w| w.actcmpau().zero().actcmpad().one());
    // 528 cycles (264 up và down) = 4 loops mỗi video line (2112 cycles)
    pwm._2_load.write(|w| unsafe { w.load().bits(263) });
    pwm._2_cmpa.write(|w| unsafe { w.compa().bits(64) });
    pwm.enable.write(|w| w.pwm4en().set_bit());
}

Chúng ta truy cập peripheral PWM0 theo cách giống hệt như chúng ta truy cập peripheral SYST trước đó. Vì crate này được auto-generated bằng svd2rust, các hàm truy cập cho các trường register của chúng ta nhận một closure thay vì một đối số số.

Mặc dù trông như rất nhiều code, Rust compiler có thể dùng nó để thực hiện một loạt kiểm tra, nhưng sau đó sinh ra machine-code khá gần với assembler viết tay! Khi code auto-generated không thể xác định rằng tất cả các đối số có thể cho một accessor function cụ thể đều hợp lệ, thì hàm đó được đánh dấu là unsafe.

Đọc (Reading)

Hàm read() trả về một object cho phép truy cập read-only các sub-field khác nhau trong register này, như được định nghĩa bởi SVD file của nhà sản xuất: 

if pwm.ctl.read().globalsync0().is_set() {
    // Làm gì đó
}

Ghi (Writing)

Hàm write() nhận một closure với một đối số duy nhất. Thông thường chúng ta gọi nó là w. Đối số này sau đó cung cấp quyền truy cập đọc-ghi vào các sub-field khác nhau trong register này. Lưu ý rằng tất cả các sub-field mà chúng ta không set sẽ được đặt thành giá trị mặc định — bất kỳ nội dung hiện có nào trong register sẽ bị mất: 

pwm.ctl.write(|w| w.globalsync0().clear_bit());

Sửa đổi (Modifying)

Nếu chúng ta muốn thay đổi chỉ một sub-field cụ thể trong register và để nguyên các sub-field khác, chúng ta có thể dùng hàm modify. Hàm này nhận một closure với hai đối số — một để đọc và một để ghi. Thông thường chúng ta gọi chúng là r và w. Đối số r dùng để kiểm tra nội dung hiện tại của register, và đối số w dùng để sửa đổi nội dung register: 

pwm.ctl.modify(|r, w| w.globalsync0().clear_bit());

Hàm modify thực sự cho thấy sức mạnh của closures ở đây. Trong C, chúng ta phải đọc vào một giá trị tạm, sửa đổi các bit đúng rồi ghi giá trị trở lại. Điều này có khả năng gây lỗi đáng kể: 

// Trong C — dễ mắc lỗi!
uint32_t temp = pwm0.ctl.read();
temp |= PWM0_CTL_GLOBALSYNC0;
pwm0.ctl.write(temp);
uint32_t temp2 = pwm0.enable.read();
temp2 |= PWM0_ENABLE_PWM4EN;
pwm0.enable.write(temp);  // Ôi! Sai biến! temp thay vì temp2
So sánh read(), write(), modify()

Dùng HAL Crate

HAL crate cho một chip thường hoạt động bằng cách implement một Custom Trait cho các cấu trúc thô được PAC cung cấp. Thường trait này sẽ định nghĩa một hàm constrain() cho các peripheral đơn lẻ hoặc split() cho các thứ như GPIO port với nhiều pin. Hàm này sẽ tiêu thụ cấu trúc peripheral thô bên dưới và trả về một object mới với API cấp cao hơn.

API này cũng có thể làm những thứ như yêu cầu hàm new của Serial port phải borrow một struct Clock nào đó, thứ chỉ có thể được tạo ra bằng cách gọi hàm cấu hình PLL và thiết lập tần số clock. Theo cách này, việc tạo một Serial port object mà không cấu hình tốc độ clock trước là statically impossible — compiler sẽ từ chối. Tất cả điều này mà không tốn chi phí runtime!

Hãy xem một ví dụ:

#![no_std]
#![no_main]

use panic_halt as _;
use cortex_m_rt::entry;
use tm4c123x_hal as hal;
use tm4c123x_hal::prelude::*;
use tm4c123x_hal::serial::{NewlineMode, Serial};
use tm4c123x_hal::sysctl;

#[entry]
fn main() -> ! {
    let p = hal::Peripherals::take().unwrap();
    let cp = hal::CorePeripherals::take().unwrap();

    // Bọc SYSCTL struct thành object với API cấp cao hơn
    let mut sc = p.SYSCTL.constrain();
    // Chọn cài đặt oscillator
    sc.clock_setup.oscillator = sysctl::Oscillator::Main(
        sysctl::CrystalFrequency::_16mhz,
        sysctl::SystemClock::UsePll(sysctl::PllOutputFrequency::_80_00mhz),
    );
    // Cấu hình PLL với các cài đặt đó
    let clocks = sc.clock_setup.freeze();

    // Bọc GPIO_PORTA struct thành object với API cấp cao hơn.
    // Lưu ý cần borrow `sc.power_control` để tự động cấp nguồn cho peripheral GPIO
    let mut porta = p.GPIO_PORTA.split(&sc.power_control);

    // Kích hoạt UART
    let uart = Serial::uart0(
        p.UART0,
        // Pin truyền (TX)
        porta.pa1.into_af_push_pull::<hal::gpio::AF1>(&mut porta.control),
        // Pin nhận (RX)
        porta.pa0.into_af_push_pull::<hal::gpio::AF1>(&mut porta.control),
        (),          // Không cần RTS
        (),          // Không cần CTS
        115200_u32.bps(),    // Baud rate
        NewlineMode::SwapLFtoCRLF,
        &clocks,     // Cần tốc độ clock để tính baud rate divisor
        &sc.power_control,  // Cần để cấp nguồn cho UART
    );

    loop {
        writeln!(uart, "Hello, World!\r\n").unwrap();
    }
}

Lưu ý cách Serial port new function yêu cầu:

  • Pin đúng loại — into_af_push_pull::<AF1>() đặt pin vào Alternate Function 1 (UART) — nếu bạn truyền sai pin, compiler sẽ báo lỗi!
  • Tham chiếu đến clocks — đảm bảo bạn đã cấu hình clock trước. Không thể tạo Serial port mà không có điều này.
  • Tham chiếu đến power_control — HAL tự động cấp nguồn cho peripheral.

Tất cả các bảo đảm này được thực thi tại compile time, không tốn bất kỳ chi phí runtime nào.

Nên bắt đầu với HAL crate cho chip của bạn nếu có. Chỉ xuống PAC khi bạn cần truy cập các thanh ghi mà HAL chưa wrap, hoặc xuống micro-architecture crate cho các tính năng chung của lõi CPU.
