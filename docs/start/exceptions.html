Ngoại lệ (Exceptions)

Exception và interrupt là cơ chế phần cứng mà bộ xử lý dùng để xử lý các sự kiện bất đồng bộ và lỗi nghiêm trọng (ví dụ: thực thi lệnh không hợp lệ). Exception có nghĩa là preemption và liên quan đến các exception handler, tức là các subroutine được thực thi để phản hồi với tín hiệu kích hoạt sự kiện.

Crate cortex-m-rt cung cấp thuộc tính exception để khai báo exception handler. 

// Exception handler cho SysTick (System Timer) exception
#[exception]
fn SysTick() {
    // ..
}

Ngoài thuộc tính exception, các exception handler trông giống như các hàm bình thường nhưng có thêm một điểm khác biệt: các exception handler không thể được gọi bởi phần mềm. Theo ví dụ trước, câu lệnh SysTick(); sẽ dẫn đến lỗi biên dịch.

Hành vi này là hoàn toàn có chủ ý và cần thiết để cung cấp một tính năng: các biến static mut được khai báo bên trong exception handler là an toàn để dùng. 

#[exception]
fn SysTick() {
    static mut COUNT: u32 = 0;

    // `COUNT` đã được chuyển thành kiểu `&mut u32` và an toàn để dùng
    *COUNT += 1;
}

Như bạn có thể biết, việc dùng biến static mut trong một hàm làm cho hàm đó non-reentrant. Hành vi không xác định xảy ra khi gọi một hàm non-reentrant, trực tiếp hoặc gián tiếp, từ nhiều hơn một exception / interrupt handler hoặc từ main và một hoặc nhiều exception / interrupt handler.

Safe Rust không bao giờ được dẫn đến hành vi không xác định, vì vậy các hàm non-reentrant phải được đánh dấu là unsafe. Thế nhưng tôi vừa nói rằng exception handler có thể dùng biến static mut một cách an toàn. Làm sao điều này có thể xảy ra? Điều này có thể xảy ra vì exception handler không thể được gọi bởi phần mềm, do đó reentrancy là không thể. Các handler này được gọi bởi chính phần cứng, được giả định là không có concurrent về mặt vật lý.

Kết quả là, trong ngữ cảnh của exception handler trong các hệ thống embedded, sự vắng mặt của các lần gọi đồng thời cùng một handler đảm bảo không có vấn đề reentrancy, ngay cả khi handler dùng static mutable variable.

Trong hệ thống multicore, nơi nhiều processor core đang thực thi code đồng thời, khả năng xảy ra reentrancy lại trở nên liên quan, ngay cả trong exception handler. Để giải quyết vấn đề này trong môi trường multicore, cần dùng các cơ chế đồng bộ hóa phù hợp trong exception handler như lock, semaphore, hoặc atomic operation.

Lưu ý rằng thuộc tính exception chuyển đổi các định nghĩa biến static bên trong hàm bằng cách bọc chúng vào các block unsafe và cung cấp cho chúng ta các biến mới thích hợp có kiểu &mut cùng tên. Vì vậy chúng ta có thể dereference reference qua * để truy cập giá trị của biến mà không cần bọc trong block unsafe.

Một ví dụ hoàn chỉnh

Đây là ví dụ dùng system timer để raise exception SysTick khoảng mỗi giây. Exception handler SysTick theo dõi số lần nó đã được gọi trong biến COUNT rồi in giá trị của COUNT ra host console bằng semihosting.

LƯU Ý: Bạn có thể chạy ví dụ này trên bất kỳ thiết bị Cortex-M nào; bạn cũng có thể chạy nó trên QEMU.

#![deny(unsafe_code)]
#![no_main]
#![no_std]

use panic_halt as _;

use core::fmt::Write;

use cortex_m::peripheral::syst::SystClkSource;
use cortex_m_rt::{entry, exception};
use cortex_m_semihosting::{
    debug,
    hio::{self, HostStream},
};

#[entry]
fn main() -> ! {
    let p = cortex_m::Peripherals::take().unwrap();
    let mut syst = p.SYST;

    // cấu hình system timer để trigger SysTick exception mỗi giây
    syst.set_clock_source(SystClkSource::Core);
    // cấu hình cho LM3S6965 có CPU clock mặc định 12 MHz
    syst.set_reload(12_000_000);
    syst.clear_current();
    syst.enable_counter();
    syst.enable_interrupt();

    loop {}
}

#[exception]
fn SysTick() {
    static mut COUNT: u32 = 0;
    static mut STDOUT: Option<HostStream> = None;

    *COUNT += 1;

    // Lazy initialization
    if STDOUT.is_none() {
        *STDOUT = hio::hstdout().ok();
    }

    if let Some(hstdout) = STDOUT.as_mut() {
        write!(hstdout, "{}", *COUNT).ok();
    }

    // QUAN TRỌNG: bỏ qua block `if` này nếu chạy trên phần cứng thực hoặc
    // debugger sẽ rơi vào trạng thái không nhất quán
    if *COUNT == 9 {
        // Kết thúc tiến trình QEMU
        debug::exit(debug::EXIT_SUCCESS);
    }
}
[dependencies]
cortex-m = "0.5.7"
cortex-m-rt = "0.6.3"
panic-halt = "0.2.0"
cortex-m-semihosting = "0.3.1"
$ cargo run --release
     Running `qemu-system-arm -cpu cortex-m3 -machine lm3s6965evb (..)
123456789

Nếu bạn chạy trên Discovery board, bạn sẽ thấy output trên console OpenOCD. Ngoài ra, chương trình sẽ không dừng khi count đạt 9.

Default exception handler

Điều mà thuộc tính exception thực sự làm là ghi đè default exception handler cho một exception cụ thể. Nếu bạn không ghi đè handler cho một exception cụ thể, nó sẽ được xử lý bởi hàm DefaultHandler, mặc định là: 

fn DefaultHandler() {
    loop {}
}

Hàm này được crate cortex-m-rt cung cấp và được đánh dấu là #[no_mangle] để bạn có thể đặt breakpoint tại "DefaultHandler" và bắt các exception chưa được xử lý.

Có thể ghi đè DefaultHandler này bằng thuộc tính exception:

#[exception]
fn DefaultHandler(irqn: i16) {
    // custom default handler
}

Tham số irqn cho biết exception nào đang được phục vụ. Giá trị âm cho biết một Cortex-M exception đang được phục vụ; và zero hoặc giá trị dương cho biết một exception dành riêng cho thiết bị, còn gọi là interrupt, đang được phục vụ.

Hard fault handler

Exception HardFault hơi đặc biệt một chút. Exception này được kích hoạt khi chương trình vào trạng thái không hợp lệ nên handler của nó không thể return vì điều đó có thể dẫn đến hành vi không xác định. Ngoài ra, crate runtime thực hiện một số công việc trước khi user định nghĩa handler HardFault được gọi để cải thiện khả năng debug.

Kết quả là, handler HardFault phải có chữ ký sau: fn(&ExceptionFrame) -> !. Tham số của handler là một con trỏ đến các register đã được push vào stack bởi exception. Các register này là snapshot của trạng thái bộ xử lý tại thời điểm exception được kích hoạt và hữu ích để chẩn đoán hard fault.

Đây là ví dụ thực hiện thao tác không hợp lệ: đọc từ một địa chỉ bộ nhớ không tồn tại.

LƯU Ý: Chương trình này sẽ không hoạt động, tức là sẽ không crash, trên QEMU vì qemu-system-arm -machine lm3s6965evb không kiểm tra memory load và sẽ vui vẻ trả về 0 khi đọc từ bộ nhớ không hợp lệ. 

#![no_main]
#![no_std]

use panic_halt as _;

use core::fmt::Write;
use core::ptr;

use cortex_m_rt::{entry, exception, ExceptionFrame};
use cortex_m_semihosting::hio;

#[entry]
fn main() -> ! {
    // đọc từ địa chỉ bộ nhớ không tồn tại
    unsafe {
        ptr::read_volatile(0x3FFF_0000 as *const u32);
    }

    loop {}
}

#[exception]
fn HardFault(ef: &ExceptionFrame) -> ! {
    if let Ok(mut hstdout) = hio::hstdout() {
        writeln!(hstdout, "{:#?}", ef).ok();
    }

    loop {}
}

Handler HardFault in giá trị ExceptionFrame. Nếu bạn chạy đoạn này, bạn sẽ thấy nội dung tương tự trên console OpenOCD. 

$ openocd
(..)
ExceptionFrame {
    r0: 0x3fff0000,
    r1: 0x00000003,
    r2: 0x080032e8,
    r3: 0x00000000,
    r12: 0x00000000,
    lr: 0x080016df,
    pc: 0x080016e2,
    xpsr: 0x61000000,
}

Giá trị pc là giá trị của Program Counter tại thời điểm exception và nó trỏ đến lệnh đã gây ra exception.

Nếu bạn xem disassembly của chương trình:

$ cargo objdump --bin app --release -- -d --no-show-raw-insn --print-imm-hex
(..)
ResetTrampoline:
 8000942:       movw    r0, #0xfffe
 8000946:       movt    r0, #0x3fff
 800094a:       ldr     r0, [r0]
 800094c:       b       #-0x4 <ResetTrampoline+0xa>

Bạn có thể tra cứu giá trị program counter 0x0800094a trong disassembly. Bạn sẽ thấy rằng thao tác load (ldr r0, [r0]) đã gây ra exception. Trường r0 của ExceptionFrame sẽ cho bạn biết giá trị của register r0 là 0x3fff_fffe tại thời điểm đó.
