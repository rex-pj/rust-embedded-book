QEMU

Chạy và debug chương trình embedded mà không cần phần cứng thực

Chúng ta sẽ bắt đầu bằng cách viết một chương trình cho LM3S6965, một vi điều khiển Cortex-M3. Chúng ta chọn mục tiêu này vì nó có thể được giả lập bằng QEMU — bạn không cần phải loay hoay với phần cứng trong phần này và có thể tập trung vào công cụ và quy trình phát triển.

QUAN TRỌNG: Chúng ta sẽ dùng tên app cho tên dự án trong hướng dẫn này. Mỗi khi bạn thấy từ app, hãy thay bằng tên bạn đã chọn cho dự án của mình. Hoặc, bạn cũng có thể đặt tên dự án là app để tránh phải thay.
Quy trình phát triển với QEMU

Tạo chương trình Rust không chuẩn

Chúng ta sẽ dùng template cortex-m-quickstart để tạo một dự án mới. Dự án được tạo ra sẽ chứa một ứng dụng bare-bone: điểm khởi đầu tốt cho một ứng dụng embedded Rust mới. Ngoài ra, dự án còn chứa thư mục examples với nhiều ứng dụng riêng biệt, làm nổi bật một số tính năng key của embedded Rust.

Dùng cargo-generate

Trước tiên cài cargo-generate:

cargo install cargo-generate

Sau đó tạo dự án mới:

cargo generate --git https://github.com/rust-embedded/cortex-m-quickstart

 Project Name: app
 Creating project called `app`...
 Done! New project created /tmp/app
cd app

Dùng git

Clone repository:

git clone https://github.com/rust-embedded/cortex-m-quickstart app
cd app

Sau đó điền vào các placeholder trong file Cargo.toml:

[package]
authors = ["{{authors}}"] # "{{authors}}" -> "John Smith"
edition = "2018"
name = "{{project-name}}" # "{{project-name}}" -> "app"
version = "0.1.0"

# ..

[[bin]]
name = "{{project-name}}" # "{{project-name}}" -> "app"
test = false
bench = false

Không dùng công cụ nào

Tải snapshot mới nhất của template cortex-m-quickstart và giải nén:

curl -LO https://github.com/rust-embedded/cortex-m-quickstart/archive/master.zip
unzip master.zip
mv cortex-m-quickstart-master app
cd app

Sau đó điền vào các placeholder trong Cargo.toml như hướng dẫn ở phần "Dùng git" ở trên.

Tổng quan chương trình

Đây là các phần quan trọng nhất của source code trong src/main.rs:

#![no_std]
#![no_main]

use panic_halt as _;

use cortex_m_rt::entry;

#[entry]
fn main() -> ! {
    loop {
        // code của bạn ở đây
    }
}

Chương trình này khác một chút so với chương trình Rust thông thường, hãy xem kỹ hơn:

#![no_std] cho biết chương trình này sẽ không link vào crate chuẩn std. Thay vào đó nó sẽ link vào tập hợp con của nó: crate core.

#![no_main] cho biết chương trình này sẽ không dùng interface main chuẩn mà hầu hết các chương trình Rust sử dụng. Lý do chính để dùng no_main là việc sử dụng interface main trong ngữ cảnh no_std yêu cầu nightly.

use panic_halt as _; — Crate này cung cấp một panic_handler định nghĩa hành vi khi xảy ra panic. Chúng ta sẽ đề cập chi tiết hơn trong chương Panicking.

#[entry] là một attribute được cung cấp bởi crate cortex-m-rt dùng để đánh dấu điểm vào (entry point) của chương trình. Vì chúng ta không dùng interface main chuẩn nên cần một cách khác để chỉ ra entry point, đó là #[entry].

fn main() -> ! — Chương trình của chúng ta sẽ là process duy nhất chạy trên phần cứng mục tiêu, vì vậy chúng ta không muốn nó kết thúc! Chúng ta dùng một divergent function (phần -> ! trong chữ ký hàm) để đảm bảo điều này tại compile time.

Cross compile

Bước tiếp theo là cross compile chương trình cho kiến trúc Cortex-M3. Điều này đơn giản như chạy cargo build --target $TRIPLE nếu bạn biết compilation target ($TRIPLE) là gì. May mắn thay, file .cargo/config.toml trong template đã có câu trả lời: 

tail -n6 .cargo/config.toml
[build]
# Chọn MỘT trong các compilation target sau
# target = "thumbv6m-none-eabi"    # Cortex-M0 và Cortex-M0+
target = "thumbv7m-none-eabi"    # Cortex-M3
# target = "thumbv7em-none-eabi"   # Cortex-M4 và Cortex-M7 (không FPU)
# target = "thumbv7em-none-eabihf" # Cortex-M4F và Cortex-M7F (có FPU)

Để cross compile cho kiến trúc Cortex-M3, chúng ta phải dùng thumbv7m-none-eabi. Target đó không được cài tự động khi cài Rust toolchain, đây là lúc thích hợp để thêm target vào toolchain nếu bạn chưa làm: 

rustup target add thumbv7m-none-eabi

Vì compilation target thumbv7m-none-eabi đã được đặt làm default trong .cargo/config.toml, hai lệnh dưới đây làm việc giống nhau: 

cargo build --target thumbv7m-none-eabi
cargo build

Kiểm tra binary

Bây giờ chúng ta có một ELF binary không phải native tại target/thumbv7m-none-eabi/debug/app. Chúng ta có thể kiểm tra nó bằng cargo-binutils.

Với cargo-readobj, chúng ta có thể in ELF headers để xác nhận đây là binary ARM:

cargo readobj --bin app -- --file-headers

Chạy trên QEMU

Tiếp theo, hãy xem cách chạy chương trình embedded trên QEMU. Lần này chúng ta sẽ dùng ví dụ hello — thứ thực sự làm gì đó hữu ích.

Đây là source code của examples/hello.rs:

//! In "Hello, world!" ra console của host dùng semihosting

#![no_main]
#![no_std]

use panic_halt as _;

use cortex_m_rt::entry;
use cortex_m_semihosting::{debug, hprintln};

#[entry]
fn main() -> ! {
    hprintln!("Hello, world!");

    // Thoát QEMU
    // CHÚ Ý: đừng chạy lệnh này trên phần cứng thực; nó có thể làm hỏng trạng thái OpenOCD
    debug::exit(debug::EXIT_SUCCESS);

    loop {}
}

Chương trình này dùng thứ gọi là semihosting để in text ra console của host. Khi dùng phần cứng thực, điều này đòi hỏi một debug session, nhưng khi dùng QEMU thì nó hoạt động ngay lập tức.

Hãy bắt đầu bằng cách compile ví dụ:

cargo build --example hello

Binary output sẽ ở tại target/thumbv7m-none-eabi/debug/examples/hello.

Để chạy binary này trên QEMU, chạy lệnh sau:

qemu-system-arm \
  -cpu cortex-m3 \
  -machine lm3s6965evb \
  -nographic \
  -semihosting-config enable=on,target=native \
  -kernel target/thumbv7m-none-eabi/debug/examples/hello
Hello, world!

Lệnh sẽ thoát thành công (exit code = 0) sau khi in text. Trên *nix bạn có thể kiểm tra bằng:

echo $?
0

Hãy phân tích lệnh QEMU đó:

Giải thích các flag của lệnh QEMU

Gõ lệnh QEMU dài như vậy rất mất công! Chúng ta có thể đặt một custom runner để đơn giản hóa quá trình. File .cargo/config.toml có một runner đã được comment out để gọi QEMU — hãy bỏ comment nó: 

[target.thumbv7m-none-eabi]
# bỏ comment dòng này để `cargo run` thực thi chương trình trên QEMU
runner = "qemu-system-arm -cpu cortex-m3 -machine lm3s6965evb -nographic -semihosting-config enable=on,target=native -kernel"

Runner này chỉ áp dụng cho target thumbv7m-none-eabi, là compilation target default của chúng ta. Giờ cargo run sẽ compile chương trình và chạy nó trên QEMU: 

cargo run --example hello --release
   Compiling app v0.1.0 (file:///tmp/app)
    Finished release [optimized + debuginfo] target(s) in 0.26s
     Running `qemu-system-arm -cpu cortex-m3 -machine lm3s6965evb -nographic -semihosting-config enable=on,target=native -kernel target/thumbv7m-none-eabi/release/examples/hello`
Hello, world!

Debug

Debug là phần không thể thiếu trong phát triển embedded. Hãy xem cách thực hiện.

Debug một thiết bị embedded bao gồm remote debugging vì chương trình chúng ta muốn debug sẽ không chạy trên máy đang chạy debugger (GDB hoặc LLDB).

Remote debugging bao gồm một client và một server. Trong setup QEMU, client là một tiến trình GDB (hoặc LLDB) và server là tiến trình QEMU đang chạy chương trình embedded.

Kiến trúc Remote Debugging với QEMU

Bước debug đầu tiên là khởi động QEMU ở chế độ debug:

qemu-system-arm \
  -cpu cortex-m3 \
  -machine lm3s6965evb \
  -nographic \
  -semihosting-config enable=on,target=native \
  -gdb tcp::3333 \
  -S \
  -kernel target/thumbv7m-none-eabi/debug/examples/hello

Lệnh này sẽ không in gì ra console và sẽ block terminal. Chúng ta đã truyền thêm hai flag:

  • -gdb tcp::3333 — Nói với QEMU chờ kết nối GDB trên TCP port 3333.
  • -S — Nói với QEMU đóng băng máy khi khởi động. Không có flag này, chương trình sẽ chạy đến cuối main trước khi chúng ta kịp khởi động debugger!

Tiếp theo, khởi động GDB ở một terminal khác và nói cho nó load debug symbols của ví dụ:

gdb-multiarch -q target/thumbv7m-none-eabi/debug/examples/hello
Bạn có thể cần dùng phiên bản GDB khác thay vì gdb-multiarch tùy vào cái bạn đã cài. Có thể là arm-none-eabi-gdb hoặc đơn giản là gdb.

Trong shell GDB, kết nối tới QEMU đang chờ kết nối trên TCP port 3333:

target remote :3333
Remote debugging using :3333
Reset () at $REGISTRY/cortex-m-rt-0.6.1/src/lib.rs:473
473     pub unsafe extern "C" fn Reset() -> ! {

Bạn sẽ thấy rằng tiến trình đang bị dừng và program counter đang trỏ đến một hàm có tên Reset. Đó là reset handler: thứ mà các lõi Cortex-M thực thi khi boot.

Reset handler này cuối cùng sẽ gọi hàm main của chúng ta. Hãy nhảy thẳng đến đó bằng breakpoint và lệnh continue. Để đặt breakpoint, đầu tiên hãy xem code ở đâu chúng ta muốn dừng, dùng lệnh list: 

list main
6       use panic_halt as _;
7
8       use cortex_m_rt::entry;
9       use cortex_m_semihosting::{debug, hprintln};
10
11      #[entry]
12      fn main() -> ! {
13          hprintln!("Hello, world!").unwrap();
14
15          // exit QEMU

Chúng ta muốn đặt breakpoint ngay trước "Hello, world!" ở dòng 13:

break 13

Hướng dẫn GDB chạy đến hàm main:

continue
Continuing.

Breakpoint 1, hello::__cortex_m_rt_main () at examples\hello.rs:13
13          hprintln!("Hello, world!").unwrap();

Chúng ta đang ở gần code in "Hello, world!". Tiếp tục dùng lệnh next:

next
16          debug::exit(debug::EXIT_SUCCESS);

Ở điểm này bạn sẽ thấy "Hello, world!" được in ra trên terminal đang chạy qemu-system-arm:

$ qemu-system-arm (..)
Hello, world!

Gọi next thêm một lần sẽ kết thúc tiến trình QEMU:

next
[Inferior 1 (Remote target) exited normally]

Bây giờ bạn có thể thoát GDB session:

quit
Sau khi thành thạo các bước trên, bạn có thể tạo file .gdbinit để tự động hóa các bước kết nối và thiết lập breakpoint, giúp workflow debug nhanh hơn.
