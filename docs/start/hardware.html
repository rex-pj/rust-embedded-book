Phần cứng thực — Hardware

Đến đây bạn đã quen phần nào với tooling và quy trình phát triển. Trong phần này chúng ta sẽ chuyển sang phần cứng thực; quy trình về cơ bản vẫn như cũ. Hãy bắt đầu.

Tìm hiểu phần cứng của bạn

Trước khi bắt đầu, bạn cần xác định một số đặc điểm của thiết bị target vì chúng sẽ được dùng để cấu hình project:

  • ARM core. Ví dụ: Cortex-M3.
  • ARM core có FPU không? Cortex-M4F và Cortex-M7F thì có.
  • Target device có bao nhiêu Flash memory và RAM? Ví dụ: 256 KiB Flash và 32 KiB RAM.
  • Flash memory và RAM được ánh xạ vào address space ở đâu? Ví dụ: RAM thường nằm ở địa chỉ 0x2000_0000.

Bạn có thể tìm thấy thông tin này trong data sheet hoặc reference manual của thiết bị.

Trong phần này chúng ta sẽ dùng phần cứng tham chiếu của mình là STM32F3DISCOVERY. Board này chứa vi điều khiển STM32F303VCT6. Vi điều khiển này có:

  • Một Cortex-M4F core với FPU độ chính xác đơn
  • 256 KiB Flash ở địa chỉ 0x0800_0000
  • 40 KiB RAM ở địa chỉ 0x2000_0000. (Còn một vùng RAM khác nhưng để đơn giản chúng ta bỏ qua)
STM32F3DISCOVERY: Bộ nhớ và kết nối debug

Cấu hình

Chúng ta sẽ bắt đầu từ đầu với một template instance mới. Tham khảo phần trước về QEMU để ôn lại cách làm này mà không dùng cargo-generate. 

$ cargo generate --git https://github.com/rust-embedded/cortex-m-quickstart
 Project Name: app
 Creating project called `app`...
 Done! New project created /tmp/app

$ cd app

Bước đầu tiên là đặt default compilation target trong .cargo/config.toml. 

tail -n5 .cargo/config.toml
# Pick ONE of these compilation targets
# target = "thumbv6m-none-eabi"    # Cortex-M0 and Cortex-M0+
# target = "thumbv7m-none-eabi"    # Cortex-M3
# target = "thumbv7em-none-eabi"   # Cortex-M4 and Cortex-M7 (no FPU)
target = "thumbv7em-none-eabihf" # Cortex-M4F and Cortex-M7F (with FPU)

Chúng ta sẽ dùng thumbv7em-none-eabihf vì nó áp dụng cho Cortex-M4F core.

LƯU Ý: Như bạn có thể nhớ từ chương trước, chúng ta cần install tất cả target và đây là target mới. Vì vậy đừng quên chạy lệnh install: rustup target add thumbv7em-none-eabihf.

Bước thứ hai là nhập thông tin memory region vào file memory.x.

$ cat memory.x
/* Linker script for the STM32F303VCT6 */
MEMORY
{
  /* NOTE 1 K = 1 KiBi = 1024 bytes */
  FLASH : ORIGIN = 0x08000000, LENGTH = 256K
  RAM : ORIGIN = 0x20000000, LENGTH = 40K
}

LƯU Ý: Nếu vì lý do nào đó bạn thay đổi file memory.x sau khi đã build lần đầu cho một build target cụ thể, hãy chạy cargo clean trước cargo build, vì cargo build có thể không theo dõi các cập nhật của memory.x.

Chúng ta sẽ bắt đầu lại với ví dụ hello, nhưng trước tiên cần thực hiện một thay đổi nhỏ.

Trong examples/hello.rs, hãy đảm bảo lệnh gọi debug::exit() được comment out hoặc xóa đi. Nó chỉ được dùng khi chạy trên QEMU. 

#[entry]
fn main() -> ! {
    hprintln!("Hello, world!").unwrap();

    // exit QEMU
    // NOTE do not run this on hardware; it can corrupt OpenOCD state
    // debug::exit(debug::EXIT_SUCCESS);

    loop {}
}

Bây giờ bạn có thể cross compile chương trình bằng cargo build và kiểm tra binary bằng cargo-binutils như trước. Crate cortex-m-rt xử lý tất cả các "ma thuật" cần thiết để khởi động chip, vì hầu hết Cortex-M CPU đều boot theo cùng một cách. 

cargo build --example hello

Debug

Debug sẽ trông hơi khác một chút. Thực ra, các bước đầu có thể khác nhau tùy thuộc vào thiết bị target. Trong phần này chúng ta sẽ trình bày các bước cần thiết để debug một chương trình chạy trên STM32F3DISCOVERY. Đây là tài liệu tham khảo; để biết thông tin debug cụ thể theo thiết bị, xem Debugonomicon.

Như trước, chúng ta sẽ dùng remote debugging và client sẽ là một tiến trình GDB. Tuy nhiên lần này, server sẽ là OpenOCD.

Như đã làm trong phần verify, kết nối discovery board vào laptop / PC và kiểm tra header ST-LINK đã được lắp.

Trên một terminal, chạy openocd để kết nối với ST-LINK trên discovery board. Chạy lệnh này từ thư mục root của template; openocd sẽ dùng file openocd.cfg để biết interface file và target file nào cần dùng. 

cat openocd.cfg
# Sample OpenOCD configuration for the STM32F3DISCOVERY development board

# Depending on the hardware revision you got you'll have to pick ONE of these
# interfaces. At any time only one interface should be commented out.

# Revision C (newer revision)
source [find interface/stlink.cfg]

# Revision A and B (older revisions)
# source [find interface/stlink-v2.cfg]

source [find target/stm32f3x.cfg]

LƯU Ý: Nếu bạn phát hiện ra mình có phiên bản cũ hơn của discovery board trong phần verify thì bạn nên sửa file openocd.cfg tại thời điểm này để dùng interface/stlink-v2.cfg.

Trên một terminal khác, chạy GDB, cũng từ thư mục root của template. 

gdb-multiarch -q target/thumbv7em-none-eabihf/debug/examples/hello

LƯU Ý: như trước, bạn có thể cần phiên bản GDB khác thay vì gdb-multiarch tùy thuộc vào phiên bản đã cài trong chương installation. Có thể là arm-none-eabi-gdb hoặc đơn giản là gdb.

Tiếp theo, kết nối GDB với OpenOCD đang chờ kết nối TCP trên port 3333. 

(gdb) target remote :3333
Remote debugging using :3333
0x00000000 in ?? ()

Bây giờ tiến hành flash (load) chương trình lên vi điều khiển bằng lệnh load. 

(gdb) load
Loading section .vector_table, size 0x400 lma 0x8000000
Loading section .text, size 0x1518 lma 0x8000400
Loading section .rodata, size 0x414 lma 0x8001918
Start address 0x08000400, load size 7468
Transfer rate: 13 KB/sec, 2489 bytes/write.

Chương trình đã được load. Chương trình này dùng semihosting nên trước khi thực hiện bất kỳ semihosting call nào, chúng ta phải báo cho OpenOCD enable semihosting. Bạn có thể gửi lệnh đến OpenOCD bằng lệnh monitor. 

(gdb) monitor arm semihosting enable
semihosting is enabled

Bạn có thể xem tất cả lệnh OpenOCD bằng cách gọi lệnh monitor help.

Như trước, chúng ta có thể nhảy thẳng đến main bằng một breakpoint và lệnh continue. 

(gdb) break main
Breakpoint 1 at 0x8000490: file examples/hello.rs, line 11.
Note: automatically using hardware breakpoints for read-only addresses.

(gdb) continue
Continuing.

Breakpoint 1, hello::__cortex_m_rt_main_trampoline () at examples/hello.rs:11
11      #[entry]

LƯU Ý: Nếu GDB block terminal thay vì dừng tại breakpoint sau khi bạn gõ lệnh continue, bạn nên kiểm tra lại thông tin memory region trong file memory.x đã được cài đặt đúng cho thiết bị của bạn (cả địa chỉ đầu và độ dài).

Bước vào hàm main bằng step.

(gdb) step
halted: PC: 0x08000496
hello::__cortex_m_rt_main () at examples/hello.rs:13
13          hprintln!("Hello, world!").unwrap();

Sau khi tiến chương trình bằng next, bạn sẽ thấy "Hello, world!" được in ra console OpenOCD, cùng với một số nội dung khác. 

$ openocd
(..)

Thông điệp chỉ hiển thị một lần vì chương trình sắp vào vòng lặp vô hạn được định nghĩa tại dòng 19: loop {}

Bây giờ bạn có thể thoát GDB bằng lệnh quit.

(gdb) quit
A debugging session is active.

        Inferior 1 [Remote target] will be detached.

Quit anyway? (y or n)

Debug giờ đây cần thêm vài bước nên chúng ta đã đóng gói tất cả các bước đó vào một GDB script tên là openocd.gdb. File này được tạo trong bước cargo generate và sẽ hoạt động mà không cần sửa đổi gì. Hãy xem qua: 

cat openocd.gdb
target extended-remote :3333

# print demangled symbols
set print asm-demangle on

# detect unhandled exceptions, hard faults and panics
break DefaultHandler
break HardFault
break rust_begin_unwind

monitor arm semihosting enable

load

# start the process but immediately halt the processor
stepi

Bây giờ chạy <gdb> -x openocd.gdb target/thumbv7em-none-eabihf/debug/examples/hello sẽ ngay lập tức kết nối GDB với OpenOCD, enable semihosting, load chương trình và bắt đầu tiến trình.

Ngoài ra, bạn có thể biến <gdb> -x openocd.gdb thành một custom runner để cargo run build chương trình và bắt đầu GDB session. Runner này được bao gồm trong .cargo/config.toml nhưng bị comment out. 

head -n10 .cargo/config.toml
[target.thumbv7m-none-eabi]
# uncomment this to make `cargo run` execute programs on QEMU
# runner = "qemu-system-arm -cpu cortex-m3 -machine lm3s6965evb -nographic -semihosting-config enable=on,target=native -kernel"

[target.'cfg(all(target_arch = "arm", target_os = "none"))']
# uncomment ONE of these three option to make `cargo run` start a GDB session
# which option to pick depends on your system
runner = "arm-none-eabi-gdb -x openocd.gdb"
# runner = "gdb-multiarch -x openocd.gdb"
# runner = "gdb -x openocd.gdb"
$ cargo run --example hello
