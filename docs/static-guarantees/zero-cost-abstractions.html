Zero Cost Abstractions

Type state cũng là một ví dụ xuất sắc về Zero Cost Abstractions — khả năng chuyển một số hành vi sang thực thi hoặc phân tích tại compile time. Các type state này không chứa dữ liệu thực sự, mà được dùng như các marker. Vì chúng không chứa dữ liệu, chúng không có biểu diễn thực sự trong bộ nhớ tại runtime: 

use core::mem::size_of;

let _ = size_of::<Enabled>();    // == 0
let _ = size_of::<Input>();      // == 0
let _ = size_of::<PulledHigh>(); // == 0
let _ = size_of::<GpioConfig<Enabled, Input, PulledHigh>>(); // == 0

Zero Sized Types

struct Enabled;

Các struct được định nghĩa như thế này được gọi là Zero Sized Types, vì chúng không chứa dữ liệu thực sự. Mặc dù các kiểu này hoạt động "thực" tại compile time — bạn có thể copy chúng, move chúng, lấy reference đến chúng, v.v. — tuy nhiên optimizer sẽ loại bỏ chúng hoàn toàn.

Trong đoạn code này:

pub fn into_input_high_z(self) -> GpioConfig<Enabled, Input, HighZ> {
    self.periph.modify(|_r, w| w.input_mode().high_z());
    GpioConfig {
        periph: self.periph,
        enabled: Enabled,
        direction: Input,
        mode: HighZ,
    }
}

GpioConfig mà chúng ta trả về không bao giờ tồn tại tại runtime. Gọi hàm này thường sẽ được rút gọn thành một lệnh assembly duy nhất — lưu một giá trị thanh ghi hằng vào một vị trí thanh ghi. Điều này có nghĩa là interface typestate chúng ta đã xây dựng là một zero cost abstraction — nó không dùng thêm CPU, RAM, hay code space để theo dõi trạng thái của GpioConfig, và tạo ra machine code giống như truy cập thanh ghi trực tiếp.

Lồng nhau (Nesting)

Nhìn chung, các abstraction này có thể được lồng nhau sâu tùy ý. Miễn là tất cả các thành phần được dùng là zero sized type, toàn bộ cấu trúc sẽ không tồn tại tại runtime.

Đối với các cấu trúc phức tạp hoặc lồng nhau sâu, việc định nghĩa tất cả các tổ hợp trạng thái có thể sẽ tẻ nhạt. Trong những trường hợp như vậy, có thể dùng macro để sinh ra tất cả các implementation.
