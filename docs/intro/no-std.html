Môi trường Rust no_std

Thuật ngữ Embedded Programming (Lập trình nhúng) được dùng cho một loạt các lớp lập trình khác nhau. Từ lập trình cho MCU 8-bit (như ST72325xx) với chỉ vài KB RAM và ROM, cho đến các hệ thống như Raspberry Pi (Model B 3+) có CPU 32/64-bit 4-core Cortex-A53 @ 1.4 GHz và 1GB RAM. Các ràng buộc/giới hạn khác nhau sẽ áp dụng khi viết code tùy thuộc vào loại mục tiêu và trường hợp sử dụng mà bạn có.

Có hai phân loại Embedded Programming tổng quát:

Môi trường Hosted

Các loại môi trường này gần giống với môi trường PC thông thường. Điều này có nghĩa là bạn được cung cấp một System Interface (Giao diện Hệ thống) [ví dụ: POSIX] cung cấp cho bạn các primitive để tương tác với các hệ thống khác nhau, như hệ thống file, networking, quản lý bộ nhớ, thread, v.v. Các thư viện chuẩn lần lượt thường phụ thuộc vào các primitive này để implement chức năng của chúng. Bạn cũng có thể có một số loại sysroot và các hạn chế về sử dụng RAM/ROM, và có thể một số HW hoặc I/O đặc biệt. Nhìn chung, cảm giác như đang code trên môi trường PC có mục đích đặc biệt.

Môi trường Bare Metal

Trong môi trường bare metal, không có code nào được load trước chương trình của bạn. Không có phần mềm được cung cấp bởi OS, chúng ta không thể load thư viện chuẩn. Thay vào đó, chương trình, cùng với các crate nó sử dụng, chỉ có thể dùng phần cứng (bare metal) để chạy. Để ngăn Rust load thư viện chuẩn, hãy dùng no_std. Các phần platform-agnostic của thư viện chuẩn có thể được truy cập thông qua libcore. libcore cũng loại trừ những thứ không phải lúc nào cũng mong muốn trong môi trường embedded. Một trong số đó là memory allocator cho dynamic memory allocation. Nếu bạn cần điều này hay bất kỳ chức năng nào khác, thường có các crate cung cấp chúng.

std vs no_std — Phân cấp thư viện Rust

libstd Runtime

Như đã đề cập ở trên, việc dùng libstd yêu cầu một số loại tích hợp hệ thống, nhưng điều này không chỉ vì libstd chỉ cung cấp một cách chung để truy cập các OS abstraction — nó còn cung cấp một runtime. Runtime này, trong số những việc khác, lo việc thiết lập stack overflow protection, xử lý các command line argument, và spawn main thread trước khi hàm main của chương trình được gọi. Runtime này cũng sẽ không khả dụng trong môi trường no_std.

Tóm tắt

#![no_std] là một crate-level attribute cho biết crate sẽ link đến core-crate thay vì std-crate. Crate libcore là một tập con platform-agnostic của crate std, không đưa ra bất kỳ giả định nào về hệ thống mà chương trình sẽ chạy trên đó. Vì vậy, nó cung cấp các API cho các language primitive như float, string và slice, cũng như các API expose các tính năng của processor như atomic operation và SIMD instruction. Tuy nhiên nó thiếu các API cho bất cứ thứ gì liên quan đến platform integration. Nhờ các thuộc tính này, code no_std và libcore có thể được dùng cho bất kỳ loại bootstrapping (stage 0) code nào như bootloader, firmware hoặc kernel.

Tổng quan

Tính năng no_std std
heap (dynamic memory) * ✓
collections (Vec, BTreeMap, v.v.) ** ✓
stack overflow protection ✘ ✓
chạy init code trước main ✘ ✓
libstd có sẵn ✘ ✓
libcore có sẵn ✓ ✓
viết firmware, kernel, hoặc bootloader code ✓ ✘

* Chỉ khi bạn dùng crate alloc và dùng một allocator phù hợp như alloc-cortex-m.

** Chỉ khi bạn dùng crate collections và cấu hình một global default allocator.

** HashMap và HashSet không có sẵn do thiếu secure random number generator.
