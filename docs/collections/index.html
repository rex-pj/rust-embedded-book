Collections — Bộ sưu tập dữ liệu

Cuối cùng bạn sẽ muốn sử dụng các cấu trúc dữ liệu động (hay còn gọi là collections) trong chương trình của mình. std cung cấp một tập hợp các collection thông dụng: Vec, String, HashMap, v.v. Tất cả các collection được implement trong std đều sử dụng một global dynamic memory allocator (hay còn gọi là heap).

Vì core theo định nghĩa không có cấp phát bộ nhớ, các implementation này không có sẵn ở đó, nhưng chúng có thể được tìm thấy trong crate alloc đi kèm với compiler.

Nếu bạn cần collections, implementation được cấp phát trên heap không phải lựa chọn duy nhất của bạn. Bạn cũng có thể sử dụng các collection có dung lượng cố định; một implementation như vậy có thể được tìm thấy trong crate heapless.

Trong phần này, chúng ta sẽ khám phá và so sánh hai implementation này.

alloc (heap) vs heapless (dung lượng cố định) — Hai cách tiếp cận

Dùng alloc

Crate alloc đi kèm với bản phân phối Rust chuẩn. Để import crate, bạn có thể use trực tiếp mà không cần khai báo nó như một dependency trong file Cargo.toml. 

#![feature(alloc)]

extern crate alloc;

use alloc::vec::Vec;

Để có thể sử dụng bất kỳ collection nào, trước tiên bạn sẽ cần dùng attribute global_allocator để khai báo global allocator mà chương trình sẽ sử dụng. Yêu cầu là allocator bạn chọn phải implement trait GlobalAlloc.

Để đầy đủ và giữ phần này khép kín nhất có thể, chúng ta sẽ implement một bump pointer allocator đơn giản và sử dụng nó làm global allocator. Tuy nhiên, chúng tôi khuyến khích mạnh mẽ bạn hãy sử dụng một allocator đã được kiểm chứng kỹ lưỡng từ crates.io trong chương trình của bạn thay vì allocator này. 

// Triển khai bump pointer allocator

use core::alloc::{GlobalAlloc, Layout};
use core::cell::UnsafeCell;
use core::ptr;

use cortex_m::interrupt;

// Bump pointer allocator cho hệ thống *đơn nhân*
struct BumpPointerAlloc {
    head: UnsafeCell<usize>,
    end: usize,
}

unsafe impl Sync for BumpPointerAlloc {}

unsafe impl GlobalAlloc for BumpPointerAlloc {
    unsafe fn alloc(&self, layout: Layout) -> *mut u8 {
        // `interrupt::free` là critical section giúp allocator
        // của chúng ta an toàn khi dùng từ bên trong interrupts
        interrupt::free(|_| {
            let head = self.head.get();
            let size = layout.size();
            let align = layout.align();
            let align_mask = !(align - 1);

            // dịch start lên đến ranh giới căn chỉnh tiếp theo
            let start = (*head + align - 1) & align_mask;

            if start + size > self.end {
                // null pointer báo hiệu điều kiện Out Of Memory
                ptr::null_mut()
            } else {
                *head = start + size;
                start as *mut u8
            }
        })
    }

    unsafe fn dealloc(&self, _: *mut u8, _: Layout) {
        // allocator này không bao giờ giải phóng bộ nhớ
    }
}

// Khai báo global memory allocator
// LƯU Ý người dùng phải đảm bảo vùng nhớ `[0x2000_0100, 0x2000_0200]`
// không được sử dụng bởi các phần khác của chương trình
#[global_allocator]
static HEAP: BumpPointerAlloc = BumpPointerAlloc {
    head: UnsafeCell::new(0x2000_0100),
    end: 0x2000_0200,
};

Ngoài việc chọn global allocator, người dùng cũng phải định nghĩa cách xử lý lỗi Out Of Memory (OOM) bằng attribute unstable alloc_error_handler. 

#![feature(alloc_error_handler)]

use cortex_m::asm;

#[alloc_error_handler]
fn on_oom(_layout: Layout) -> ! {
    asm::bkpt();

    loop {}
}

Khi tất cả đã sẵn sàng, người dùng cuối cùng có thể sử dụng các collection trong alloc.

#[entry]
fn main() -> ! {
    let mut xs = Vec::new();

    xs.push(42);
    assert!(xs.pop(), Some(42));

    loop {
        // ..
    }
}

Nếu bạn đã dùng các collection trong crate std thì chúng sẽ quen thuộc vì đây là cùng một implementation.

Dùng heapless

heapless không cần thiết lập vì các collection của nó không phụ thuộc vào global memory allocator. Chỉ cần use các collection của nó và tiến hành khởi tạo: 

// heapless phiên bản: v0.4.x
use heapless::Vec;
use heapless::consts::*;

#[entry]
fn main() -> ! {
    let mut xs: Vec<_, U8> = Vec::new();

    xs.push(42).unwrap();
    assert_eq!(xs.pop(), Some(42));
    loop {}
}

Bạn sẽ nhận thấy hai sự khác biệt giữa các collection này và các collection trong alloc.

Thứ nhất, bạn phải khai báo trước dung lượng của collection. Các collection heapless không bao giờ cấp phát lại và có dung lượng cố định; dung lượng này là một phần của type signature của collection. Trong trường hợp này, chúng ta đã khai báo rằng xs có dung lượng 8 phần tử, tức là vector có thể chứa tối đa 8 phần tử. Điều này được biểu thị bằng U8 (xem typenum) trong type signature.

Thứ hai, phương thức push và nhiều phương thức khác trả về một Result. Vì các collection heapless có dung lượng cố định, tất cả các thao tác chèn phần tử vào collection đều có thể thất bại. API phản ánh vấn đề này bằng cách trả về Result cho biết thao tác có thành công hay không. Ngược lại, các collection alloc sẽ tự cấp phát lại trên heap để tăng dung lượng.

Từ phiên bản v0.4.x, tất cả các collection heapless lưu trữ tất cả các phần tử của chúng inline. Điều này có nghĩa là một thao tác như let x = heapless::Vec::new(); sẽ cấp phát collection trên stack, nhưng cũng có thể cấp phát collection trên biến static, hoặc thậm chí trên heap (Box<Vec<_, _>>).

Đánh đổi — Trade-offs

Hãy ghi nhớ những điều này khi chọn giữa các collection được cấp phát trên heap, có thể di chuyển, và các collection dung lượng cố định.

Out Of Memory và xử lý lỗi

Với cấp phát heap, Out Of Memory luôn là một khả năng và có thể xảy ra ở bất kỳ nơi nào mà một collection có thể cần tăng kích thước: ví dụ, tất cả các lời gọi alloc::Vec.push đều có thể tạo ra điều kiện OOM. Do đó một số thao tác có thể thất bại ngầm. Một số collection alloc cung cấp các phương thức try_reserve cho phép bạn kiểm tra các điều kiện OOM tiềm năng khi tăng kích thước collection, nhưng bạn cần chủ động sử dụng chúng.

Nếu bạn chỉ dùng các collection heapless và không dùng memory allocator cho bất cứ điều gì khác thì điều kiện OOM là không thể xảy ra. Thay vào đó, bạn sẽ phải xử lý trường hợp collections bị hết dung lượng theo từng trường hợp cụ thể. Tức là bạn phải xử lý tất cả các Result được trả về bởi các phương thức như Vec.push.

Lỗi OOM có thể khó debug hơn so với việc unwrap tất cả các Result được trả về bởi heapless::Vec.push, vì vị trí quan sát được của lỗi có thể không trùng với vị trí gây ra vấn đề. Ví dụ, ngay cả vec.reserve(1) cũng có thể kích hoạt OOM nếu allocator gần cạn kiệt vì một collection khác đang bị rò rỉ bộ nhớ (rò rỉ bộ nhớ là có thể xảy ra trong safe Rust).

Sử dụng bộ nhớ

Phân tích sử dụng bộ nhớ của các collection được cấp phát heap khó khăn vì dung lượng của các collection tồn tại lâu dài có thể thay đổi lúc runtime. Một số thao tác có thể ngầm cấp phát lại collection làm tăng mức sử dụng bộ nhớ, và một số collection cung cấp các phương thức như shrink_to_fit có thể giảm bộ nhớ được sử dụng bởi collection — cuối cùng, tùy thuộc vào allocator để quyết định có thực sự thu nhỏ phần cấp phát bộ nhớ hay không. Ngoài ra, allocator có thể phải xử lý phân mảnh bộ nhớ (memory fragmentation) có thể làm tăng biểu kiến mức sử dụng bộ nhớ.

Mặt khác, nếu bạn chỉ dùng các collection dung lượng cố định, lưu trữ hầu hết chúng trong các biến static và đặt kích thước tối đa cho call stack thì linker sẽ phát hiện nếu bạn cố dùng nhiều bộ nhớ hơn mức thực tế có sẵn.

Hơn nữa, các collection dung lượng cố định được cấp phát trên stack sẽ được báo cáo bởi flag -Z emit-stack-sizes, có nghĩa là các công cụ phân tích stack usage (như stack-sizes) sẽ bao gồm chúng trong phân tích của mình.

Tuy nhiên, các collection dung lượng cố định không thể co lại, điều này có thể dẫn đến load factor thấp hơn (tỷ lệ giữa kích thước collection và dung lượng của nó) so với những gì các collection có thể di chuyển đạt được.

Worst Case Execution Time (WCET)

Nếu bạn đang xây dựng các ứng dụng nhạy cảm về thời gian hoặc ứng dụng real-time cứng, bạn sẽ quan tâm, có thể rất nhiều, đến worst case execution time của các phần khác nhau trong chương trình.

Các collection alloc có thể cấp phát lại, vì vậy WCET của các thao tác có thể tăng kích thước collection cũng sẽ bao gồm thời gian để cấp phát lại collection, điều này phụ thuộc vào dung lượng lúc runtime của collection. Điều này khiến việc xác định WCET của ví dụ thao tác alloc::Vec.push trở nên khó khăn vì nó phụ thuộc vào cả allocator đang được sử dụng và dung lượng lúc runtime của nó.

Mặt khác, các collection dung lượng cố định không bao giờ cấp phát lại, vì vậy tất cả các thao tác đều có thời gian thực thi có thể dự đoán được. Ví dụ, heapless::Vec.push thực thi trong thời gian hằng số.

Dễ sử dụng

alloc yêu cầu thiết lập global allocator còn heapless thì không. Tuy nhiên, heapless yêu cầu bạn chọn dung lượng của mỗi collection mà bạn khởi tạo.

API của alloc sẽ quen thuộc với hầu như mọi Rust developer. API của heapless cố gắng bắt chước API của alloc nhưng sẽ không bao giờ hoàn toàn giống nhau do phải xử lý lỗi tường minh — một số developer có thể cảm thấy việc xử lý lỗi tường minh là quá nhiều hoặc quá cồng kềnh.
