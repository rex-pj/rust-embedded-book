Registers

Chương này là một phần kỹ thuật chuyên sâu. Bạn có thể bỏ qua nó bây giờ và quay lại sau nếu muốn. Tuy nhiên, có rất nhiều nội dung hay ở đây, vì vậy tôi khuyến khích bạn nên đọc.

Đã đến lúc khám phá xem việc gọi display_pins.row1.set_high() hay button_a_pin.is_high() thực sự làm gì bên dưới.

Nói ngắn gọn, gọi display_pins.row1.set_high() chỉ đơn giản là ghi vào một số vùng bộ nhớ đặc biệt. Hãy vào thư mục 09-registers và chạy code khởi đầu từng câu lệnh một (src/main.rs).

Địa chỉ kỳ lạ này là gì?

Địa chỉ 0x50000504 trỏ đến một register. Một register là một vùng bộ nhớ đặc biệt điều khiển một peripheral. Peripheral là một linh kiện điện tử nằm ngay cạnh processor trong package microcontroller và cung cấp cho processor các tính năng bổ sung. Xét cho cùng, processor, khi hoạt động độc lập, chỉ có thể thực hiện toán học và logic.

Register cụ thể này điều khiển các pin GPIO (GPIO là một peripheral) và có thể dùng để kéo từng pin đó xuống thấp (low) hoặc lên cao (high).

(Trên nRF52833 có hơn 32 GPIO, nhưng CPU là 32-bit. Vì vậy, các GPIO pin được tổ chức thành hai nhóm "P0" và "P1", mỗi nhóm có một bộ register để đọc, ghi và cấu hình. Địa chỉ trên là địa chỉ của output register cho các pin P0.)

Đây là đoạn code demo thao tác register trực tiếp:

rust
#![no_main]
#![no_std]

#[allow(unused_imports)]
use registers::entry;

#[entry]
fn main() -> ! {
    registers::init();

    unsafe {
        // Một địa chỉ "magic"!
        const PORT_P0_OUT: u32 = 0x50000504;

        // Bật hàng trên cùng
        *(PORT_P0_OUT as *mut u32) |= 1 << 21;

        // Bật hàng dưới cùng
        *(PORT_P0_OUT as *mut u32) |= 1 << 19;

        // Tắt hàng trên cùng
        *(PORT_P0_OUT as *mut u32) &= !(1 << 21);

        // Tắt hàng dưới cùng
        *(PORT_P0_OUT as *mut u32) &= !(1 << 19);
    }

    loop {
        core::hint::spin_loop();
    }
}

Được rồi. Nhưng làm thế nào để biết register này làm gì? Đã đến lúc RTRM (Read the Reference Manual)!

An aside: LEDs, digital outputs and voltage levels

Một pin là một tiếp điểm điện. Microcontroller của chúng ta có nhiều pin và một số trong đó được kết nối với các LED (Light Emitting Diode). Một LED sẽ phát sáng khi được cấp điện áp. Như tên gọi, LED cũng hoạt động như một "diode". Diode chỉ cho phép điện chạy theo một chiều. Kết nối LED đúng chiều thì có ánh sáng. Kết nối ngược thì không có gì xảy ra.

May mắn cho chúng ta, các pin của microcontroller được kết nối sao cho chúng ta có thể điều khiển LED đúng chiều. Tất cả những gì chúng ta phải làm là áp đủ điện áp qua các pin để bật LED. Các pin gắn với LED thường được cấu hình là digital outputs và có thể xuất hai mức điện áp khác nhau: "thấp" (low), 0 Volt, hoặc "cao" (high), 3 Volt. Mức "cao" (điện áp) sẽ bật LED còn mức "thấp" sẽ tắt LED.

Các trạng thái "thấp" và "cao" này ánh xạ trực tiếp vào khái niệm digital logic. "thấp" là 0 hoặc false và "cao" là 1 hoặc true. Đó là lý do tại sao cấu hình pin này được gọi là digital output.

Ngược lại với digital output là digital input. Giống như digital output có thể là 0 hoặc 1, digital input cũng có thể là 0 hoặc 1. Điểm khác biệt là digital output có thể tạo ra điện áp, còn digital input thì đọc điện áp. Khi microcontroller đọc mức điện áp cao hơn ngưỡng cao, nó sẽ hiểu là 1; khi đọc mức điện áp thấp hơn ngưỡng thấp, nó sẽ hiểu là 0.

RTRM — Đọc Reference Manual

Để tìm hiểu địa chỉ register và ý nghĩa từng bit, cần tra nRF52833 Product Specification. Peripheral GPIO P0 có base address là 0x5000_0000. Register OUT nằm ở offset 0x504, tức địa chỉ đầy đủ là 0x50000504.

Register này 32-bit, mỗi bit tương ứng một pin: đặt bit lên 1 → pin xuất điện áp cao; đặt về 0 → pin xuất mức thấp. Giá trị reset là 0 (tất cả output tắt).

Theo pinmap của MB2: ROW1 (hàng trên cùng) là P0.21, ROW5 (hàng dưới cùng) là P0.19.

Khi chạy chương trình trên và quan sát qua GDB, giá trị register thay đổi như sau:

(mis)Optimization — Vấn đề với compiler tối ưu hóa

Khi biên dịch với --release, LLVM có thể tối ưu hóa bốn lần ghi vào register thành một lần duy nhất — vì nó không biết rằng địa chỉ đó có tác dụng phụ trên phần cứng. Kết quả là LED sẽ không nhấp nháy như mong đợi.

Ta có thể xem assembly để kiểm chứng. Dùng lệnh sau để xuất assembly:

# Build debug
cargo objdump -- --disassemble --no-show-raw-insn --source > debug.dump
# Build release
cargo objdump --release -- --disassemble --no-show-raw-insn --source > release.dump

Giải pháp: dùng volatile operations qua core::ptr. Đây là chương trình hoàn chỉnh (examples/volatile.rs):

rust
#![no_main]
#![no_std]

use core::ptr;

#[allow(unused_imports)]
use registers::entry;

#[entry]
fn main() -> ! {
    registers::init();

    unsafe {
        const PORT_P0_OUT: u32 = 0x50000504;

        // Bật hàng trên cùng
        let out = ptr::read_volatile(PORT_P0_OUT as *mut u32);
        ptr::write_volatile(PORT_P0_OUT as *mut u32, out | 1 << 21);

        // Bật hàng dưới cùng
        let out = ptr::read_volatile(PORT_P0_OUT as *mut u32);
        ptr::write_volatile(PORT_P0_OUT as *mut u32, out | 1 << 19);

        // Tắt hàng trên cùng
        let out = ptr::read_volatile(PORT_P0_OUT as *mut u32);
        ptr::write_volatile(PORT_P0_OUT as *mut u32, out & !(1 << 21));

        // Tắt hàng dưới cùng
        let out = ptr::read_volatile(PORT_P0_OUT as *mut u32);
        ptr::write_volatile(PORT_P0_OUT as *mut u32, out & !(1 << 19));
    }

    loop {
        core::hint::spin_loop();
    }
}

Volatile read/write báo cho compiler: "Các thao tác này có tác dụng phụ — đừng tối ưu hóa chúng đi." Build release với volatile giữ đủ 4 cặp load-modify-store như thiết kế.

0xBAAAAAAD — Địa chỉ không hợp lệ

Điều gì xảy ra nếu bạn đọc từ một địa chỉ không hợp lệ, ví dụ 0x5000_A784 (không tương ứng với peripheral register nào)?

rust
#![no_main]
#![no_std]

use core::ptr;

#[allow(unused_imports)]
use registers::entry;

#[entry]
fn main() -> ! {
    registers::init();

    unsafe {
        ptr::read_volatile(0x5000_A784 as *const u32);
    }

    loop {
        core::hint::spin_loop();
    }
}

Processor không thể hoàn thành thao tác đó và ném ra một hardware exception (HardFault). Debugger dừng lại tại handler HardFault_. GDB hiện thị ExceptionFrame với toàn bộ trạng thái register khi exception xảy ra:

r0: 0x5000a784,
r1: 0x3,
r2: 0x2001ff24,
r3: 0x0,
r12: 0x1,
lr: 0x4403,
pc: 0x43ea,
xpsr: 0x1000000

pc (0x43ea) trỏ tới instruction gây ra lỗi, r0 chứa địa chỉ không hợp lệ (0x5000a784). Disassembly tại pc cho thấy instruction đó là:

ldr r0, [r0, #0]

Đây là instruction load — đọc bộ nhớ tại địa chỉ trong r0. Vì địa chỉ đó không tồn tại, processor ném HardFault. Đây là cách Rust embedded xử lý memory access violations — không âm thầm bỏ qua mà crash rõ ràng với thông tin đầy đủ.

Spooky action at a distance

OUT không phải là register duy nhất điều khiển các pin của Port P0. Còn có hai register nữa: OUTSET và OUTCLR.

Đoạn code sau (examples/spooky.rs) in ra giá trị của P0.OUT sau mỗi thao tác, nhưng thay vì ghi thẳng vào OUT, nó ghi vào OUTSET và OUTCLR:

rust
#![no_main]
#![no_std]

use core::ptr;

#[allow(unused_imports)]
use registers::{entry, rprintln};

// In ra giá trị hiện tại của P0.OUT
fn print_out() {
    const P0_OUT: u32 = 0x5000_0504;

    let out = unsafe { ptr::read_volatile(P0_OUT as *const u32) };

    rprintln!("P0.OUT = {:#08x}", out);
}

#[entry]
fn main() -> ! {
    registers::init();

    unsafe {
        // Một loạt địa chỉ "magic"!
        const P0_OUTSET: u32 = 0x5000_0508;
        const P0_OUTCLR: u32 = 0x5000_050C;

        // In giá trị ban đầu của OUT
        print_out();

        // Bật hàng LED trên cùng
        ptr::write_volatile(P0_OUTSET as *mut u32, 1 << 21);
        print_out();

        // Bật hàng LED dưới cùng
        ptr::write_volatile(P0_OUTSET as *mut u32, 1 << 19);
        print_out();

        // Tắt hàng LED trên cùng
        ptr::write_volatile(P0_OUTCLR as *mut u32, 1 << 21);
        print_out();

        // Tắt hàng LED dưới cùng
        ptr::write_volatile(P0_OUTCLR as *mut u32, 1 << 19);
        print_out();
    }

    loop {
        core::hint::spin_loop();
    }
}

Output:

text
15:13:24.055: P0.OUT = 0x000000
15:13:24.055: P0.OUT = 0x200000
15:13:24.055: P0.OUT = 0x280000
15:13:24.055: P0.OUT = 0x080000
15:13:24.055: P0.OUT = 0x000000

Điều kỳ lạ: mặc dù chúng ta đọc cùng một địa chỉ nhiều lần mà không trực tiếp sửa đổi nó, giá trị vẫn thay đổi — đây là tác dụng phụ của việc ghi vào OUTSET hoặc OUTCLR.

Lưu ý rằng OUTSET và OUTCLR không cho phép đọc trạng thái output hiện tại của Port P0 — khác với OUT vừa đọc được vừa ghi được.

Type-safe manipulation — Thao tác an toàn kiểu

Làm việc trực tiếp với địa chỉ hex rất dễ xảy ra lỗi: nhầm địa chỉ, truy cập register read-only để ghi, hay vô tình xóa các bit reserved. PAC (Peripheral Access Crate) giải quyết vấn đề này bằng cách cung cấp một API type-safe được tự động sinh ra từ file SVD (System View Description) của Nordic.

Thay vì ghi thẳng vào địa chỉ, bạn dùng API type-safe:

rust
#![no_main]
#![no_std]

#[allow(unused_imports)]
use registers::entry;

#[entry]
fn main() -> ! {
    let (p0, _p1) = registers::init();

    // Bật hàng trên cùng
    p0.out.modify(|_, w| w.pin21().set_bit());

    // Bật hàng dưới cùng
    p0.out.modify(|_, w| w.pin19().set_bit());

    // Tắt hàng trên cùng
    p0.out.modify(|_, w| w.pin21().clear_bit());

    // Tắt hàng dưới cùng
    p0.out.modify(|_, w| w.pin19().clear_bit());

    loop {
        core::hint::spin_loop();
    }
}

Phương thức modify() đọc giá trị register hiện tại, truyền vào closure để thay đổi, rồi ghi kết quả lại. API này:

  • Chỉ expose các thao tác được phép (read-only register không có write())
  • Dùng builder pattern để ngăn sửa đổi các bit reserved
  • Không có unsafe code hay địa chỉ hex magic
  • Zero-cost: compiler sinh ra cùng instructions như code unsafe gốc
