Registers

Câu hỏi đặt ra: điều gì thực sự xảy ra khi gọi display_pins.row1.set_high() hay button_a_pin.is_high()? Câu trả lời là: chúng ta đang ghi vào các vùng bộ nhớ đặc biệt gọi là register.

Một register là một vùng bộ nhớ đặc biệt dùng để điều khiển một peripheral. Peripheral cung cấp những tính năng mở rộng ngoài khả năng tính toán và logic gốc của processor.

Đây là đoạn code demo thao tác register trực tiếp:

rust
#![no_main]
#![no_std]

#[allow(unused_imports)]
use registers::entry;

#[entry]
fn main() -> ! {
    registers::init();

    unsafe {
        // Một địa chỉ "magic"!
        const PORT_P0_OUT: u32 = 0x50000504;

        // Bật hàng trên cùng
        *(PORT_P0_OUT as *mut u32) |= 1 << 21;

        // Bật hàng dưới cùng
        *(PORT_P0_OUT as *mut u32) |= 1 << 19;

        // Tắt hàng trên cùng
        *(PORT_P0_OUT as *mut u32) &= !(1 << 21);

        // Tắt hàng dưới cùng
        *(PORT_P0_OUT as *mut u32) &= !(1 << 19);
    }

    loop {
        core::hint::spin_loop();
    }
}

Địa chỉ 0x50000504 là địa chỉ của register OUT thuộc GPIO Port 0. Register 32-bit này điều khiển các pin output — mỗi bit tương ứng với một pin. Đặt bit lên 1 → pin xuất ra điện áp cao; đặt về 0 → pin xuất ra mức thấp.

Các chân GPIO hoạt động như các tiếp điểm điện. Digital output có thể tạo ra hai mức điện áp: "thấp" (0V, logic 0) hoặc "cao" (3V, logic 1). LED sáng khi được cấp điện áp cao đúng chiều. Digital input thì đọc mức điện áp thay vì tạo ra nó.

Vì nRF52833 có hơn 32 GPIO pin nhưng là kiến trúc 32-bit, các pin được tổ chức thành hai nhóm "P0" và "P1" — mỗi nhóm có bộ register riêng.

RTRM — Đọc Reference Manual

Để tìm hiểu địa chỉ register và ý nghĩa từng bit, cần tra nRF52833 Product Specification. Peripheral GPIO P0 có base address là 0x5000_0000. Register OUT nằm ở offset 0x504, tức địa chỉ đầy đủ là 0x50000504.

Register này 32-bit, mỗi bit tương ứng một pin: đặt bit lên 1 → pin xuất điện áp cao; đặt về 0 → pin xuất mức thấp. Giá trị reset là 0 (tất cả output tắt).

Theo pinmap của MB2: ROW1 (hàng trên cùng) là P0.21, ROW5 (hàng dưới cùng) là P0.19.

Khi chạy chương trình trên và quan sát qua GDB, giá trị register thay đổi như sau:

(mis)Optimization — Vấn đề với compiler tối ưu hóa

Khi biên dịch với --release, LLVM có thể tối ưu hóa bốn lần ghi vào register thành một lần duy nhất — vì nó không biết rằng địa chỉ đó có tác dụng phụ trên phần cứng. Kết quả là LED sẽ không nhấp nháy như mong đợi.

Giải pháp: dùng volatile operations qua core::ptr:

use core::ptr;

// Thay vì:
*(PORT_P0_OUT as *mut u32) |= 1 << 21;

// Dùng:
let out = ptr::read_volatile(PORT_P0_OUT as *mut u32);
ptr::write_volatile(PORT_P0_OUT as *mut u32, out | 1 << 21);

Volatile read/write báo cho compiler biết: "Các thao tác này có tác dụng phụ — đừng tối ưu hóa chúng đi." Build release với volatile sẽ giữ đủ bốn lần load-modify-store như thiết kế.

0xBAAAAAAD — Địa chỉ không hợp lệ

Điều gì xảy ra nếu bạn đọc từ một địa chỉ không hợp lệ, ví dụ 0x5000_A784 (không tương ứng với peripheral register nào)?

unsafe {
    ptr::read_volatile(0x5000_A784 as *const u32);
}

Processor không thể hoàn thành thao tác đó và ném ra một hardware exception (HardFault). Debugger dừng lại tại handler HardFault_. Bằng cách kiểm tra ExceptionFrame, bạn có thể xác định instruction nào gây ra lỗi — pc trỏ tới instruction đó, và r0 chứa địa chỉ sai (0x5000a784). Đây là cách Rust embedded xử lý memory access violations.

Spooky action at a distance

OUT không phải là register duy nhất điều khiển các pin của Port P0. Còn có hai register nữa: OUTSET và OUTCLR.

Đoạn code sau (examples/spooky.rs) in ra giá trị của P0.OUT sau mỗi thao tác, nhưng thay vì ghi thẳng vào OUT, nó ghi vào OUTSET và OUTCLR:

rust
#![no_main]
#![no_std]

use core::ptr;

#[allow(unused_imports)]
use registers::{entry, rprintln};

// In ra giá trị hiện tại của P0.OUT
fn print_out() {
    const P0_OUT: u32 = 0x5000_0504;

    let out = unsafe { ptr::read_volatile(P0_OUT as *const u32) };

    rprintln!("P0.OUT = {:#08x}", out);
}

#[entry]
fn main() -> ! {
    registers::init();

    unsafe {
        // Một loạt địa chỉ "magic"!
        const P0_OUTSET: u32 = 0x5000_0508;
        const P0_OUTCLR: u32 = 0x5000_050C;

        // In giá trị ban đầu của OUT
        print_out();

        // Bật hàng LED trên cùng
        ptr::write_volatile(P0_OUTSET as *mut u32, 1 << 21);
        print_out();

        // Bật hàng LED dưới cùng
        ptr::write_volatile(P0_OUTSET as *mut u32, 1 << 19);
        print_out();

        // Tắt hàng LED trên cùng
        ptr::write_volatile(P0_OUTCLR as *mut u32, 1 << 21);
        print_out();

        // Tắt hàng LED dưới cùng
        ptr::write_volatile(P0_OUTCLR as *mut u32, 1 << 19);
        print_out();
    }

    loop {
        core::hint::spin_loop();
    }
}

Output:

text
15:13:24.055: P0.OUT = 0x000000
15:13:24.055: P0.OUT = 0x200000
15:13:24.055: P0.OUT = 0x280000
15:13:24.055: P0.OUT = 0x080000
15:13:24.055: P0.OUT = 0x000000

Điều kỳ lạ: mặc dù chúng ta đọc cùng một địa chỉ nhiều lần mà không trực tiếp sửa đổi nó, giá trị vẫn thay đổi — đây là tác dụng phụ của việc ghi vào OUTSET hoặc OUTCLR.

Lưu ý rằng OUTSET và OUTCLR không cho phép đọc trạng thái output hiện tại của Port P0 — khác với OUT vừa đọc được vừa ghi được.

Type-safe manipulation — Thao tác an toàn kiểu

Làm việc trực tiếp với địa chỉ hex rất dễ xảy ra lỗi: nhầm địa chỉ, truy cập register read-only để ghi, hay vô tình xóa các bit reserved. PAC (Peripheral Access Crate) giải quyết vấn đề này bằng cách cung cấp một API type-safe được tự động sinh ra từ file SVD (System View Description) của Nordic.

Thay vì ghi thẳng vào địa chỉ, bạn viết:

p0.out.modify(|_, w| w.pin21().set_bit());

Phương thức modify() đọc giá trị register hiện tại, truyền vào closure để thay đổi, rồi ghi kết quả lại. API này:

  • Chỉ expose các thao tác được phép (read-only register không có write())
  • Dùng builder pattern để ngăn sửa đổi các bit reserved
  • Không có unsafe code hay địa chỉ hex magic
  • Zero-cost: compiler (với LTO) sinh ra cùng instructions như code unsafe gốc

Toàn bộ API này được tự động tạo ra — không ai viết tay. Vendor (Nordic) cung cấp file SVD mô tả tất cả registers và quyền truy cập; công cụ svd2rust chuyển nó thành Rust code.
