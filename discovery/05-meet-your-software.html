Làm quen với Phần mềm

Chương này giới thiệu những kiến thức cơ bản về việc build, chạy và debug các chương trình đơn giản. Thay vì đi sâu ngay vào các khái niệm lập trình Rust embedded cho MB2, chúng ta sẽ tập trung vào việc làm quen với quy trình làm việc thực tế.

Bắt đầu

Tài liệu và mã nguồn của cuốn sách có thể lấy về bằng lệnh:

bash
git clone http://github.com/rust-embedded/discovery-mb2

Mã nguồn markdown và code chương trình nằm trong thư mục discovery-mb2/mdbook/src. Mỗi thư mục chương đều chứa cả tài liệu lẫn ví dụ code hoàn chỉnh. Các tham chiếu đến file như src/main.rs đều là đường dẫn tương đối so với chương đang học.

Hiểu cách tổ chức code Rust

Dự án Rust có hai loại code chính: binary (chương trình thực thi) và library (thư viện module). Trong cuốn sách này, binary là trọng tâm chính.

Chương trình binary có thể nằm ở ba vị trí chuẩn:

  • src/main.rs — Tự động được biên dịch và chạy bằng cargo embed hoặc cargo run
  • examples/foo.rs — Biên dịch và chạy bằng cargo embed --example foo hoặc cargo run --example foo
  • src/bin/bar.rs — Biên dịch và chạy bằng cargo embed --bin bar hoặc cargo run --bin bar

Các vị trí này theo quy ước đã được thiết lập sẵn của Cargo, dù ban đầu có thể trông hơi phức tạp.

Thiết lập embedded

Thư mục 05-meet-your-software chứa chương trình đầu tiên để chúng ta đi qua toàn bộ quy trình embedded.

Build (Biên dịch)

Bước đầu tiên là build binary crate bằng cross-compilation — điều này cần thiết vì microcontroller có kiến trúc khác với máy tính thông thường. Cross-compiling trong Rust rất đơn giản: chỉ cần truyền thêm flag --target cho rustc hoặc Cargo.

Chọn target

micro:bit v2 chứa bộ xử lý Cortex-M4F. Trình biên dịch Rust hỗ trợ nhiều cross-compilation target cho kiến trúc Cortex-M:

  • thumbv6m-none-eabi (Cortex-M0 và M1)
  • thumbv7m-none-eabi (Cortex-M3)
  • thumbv7em-none-eabi (Cortex-M4 và M7)
  • thumbv7em-none-eabihf (Cortex-M4F và M7F)
  • thumbv8m.main-none-eabi (Cortex-M33 và M35P)
  • thumbv8m.main-none-eabihf (Cortex-M33F và M35PF)

Với micro:bit v2, cần dùng target thumbv7em-none-eabihf — phiên bản có hỗ trợ tăng tốc phần cứng cho số thực dấu chấm động (floating-point).

💡
Ghi chú

Hậu tố hf trong eabihf là viết tắt của "hard float" — tức là chip có đơn vị xử lý số thực bằng phần cứng (FPU). Điều này quan trọng vì nếu dùng target sai, code sẽ không chạy được hoặc chạy rất chậm.

Cài đặt và biên dịch

Trước khi cross-compile, cần cài đặt standard library cho target:

bash
$ rustup target add thumbv7em-none-eabihf

Sau đó biên dịch code ví dụ:

bash
$ cargo build --example init
   Compiling semver-parser v0.7.0
   Compiling proc-macro2 v1.0.86
   ...

    Finished dev [unoptimized + debuginfo] target(s) in 33.67s
📝
Lưu ý

Hãy chắc chắn biên dịch crate này không có tối ưu hóa. File Cargo.toml được cung cấp và lệnh build trên đây sẽ đảm bảo tắt tối ưu hóa, miễn là bạn không truyền flag --release cho cargo.

Kiểm tra kết quả

Xác nhận file thực thi là binary Arm bằng lệnh:

bash
$ cargo readobj --example init -- --file-headers

Output sẽ chứa Machine: Arm trong ELF header, xác nhận đã cross-compile thành công.

Flash (Nạp firmware)

Flashing là quá trình chuyển chương trình vào bộ nhớ vĩnh viễn (flash) của microcontroller. Sau khi flash, microcontroller sẽ chạy chương trình đó mỗi khi khởi động.

Chương trình trở thành phần mềm duy nhất chạy trên thiết bị — không có hệ điều hành hay tiến trình nền nào cả. Chương trình có toàn quyền kiểm soát phần cứng.

Quy trình flash hoạt động như thế nào

micro:bit có ba khối đen riêng biệt khi nhìn từ phía cổng USB:

  1. Speaker — thiết bị xuất âm thanh
  2. NRF52833 MCU — microcontroller chính
  3. NRF52820 MCU — microcontroller phụ đóng vai trò bridge (cầu nối)

MCU phụ thực hiện ba chức năng quan trọng:

  • Quản lý nguồn điện và reset của NRF52833 qua kết nối USB
  • Thiết lập cầu nối serial-to-USB
  • Cung cấp khả năng lập trình và debug thông qua giao thức SWD

Kiến trúc bridge này cho phép giao tiếp giữa máy tính và MCU chính, hỗ trợ việc nạp firmware, kiểm tra trạng thái chương trình khi debug, và nhiều tác vụ phát triển khác.

Lệnh thực thi

bash
$ cargo embed --example init
  (...)
     Erasing sectors ✔ [00:00:00] [####################]  2.00KiB/ 2.00KiB @  4.21KiB/s (eta 0s )
 Programming pages   ✔ [00:00:00] [####################]  2.00KiB/ 2.00KiB @  2.71KiB/s (eta 0s )
    Finished flashing in 0.608s

Công cụ cargo embed tự động build và flash binary. Sau khi flash xong, tiến trình vẫn tiếp tục chạy để hỗ trợ các thao tác debug tiếp theo.

Debug (Gỡ lỗi)

Cơ chế hoạt động

Trước khi debug chương trình, hãy hiểu rõ cơ chế bên dưới. Cài đặt default.gdb.enabled = true trong file Embed.toml sẽ khởi động một "GDB stub" sau khi flash. Server này nhận các lệnh GDB như đặt breakpoint rồi chuyển tiếp qua USB đến debug probe trên chip thứ hai, chip đó lại giao tiếp với microcontroller chính.

Bắt đầu debug!

Khi cargo-embed đang chạy, mở một terminal mới, điều hướng đến thư mục dự án rồi khởi động debugger:

bash
$ gdb ../../../target/thumbv7em-none-eabihf/debug/examples/init

Đường dẫn lồng nhau phản ánh cấu trúc workspace, trong đó tất cả các ví dụ dùng chung một thư mục target.

Kết nối đến GDB stub (mặc định chạy trên localhost:1337):

gdb
(gdb) target remote :1337
Remote debugging using :1337
0x00000116 in nrf52833_pac::{{impl}}::fmt...

Đặt breakpoint tại hàm main rồi tiếp tục chạy:

gdb
(gdb) break main
Breakpoint 1 at 0x104: file src/05-meet-your-software/examples/init.rs, line 9.
(gdb) continue
Continuing.

Breakpoint 1, init::__cortex_m_rt_main_trampoline ()...
⚠️
Lưu ý cho người dùng Windows

Xin lỗi người dùng Windows. GDB đi kèm với GNU Arm Embedded Toolchain không hỗ trợ chế độ TUI này. Lệnh layout src bên dưới sẽ không hoạt động trên Windows.

Để xem code nguồn ngay trong terminal, bật chế độ TUI:

gdb
(gdb) layout src

Kiểm tra giá trị biến:

gdb
(gdb) print x
$1 = 42
(gdb) print &x
$2 = (*mut i32) 0x20003fe8

Lệnh info locals hiển thị toàn bộ biến cục bộ cùng một lúc.

📝
Nếu chương trình không halt sau khi khởi động

Nếu chương trình không halt sau khi khởi động và bạn đang ở đâu đó sâu hơn trong chương trình, hãy thử chạy monitor reset halt để reset. Đây là do một bug trong probe-rs.

Đặt breakpoint ở số dòng cụ thể:

gdb
(gdb) break 13

Liệt kê tất cả breakpoints hiện có, rồi xóa một breakpoint cụ thể:

gdb
(gdb) info break
(gdb) delete 1
📝
Giới hạn breakpoint phần cứng

Processor chỉ hỗ trợ một số lượng giới hạn breakpoints. Hãy theo dõi số lượng breakpoint hiện có và xóa những cái không còn cần thiết.

Dùng next để bước qua từng dòng lệnh; dùng stepi để bước từng lệnh assembly. Xem assembly với ngữ cảnh source:

gdb
(gdb) disassemble /m

Bật chế độ TUI assembly:

gdb
(gdb) layout asm

Thoát chế độ TUI:

gdb
(gdb) tui disable

Reset lại chương trình:

gdb
(gdb) monitor reset
(gdb) c

Để kết thúc phiên debug:

gdb
(gdb) quit
⚠️
Lưu ý

Phải để cargo embed chạy trong terminal đầu tiên trong suốt quá trình debug. GDB cần kết nối đến GDB stub server do cargo embed tạo ra. Nếu đóng terminal đó, kết nối debug sẽ bị mất.

Bật đèn LED đầu tiên

micro:bit v2 có màn hình LED matrix 5×5. Để bật một LED, chương trình cần thiết lập các chân GPIO đúng mức điện áp tương ứng.

Với LED ở góc trên bên trái:

  • LED này kết nối với dây ROW1 và COL1
  • Hai dây này dẫn đến các chân P0.21 và P0.28 trên nRF52833
  • Đặt P0.21 ở mức HIGH (xuất 3.3V) và P0.28 ở mức LOW (nhận điện áp) sẽ làm LED sáng lên

Chạy ví dụ bằng lệnh:

bash
cargo run --example light-up

Code ví dụ (examples/light-it-up.rs):

rust
#![deny(unsafe_code)]
#![no_main]
#![no_std]

use cortex_m_rt::entry;
use embedded_hal::digital::OutputPin;
use microbit::board::Board;
use panic_halt as _;

#[entry]
fn main() -> ! {
    let mut board = Board::take().unwrap();

    board.display_pins.col1.set_low().unwrap();
    board.display_pins.row1.set_high().unwrap();

    loop {
        core::hint::spin_loop();
    }
}
